# mutsu — June 2026 implementation status

## Overview

June was focused on **decoupling the bytecode VM from the tree-walking Interpreter (PLAN.md 🔴 top priority)**.
mutsu's VM was in practice using the Interpreter as shared execution state plus a fallback target
([ANALYSIS.md](../ANALYSIS.md) §1), and we are peeling that apart step by step in strangler-fig fashion.

**Important policy (user directive):** the goal here is coupling reduction = architectural improvement, not performance.
Since CI (`make test` + the comprehensive `make roast`) gates every merge, essential refactors should be done boldly with CI
as the safety net (added the "Refactor boldly" section to CLAUDE.md).

### ★ Method-call hot path: caching single-candidate dispatch (#3870, 2026-06-28)

`push_method_dispatch_frame` was executing `resolve_all_methods_with_owner` on every method call (MRO walk, looking up
each ancestor in the String-keyed registry, arg-matching every overload, and cloning the matched `MethodDef`s into a `Vec`),
returning early when the candidate count is ≤1 (the overwhelming majority — single methods that never produce a
`nextsame`/`callsame` deferral frame). Now we **memoize, per `(class, method)` (interned Symbols), whether there are ≥2 structural
candidates** (counting overloads before arg-matching, under the same rules as the resolver — non-private, submethods at receiver
level only — with no cloning). Once known to be ≤1, the push returns immediately, skipping the whole per-call resolve. **Soundness**:
arg-matching only reduces the candidate count, so structural ≤1 stays ≤1 after arg-matching (a frame is never pushed) — the result
depends only on registry shape, making the `(class, method)` key sound, and it is invalidated together with the other method caches
on registry changes. The cache is an FxHashMap (consistent with #3867).

**Measurements (instruction counts, pinned to a single P-core)**: method-call 73.9B→70.0B = **-5.2%**, bench-class 4.170B→4.022B = **-3.5%**.
**Combined with #3867, -12.2% on method-call / -7.6% on bench-class versus pre-campaign main.** `make test` PASS (Files=1304);
nextsame/callsame/samewith/proto/multi/redispatch unit tests plus 30 whitelisted S12/S14/multi/inheritance roast files all PASS.

**Remaining (diffuse, large-scale)**: remaining perf cost = malloc/free ~18% (the per-call `HashMap<String,Value>` clone in
`attributes.to_map()`, the attribute-key `format!` in `call_compiled_method`, Value clones) plus SipHash ~8% (String-keyed maps —
attribute to_map, `class_role_param_bindings`, the registry on cache miss) plus `Symbol::intern`/`with_str` ~7%. All of these wait on
reworking attribute materialization, or Symbol-keying the registry (large), or NaN-boxing (post-GC, ADR-0001 layer 3b) — none can be taken in a single clean slice.

### ★ Method-call hot path: evicting SipHash with FxHashMap (#3867, 2026-06-28)

A perf slice following the method-call campaign (#3853/#3857/#3859). A `perf` profile showed that
**~7% of the method-call bench's self time was `SipHasher::write` / `BuildHasher::hash_one`** (hitting 4–8-byte
integer keys with a cryptographic-strength hash). Switched the interpreter's hottest maps to `rustc_hash::FxHashMap`
(an existing dependency):
- **`Env`'s `Symbol -> Value` overlay** (`src/env.rs`) — the hottest map, consulted on every variable read/write.
  Same for `GLOBAL_BASE` (built-in enum constants).
- **The method dispatch caches** (`src/runtime/mod.rs`) — `fast_method_cache` /
  `method_resolve_cache` / `multi_resolve_cache` / `multi_type_cacheable` (all keyed by `(Symbol, Symbol)`).

The public `Env` API (`From` / `from_symbol_map` / `set_global_base`) still accepts std `HashMap`, converting at the cold
construction boundary. `iter`/`keys`/`values` return `std::collections::hash_map::Iter` (hasher-independent) so they are unchanged;
iteration order is unspecified under both hashers, so consumers like the pseudo-stash/`clone_for_thread` are unaffected.

**Measurements (instruction counts, `perf stat`, pinned to a single P-core = deterministic, load-independent)**: method-call (300k iters)
79.7B→73.9B = **-7.3%**, bench-class 4.351B→4.170B = **-4.2%**. `make test` PASS (Files=1301).

**Remaining levers (latest profile)**: malloc/free ~17% (the `class_sym.resolve()` String alloc on the dispatch path,
the per-call clone in `attributes.to_map()`, the attribute-key `format!` in `call_compiled_method`, Value clones) /
`Symbol::intern(method)` on every call ~3.5%. Removing `resolve()` is a large refactor that Symbol-keys
`call_compiled_method`/`push_method_dispatch_frame`/the registry (PLAN §G "symbol intern round-trip"). Value clones wait on NaN-boxing
(post-GC, ADR-0001 layer 3b).

### ★ HTTP::Server::Tiny works end-to-end — stack-size fix for `Thread.start` workers (2026-06-28)

While investigating loading HTTP::Server::Tiny (a pure-Raku asynchronous HTTP server, PLAN.md §H) on mutsu and running it over real TCP,
a worker thread hit a **stack overflow** mid-request ("has overflowed its stack", aborting the process).
Pinpointed the cause with a gdb backtrace: the stack at overflow was **only 83 frames** — not infinite recursion; the worker thread's
**stack was simply too small**. Only `Thread.start` (`socket_thread.rs`) was using a bare `std::thread::spawn` (OS default ~2-8 MiB),
having slipped past the 256 MiB `spawn_user_thread` (`USER_THREAD_STACK_SIZE`) used by `start {}` / Promise / Supply workers.
The async server's react loop constructs objects whose BUILD re-enters the VM, and that deep nesting exceeded 8 MiB.

Fix: unified all worker spawns that execute user code onto `spawn_user_thread`. (1) `Thread.start` (the direct cause),
(2) `$*SCHEDULER.cue` scheduled callbacks (`scheduler.rs`), (3) hyper/race batch threads (`vm_hyper_race_parallel.rs` —
generalized `spawn_user_thread` to a `JoinHandle<T>` generic since these carry return values). As a result, **unmodified upstream
HTTP::Server::Tiny serves real HTTP responses to mutsu's `IO::Socket::INET` client** (GET = `hello`; a POST + body +
`start{}`-Promise-returning app = `method=POST len=8` round-trips correctly via the content-length path / TempFile and IO::Blob).
Regression pin = `t/thread-deep-stack.t` (deep recursion on a `Thread.start` worker).

### ★ §7-8 giant-file split sweep — clearing out the worst offenders against the 500-line rule (2026-06-28)

Against ANALYSIS §6/§8's "major violations of the 500-line rule", ran a campaign splitting files over 500 lines via **pure relocation into cohesive submodules**. Two established styles were followed:

- **Files of impl blocks** (a monolithic `impl Interpreter`) → split into **flat sibling files** in the same directory, adding `mod` declarations to the parent mod (`runtime/resolution.rs` #3792 is the prototype; visibility-only: private `fn`s called from siblings get promoted to `pub(crate)`).
- **Files of free functions** (parser, builtins) → **facade file + subdirectory** (`parser/primary/misc.rs` #3793 is the prototype; the facade preserves every external caller's path via `pub(crate) use submod::*`, and `super::` references in the bodies are rewritten to absolute `crate::...` paths).

Both are **pure relocation** (the multiset of functions/methods is byte-identical; behavior unchanged). CI (`make test` + the comprehensive `make roast`) is the safety net. Ran worktree-isolated agents at up to 4 in parallel; PRs editing shared mod files like `src/runtime/mod.rs` auto-merge via git 3-way when their alphabetical positions differ, with rebases resolving any conflicts.

Main splits (excerpt; one PR each):

### ★ registry-Arc campaign complete — registry clones no longer deep-clone `FunctionDef` bodies (slices 1-8)

Every time a routine declaring a `my sub` was entered, `snapshot_routine_registry` was **deep-cloning every routine body in the
program** via `functions.clone()` (for lexical-scope rollback). By converting every map in the registry that held `FunctionDef` directly
to `Arc<FunctionDef>`, the snapshot became an O(n) refcount bump:

- **Slice 1 (idempotence)**: re-execution of `RegisterSub` (a `my sub` inside a hot routine) is now classified as `SubRegisterOutcome::{Installed, Unchanged}` via a compile-time fingerprint (`CompiledCode.sub_fingerprints`), detecting identity-preserving re-registration in O(1) without re-deriving the `FunctionDef`.
- **Slice 2 (Arc-ifying `functions`)**: the registry holds `HashMap<Symbol, Arc<FunctionDef>>` (~28% improvement on a 200-sub microbench). There are zero in-place mutation sites, so no `make_mut` needed.
- **Slice 3 (threading Arc through dispatch resolution)**: the hot single-result resolvers now return `Option<Arc<FunctionDef>>`. The per-call deep clone of the resolution body becomes an Arc bump.
- **Slice 4 (derive-once cache)**: re-installing a `my sub` used to re-derive AST→`FunctionDef` every time; now cached in `prepared_fn_defs` keyed by `(fingerprint, package)`.
- **Slice 5 (Arc-ifying `MultiDispatchEntry`)**: the redispatch stack candidate list became `Vec<Arc<FunctionDef>>`. The per-step candidate clone in `nextsame`/`callsame`/`callwith` becomes a refcount bump.
- **Slices 6/7/8**: Arc-ified the remaining `proto_functions` (6), grammar `token_defs` (7), and `our_scoped_functions` (8). No registry clone — `snapshot_routine_registry` / `clone_for_thread` / EVAL copies / block-scope restore — deep-clones a `FunctionDef` body anymore.

The only remaining §1.1 item is resolution caching for method dispatch (multi already started in #3684).

### ★ §D(b) / §B complete — removing the tree-walk for user method bodies (deleting the non-delegation arm of `run_instance_method_resolved`, #3664–#3680)

At the end of June, we **deleted the non-delegation arm of `run_instance_method_resolved` (`run_block(method_def.body)`, ~470 lines) — the only remaining true user-code tree-walk** —
making the bytecode VM the **sole execution engine** for user method bodies (user-code tree-walking is gone). The original PLAN estimated this as
"deep multi-slice work requiring design to extend captured-outer/closure-env writeback coherence across the compiled-method redispatch boundary", but it was completed by de-risking across 11 PRs:

1. **#3664** `pending_rw_writeback_sources` is **merged** instead of restored (preserving the body's captured-outer writes).
2. **#3666** exclude attribute twigils (`.x`/`!x`/`@.`/`@!`/`%.`/`%!`) from `free_var_writes` (new `is_attribute_accessor_name`), plus slotless attr inc/dec as a cell RMW (new `try_slotless_attr_incdec`; `$.count++` persists when compiled).
3. **#3668** compile methods that write captured-outer lexicals (gate relaxation; only dynamic/special twigils keep the tree-walk).
4. **#3670** fixed the bug where `can_skip_merge` lacked `has_calls` (env written by a closure call `$f()` was discarded by method dispatch → nested-closure dynvar loss). Pre-existing bug.
5. **#3672** fully removed the `free_var_writes` gate filter.
6. **#3674** qualified calls / `.*` walk / proto dispatch go compiled via the helper (6 direct callers).
7. **#3677** Mixin self-attribute writeback fix (`self_instance_attrs` recursively unwraps `Value::Mixin(inner,_)`→inner instance cell; read/write_self_attr_cell plus the helper's final_attrs commit), and made on-demand compile (`compile_method_def_in_place`) **unconditional** in the helper → runtime-`does` mixins / late-added methods get compiled.
8. **#3678** compile empty bodies too (the helper's fallback becomes delegation-only), plus a compiled-fast-path **arity-validation fix** (named args were mis-bound as positionals; missing required positionals were silently bound to Nil → calls involving named/missing-required now route to the full path). Delegation forwarders skip compilation via `delegation.is_some()` (caught by roast S14-roles/composition).
9. **#3680** (capstone) on-demand compile for `builtins_dispatch_next` too → confirmed zero non-delegation hits across both t/ and roast with `MUTSU_PROBE_TREEWALK` → **deleted the non-delegation tree-walk arm**. `run_instance_method_resolved` is now delegation-forwarder-only (non-delegation entry is a defensive Err).

**Remaining (separate axes, optional)**: the cosmetic rename `run_instance_method_resolved`→`forward_resolved_delegation`. Captured-outer lexical writeback for hyper `@objs».meth` (a separate pre-existing gap).

### ★ Multi-method resolution cache (§B follow-up, #3684)

Multi-method dispatch was re-running the MRO + specificity walk every time (the catch-all cached non-multi only; the mut-variant path for variable receivers had no cache). Added a **sound** resolution cache keyed by `(class, method, positional argument types)`. Enabled **only for type+arity-deterministic multis** (excluded via a memoized check if any candidate has `where`/literal/subset/`:D:U` smiley/coercion; ambiguous cases are never cached and raise `X::Multi::Ambiguous` every time; arg-type keys are Instance→class / scalar→`value_type_name` / bail on Junction/Mixin/container/Pair). Both variants share the cache via `resolve_method_cached`. As a byproduct, also fixed a latent bug where `exec_register_class_op` was not invalidating the method cache (stale resolution for a same-named `my class`; caught by S12-methods/multi.t). ~15% speedup on a hot 2-candidate multi loop (6.71→5.68s/400k). pin=`t/multi-resolve-cache.t`(15).

### §D(b) removing the tree-walk dispatch chain — VM-native pure Str methods (`.starts-with`/`.ends-with`/`.substr-eq`)

Started §D(b) (structural removal of the tree-walk method-dispatch chain). Per-name measurement of method fallbacks showed that, excluding
non-targets (MOP reflection `name`/`WHAT`/`WHY`/`can`) and separate axes (concurrency `tap`/`stdout`, lazy iterator protocol `push-*`/`pull-one`),
the **largest removable bounce** was `starts-with` (513). The cause was a common pattern: **pure methods placed on the slow path for
named-arg support were bouncing to the interpreter even in their plain form**.

- **`.starts-with`/`.ends-with` (#3580)**: they lived in `dispatch_prefix_suffix_check` for the case-/mark-insensitive forms
  (`:i`/`:ignorecase`/`:m`/`:ignoremark`), but plain `.starts-with($needle)` is a pure prefix/suffix check. Added a
  Str-receiver-gated arm to `native_method_1arg`. The named-arg form takes 2 arguments (positional + Pair), so it never reaches the 1-arg native path and falls through naturally.
- **`.substr-eq($needle, Int $pos)` (#3581)**: same pattern. Added the plain 2-argument form — non-negative in-bounds Int position, Str receiver — to `native_method_2arg`.
  Whatever/negative/out-of-range positions (which produce a Failure) and the named-arg form (arity 3) keep falling through.

In both cases **user-defined methods are resolved by the VM before the native path = no shadowing** (same guarantee as the existing `contains` arm), and the fall-through cases are byte-identical.
pin=`t/starts-ends-with-native.t`(22)/`t/substr-eq-native.t`(16). Zero regressions across all of t/, the S32-str family, and the roast sample.

**★Lesson**: for pure methods placed on the slow path for named-arg support, the named-arg form increases the arity and takes a different dispatch path, so
**gating "just the plain case" on the receiver type drains them safely**. **The clean string drains are now exhausted** (`comb`'s simple form is already native /
`trans` is complex with ranges, list-pairs, regexes / scalar `.new` was finished in the ctor work). The remaining categories (iterator protocol = lazy, MOP = reflection,
concurrency) all presuppose different substrates — they are not pure §D(b) drains.

### Fixed the `use` `ORIGINAL_SOURCE` clobber bug, plus general fixes for the latent phaser block-value bugs it exposed (§D)

Fixed the **`use` `ORIGINAL_SOURCE` clobber bug** that PLAN.md §2-D had left as "separated, different axis (needs separate work)", and
also generally fixed the family of latent phaser bugs it exposed (the work foreshadowed in the "Lesson" section of the previous test-assertion OTF slice, #3570).

**(1) `ORIGINAL_SOURCE` clobber fix.** Module export scanning (`extract_exported_names`→`parse_program_partial`→
`set_original_source(module_src)`) was overwriting the parser thread-local `ORIGINAL_SOURCE` with the module's temporary `String`
(dropped immediately afterward), leaving it dangling, after which the using file's `current_line_number` collapsed to all 1s (triggered by
real `.rakumod` loads such as `use Test::Util`). Added `primary::snapshot_source_state`/`restore_source_state` (save/restore of
`ORIGINAL_SOURCE` plus the heredoc `LEAKED_REGIONS`) and changed `parse_program_partial` to save and restore the caller's source state
around nested sub-parses. Now `$?LINE`, die lines, and caller-line markers after `use Mod;` match raku (minimal repro = `$?LINE` after
`use`-ing a custom module with `-I lib`).

**(2) General fixes for the exposed phaser block-value bugs (real bugs, independent of line numbers).** Fixed two families of bugs that
only happened to PASS while the clobber squashed line numbers to 1:
- **A trailing ENTER phaser becomes the block's value.** `sub f() { ENTER 'X' }` / `do { ENTER 7 }` / `{ ENTER 5 }` (closure) /
  `-> { ...; ENTER 'L' }` were returning **Nil** (the ENTER section runs before the block's value-result baseline is recorded, so a value
  it leaves on the stack is not detected). New opcodes `PushEnterResult`/`LoadEnterResult` (the VM's `enter_result_stack`) bridge the
  ENTER section's value to the end of the body. Matches Raku semantics (if the textually last statement is an ENTER, its entry-time value is the block value).
- **Exclude trailing `SetLine` markers from the value-determining statement.** With line numbers correct, `SetLine` markers appear between
  statements, and in phaser-only blocks (`{ KEEP{} KEEP{} UNDO{} UNDO{} }`) a trailing `SetLine` became "the last statement", producing a
  spurious `True` that **wrongly fired KEEP** (correct = UNDO). Fixed by making the value-determining statement "the last non-`SetLine` statement".

**(3) Refactor (CLAUDE.md "Refactor boldly").** Nearly identical BlockScope phaser compression was **duplicated in 5 places** (top-level routine
body / `Stmt::Block` / do-block / sub body / closure body). Consolidated into a shared helper `compile_phaser_block_scope(stmts, result_on_stack)`
(`result_on_stack=false` = statement context goes via the topic; `true` = do-block expression context leaves it on the stack). Sub body / closure body
are inline fixes because of `is rw` escape semantics, but share the same ENTER/`SetLine` logic. Diff is net **−228/+270** (effectively smaller thanks to the deduplication).

Verification: pin=`t/enter-phaser-rvalue.t`(18). All of t/ (1146 files, 11502 tests) PASS, all 17 whitelisted S04-phasers files PASS, and zero regressions
across the 270 whitelisted files that `use Test::Util` plus a 428-file 1/3 roast sample. **Lesson: nested parses clobber not just `SCOPES` but also
`ORIGINAL_SOURCE`/`LEAKED_REGIONS`. Normalizing line numbers exposes tests that "passed by accident under clobbered line 1", so proceed in
lockstep with general fixes for the exposed bugs. A phaser block's value needs materialization in a separate phase from the ENTER section
(bridge the entry-time value to the end of the body via a dedicated stack).**

### VM-native non-mut list methods on `is Array` instances (§D / §C Phase-3 fork unblock)

For instances of `class S is Array {}`, delegating the non-mut list methods (`.sort`/`.reverse`/`.map`/`.first`/`.unique`/
`.join`/`.head`/`.tail`/`.elems`, etc.) to the backing `__mutsu_array_storage` was bouncing all the richer methods to the
tree-walk interpreter (the "Array-backed instance method" fork in ledger §1). These are
**already VM-native for plain arrays** (`try_native_array_map`/`try_native_first`/`try_native_minmax`/`try_native_method`),
so the is-Array fork (`vm_call_method_mut_ops.rs`; `$s.map` etc. go through CallMethodMut for variable receivers) now calls the same
native helpers against the storage. These borrow `&storage` and return fresh values — they do not mutate the instance — so
by-value delegation is correct. The fallback shrank to just `grep` (whose result shares an rw view into the source; `for $s.grep {$_++}` writes back)
and the mutating methods (`splice`/`ASSIGN-POS`/…) — the part that needs first-class element-cell write-back (lever B).
Output is byte-identical to main (matches the existing interpreter fallback). pin=`t/native-array-backed-instance-methods.t`(13).

**Follow-up: correctness fixes for is-Array list methods (whitelist expansion)**: during the above investigation, the
mut-path delegation gate for `is Array` instances (the `is_array_method` whitelist in `vm_call_method_mut_ops.rs`) turned out to be
**missing many list methods**:
`$s.minmax`/`.min`/`.max`/`.sum`/`.kv`/`.pairs`/`.keys`/`.values`/`.reduce`/`.classify`/`.combinations`/`.permutations`/
`.all`/`.any` etc. were not delegated to storage and instead **treated the instance itself as a one-element list**, returning wrong results
(`S.new` / `Nil` / "No such method") — variable receivers go through CallMethodMut = the mut path, while the non-mut path was correct via blanket delegation.
Added numeric reduction (min/max/minmax/sum/reduce/produce) / index views (kv/pairs/antipairs/keys/values) / grouping and
combinatorics (classify/categorize/combinations/permutations/rotor/batch) / selection (pick/roll) / junctions
(all/any/none/one) to the whitelist; all now match raku. Extended pin=`t/native-array-backed-instance-methods.t`(28).

### VM-izing non-trivial proto bodies (§D multi-dispatch, #3550 / #3552)

A **proto with a body**, like `proto foo($x) { say "x"; {*} }`, used to execute the whole proto body on the
tree-walking interpreter (`eval_block_value`) (measured 100% fallback). VM-ized in 2 slices:

- **(1) Compiled execution of the proto body (#3550)**: added `vm_try_run_nontrivial_proto_body` to the proto fork of
  `dispatch_func_call_inner` (after trivial resolution). The body, with `{*}` rewritten to `__PROTO_DISPATCH__()`, runs compiled via
  `otf_compile_function_def` (a shared helper extracted from `compile_and_call_function_def`) plus
  `call_compiled_function_named`. A proto body is a dispatcher, not a candidate, so no
  multi-dispatch frame is pushed; only `proto_dispatch_stack` is pushed/popped (new accessor; args = the ORIGINAL proto
  args, matching the interpreter's `call_proto_function`). Gate = proto sig + `def_is_otf_compilable` + non-`state`.
  Ineligible protos (class/role/start bodies; default/where/code-param sigs) keep the interpreter fallback.
- **(2) VM-izing candidate dispatch (#3552)**: intercept `__PROTO_DISPATCH__` at the top of `dispatch_func_call_inner` →
  `vm_call_proto_dispatch` runs the winner (`resolve_proto_candidate_with_types`) OTF via `compile_and_call_function_def`
  (the same path as the trivial fork). Candidate bodies plus `nextsame`/`callsame`/`samewith` redispatch become VM-native.
  Proto methods (method_ctx) / no-match / ambiguity / empty-sig-with-args / non-OTF / state delegate to the interpreter's
  `call_proto_dispatch` (made `pub(crate)`) — X::Multi::NoMatch/Ambiguous generation and proto_method_skip stay interpreter-owned.
  The builtin-shadow hazard does not apply (candidate name = proto name with multis → `otf_call_cache` insert is skipped).

Measured on `t/proto-nontrivial-body-vm.t`: interpreter_fallbacks **50.1%→0.5%** (`__PROTO_DISPATCH__` 188→0).
pin=`t/proto-nontrivial-body-vm.t`(13) / `t/proto-candidate-otf-dispatch.t`(7). The failure counts of the 24 proto-involving roast files
match main exactly (zero regressions). **Remaining, separate axis**: `is rw` proto-param writeback (`inc($v)`→10; identical on main/(1)/(2))
is a deep problem — the `proto_dispatch_stack` args carry the proto's original values rather than the caller's containers (aliases), so writes don't propagate
(the interpreter path is broken the same way; trivial protos work because they bypass the body and call with the original args directly).

### `{*}` rw-redispatch — candidates see body-mutated proto params and write back (§D multi-dispatch follow-up)

Resolved the `is rw` proto-param writeback that was the "remaining, separate axis" above. With `proto pr($x is rw){ $x=99; {*} }` /
`multi pr(Int $x is rw){ $x=$x+1 }`, `pr($v)` ($v=10) gives **100** in raku (the candidate sees the body-mutated `$x`=99, writes
100, and it propagates to $v) but mutsu gave **99** (the candidate didn't see the body mutation, and the candidate's rw write was lost).

**★Initially classified as "entangled with the §C varref container substrate (heavier than first assumed)" — that was a misclassification**
(the same shape as when the terminator was misclassified as "parser auto-curly" but was really EVAL writeback). The real fix is achievable with the arg_sources mechanism:

- Added a `proto_rw_redispatch_args` helper to `vm_call_proto_dispatch`. When the proto has a **scalar `is rw`/`is raw`
  positional param** in a simple-positional sig, rebuild the **current values** of the proto params at the `{*}` point.
  Scalar rw params are **slot-only** mid-body (env not yet flushed), so read them from the live body locals (the slots of the proto body's
  `code.locals` held by `dispatch_func_call_inner`).
- At the same time, `set_pending_call_arg_sources` with **arg_sources = the proto param names** before resolution. Even with the
  rebuilt plain values, the candidate's `is rw` passes multi-dispatch's writability check when arg_sources are present
  (`args_matching.rs:211` `has_arg_source`; a named source is equivalent to a VarRef).
- As a result, the candidate's rw writeback lands on **`$x` in the proto frame env** → `apply_pending_rw_writeback` at the
  `__PROTO_DISPATCH__` call site writes it back into the proto body locals → `apply_rw_bindings_to_env` at proto exit propagates it
  through the proto's rw binding (proto `$x`→caller `$v`) all the way to `$v`. The chain: candidate → proto param → caller.

The gate covers only protos with scalar rw/raw params (non-rw protos are completely unchanged). pin=`t/proto-rw-redispatch-coherence.t`(9)
= proto param name ≠ candidate name (positional aliasing) / non-leading positional rw / `is raw` / body non-mutating / candidate non-mutating / non-rw proto unchanged.

**Follow-up: extended to proto methods (same session)**: the rw redispatch for **proto methods** — initially "remaining gap (1)" — was also
solved with the same mechanism. Proto methods fall into the interpreter's `call_proto_dispatch` via `method_ctx`, where the arguments themselves
were likewise lost, producing `Cannot resolve caller m(C:D: )` (the candidate's `is rw` rejects a plain value as a writable arg).
Generalized `proto_rw_redispatch_args` to **take the proto's param_defs directly**, callable both from the VM sub path (`resolve_proto_function`)
and the interpreter proto-method path (resolving the proto def from the invocant class via `lookup_proto_method`).
Proto methods always carry the implicit `%_` (named catch-all), so the difference from proto subs is determining arity from the
fixed positional params (excluding `%_`/`@_`/slurpies/named). The interpreter method path binds params into the env, so current values are read from the env (`code=None`).
pin=`t/proto-method-rw-redispatch.t`(5).

**Remaining gap (separate axis)**: the **nextsame+rw chain** (the first candidate's rw propagates, but an rw write passed to the next candidate via
nextsame is lost in the separate `multi_dispatch_stack` mechanism; the 2-candidate case improved 40→41 but is still short of raku's 1041).

### Fixed nested classes as attribute types → Template::Mustache revived (PLAN §H)

Fixed a bug where using a nested class as an attribute's type failed to resolve at construction time:

```raku
class Outer {
    class Inner { }
    has Inner $.x;   # the Inner type object is the default
}
Outer.new;           # previously: "Undeclared name: Inner used at line 1"
```

- **Root cause**: the default of `has Inner $.x` (no explicit default) is the `Inner` type object (`BareWord("Inner")`).
  It is evaluated at construction time, but at that point there is no method-class / package context, and the nested class's short name `Inner` is
  suppressed (registered so it is invisible outside the class), so `resolve_suppressed_type` could not find `Outer::Inner` and failed.
  The trigger point was attribute-default evaluation in `build_native_default_instance` (`methods_object.rs`), which did not even switch
  the package when `has_class_scoped_subs` was false (no class-scoped subs).
- **Fix**: record the class under construction in a `constructing_class` field, set only for the duration of attribute-default evaluation.
  `resolve_suppressed_type` can now resolve `<Class>::Inner` in this context. **Scoping is not broken**:
  the field is only raised during default evaluation, so `Inner.new` outside the class (Undeclared in raku) continues to fail.
  Guarantee = `t/nested-class-attribute-type.t` (8 cases, including negative cases).
- **Ripple effect**: Template::Mustache's `Logger` class contains an inner `class LoggersMap is Hash {...}` used via `has LoggersMap $.routines;`.
  This bug made `.render` die entirely with `Undeclared name: LoggersMap`; that is now resolved and **Mustache works again** (`Hello World!`).
  Note: this bug was exposed by the precomp-cache staleness fix (previous item) — an old, correct cache had been hiding Mustache's behavior.

### Precomp-cache staleness fix, plus a thin working DBDish::SQLite compatibility layer (exe-mtime stamp)

While preparing the web blog stack's DB layer as a "reusable pure-Raku module", discovered and fixed a **real bug** in the
precomp cache (PLAN §H).

- **Symptom**: writing the NativeCall SQLite binding as a module (`unit module DB`) dies with `Undeclared name: Pointer`
  even though main never references `Pointer`. The identical content under a different module name works.
- **Root cause**: the precomp cache stores the **AST after prelude injection** (after `inject_nativecall_prelude` prepends the `Pointer` class),
  but the cache version stamp was only `CARGO_PKG_VERSION` (a fixed "0.1.0" in dev builds) plus the enum `CACHE_FORMAT_VERSION`,
  and did not capture changes to the injection logic. A **prelude-less AST** cached before #3527 ("inject the Pointer prelude into modules too")
  kept being treated as valid and reused because the source mtime/hash matched. CLAUDE.md's

  the binary rather than the source" changes — injection, `merge_unit_class`, roast directive preprocessing — a rebuild automatically invalidates the cache.

  CREATE/INSERT/ORDER BY/WHERE SELECT against real SQLite. Guarantee = `t/dbdish-lite.t` (graceful skip when libsqlite3 is absent).
  = the web blog's DB layer is now in place as a reusable thin DBDish-compatible layer.

### Removing the grep-rw-view side-channel mechanism — `.grep` becomes first-class element cells (#3466, 2026-06-23)

Source writeback for `for @a.grep(...) { $_++ }` / `@a.grep(...)>>++` was implemented via a dedicated side channel from the tree-walk era:
`for_grep_view` (an Interpreter field, save/restored in vm.rs), `GrepView`/`grep_source` bindings embedded in the filtered
`ArrayData`, and the `overwrite_array_items_by_identity_for_vm` writeback that identified the source array by Arc identity.
Eliminated all of it and replaced it with **first-class containers (element cells)** (PLAN §C):

- `.grep` over an array promotes the matched source slots to shared `ContainerRef` cells, and the result references the same cells.
  The writeback loop goes through the ordinary element-cell write path back into @a's slots (for-loops already write via cells).
  `>>++`/`>>--` inc the cells in place in the hyper loop. `=` assignment deconts the cells, so a named copy
  (`my @g = @a.grep(...)`) does not write back.
- Since the result carries real cells, every read path above it must decont. **Of the ~30 ops measured, only 6 places were missing**
  (each also a pre-existing bug for `:=`-bound element cells): `compare_values` (.sort/min/max) / `collect_minmax_candidates`
  (.minmax) / the hyper method loop (`>>.method`) / 3 Instance checks in `.join` (narg fast / runtime / 0-arg) /
  **the method-dispatch invocant** (decont a `ContainerRef` invocant in `exec_call_method_op` except for `.VAR`) /
  `smart_match`'s LHS (`does-ok ... .grep(Rational).head`, roast S32-num/rat.t).
- **★The central insight is an asymmetry**: deconting the CallMethod invocant is safe (a read) but **CallMethodMut must NOT decont** —
  otherwise `my @x := @src; @x.push`'s `:=`-bound container mutation would no longer reach the source. Decont only on read dispatch.
  `.head`/`.first` returning a cell is correct (in Raku `.head` is rw) → deref on the consumer side.
- Net **-90 lines**. pin=`t/grep-cell-read-deref.t` (19). Physically deleted the `for_grep_view` field, the `GrepView`/`grep_source`
  ArrayData fields, and `grep_view_writes`/`grep_view_topic_value`. CI (make test + make roast, release) green.

**★Lesson**: the blast radius of cell-ification is an order of magnitude larger than GrepView (metadata only, read-transparent) —
every read consumer must deref — but the measured gap was small. The prior PLAN's "every read path needs deref; impossible as a single slice"
was an overestimate. Preparing this read-deref chokepoint also lays the groundwork for the next Phase-2 "final SlotRef kill" (it hits the same 6 places).

### The final SlotRef kill — unifying `HashSlotRef`+`DeferredHashAccess` into a single `HashEntryRef` (#3472, 2026-06-23)

The last task of first-class containers Phase 2. Unified the 2 surviving variants of the `SlotRef` era (`HashSlotRef{hash,key}` = the
single `:=` bind token for a missing key, and `DeferredHashAccess{parent_slot,key}` = its 1-level-nested version) into a single path-based
`Value::HashEntryRef { hash, path: Vec<String> }`.

**Confirmed that deleting all variants is impossible in principle**: a `:=` bind to a missing key (`my $b := %h<x>`) requires a
deferred-vivification token that does not pollute `:exists`/iteration — "reads as Any, writes materialize into the parent hash". This token
must hold `(hash Arc, key path)`, and since it rides in a `Value` slot it can only be a Value variant (`ContainerRef`
cells are exclusively aliases of already-materialized entries). The `is raw` reduce lvalue descent shares the same token. It cannot be removed
without the anti-goal ptr-keyed side table.

∴ Achieved as "expunging the `SlotRef` concept and name": folded the `parent_slot` chain into `path: Vec<String>`, unified
`hash_slot_read/write` + `deferred_hash_read` into `hash_entry_read/write` (shared walk-create helper
`hash_entry_terminal`), and deleted the dead `hash_slot_ref_lazy`. `git grep 'SlotRef\|DeferredHashAccess' src/`=0.
**As a byproduct, missing-path binds of 3 or more levels (`$b := %h<a><b><c>`) now work** (the old 2-variant chain only reached 2 levels
= diverged from raku). `value_is_defined`/`truthy` treat an unmaterialized token as unconditionally undefined/falsy
(raku-correct: no retro-binding on external writes); display/type/isa read through. pin=`t/hash-entry-ref-deep-bind.t`.

### Stability: the VM's panic→`X::` conversion boundary (ANALYSIS §2.1)

Rust panics originating from user code (integer overflow, `Vec` capacity overflow, index OOB, `unwrap`/`unreachable!`,
etc.) used to crash the whole process (exit 101 / core dump). These are now caught at a boundary inside the VM and
converted into a catchable `X::AdHoc` (`Internal error: <payload>`). Crash → the normal error path
(message display, exit 1, catchable via `try`/`CATCH`).

- **Two-tier boundary**:
  - Wrap `VM::run` in `catch_unwind` (the loop body was carved out into `run_inner` taking `&mut self`, eliminating the
    move-return of `self` to make `catch_unwind(AssertUnwindSafe)` possible). Safety net for panics in top-level
    statements (outside `try`). EVAL / sub-VMs also go through `VM::run`, so they are covered automatically.
  - `try`/`CATCH` body execution uses `run_range_guarded`. As a result, even a panic nested arbitrarily deep inside a try
    (crossing unguarded `run_range`/`run_reuse` frames) unwinds back to this single boundary, is converted to
    `Err`, and flows to the CATCH handler through the existing exception machinery. Intermediate frames need no guard.
- **No perf impact**: the hot loop body `run_range` remains unguarded. Boundaries exist only at `try` blocks and the top
  level.
- **Suppressing the stderr backtrace**: a `thread_local` flag + custom panic hook suppresses the default backtrace output

### Track C Stage 2: VM ownership + compiled dispatch of the react/supply drive loops (#3038 / #3039)

Eliminated the "whenever bodies are still tree-walked as AST — a true fallback" left over from Stage 1. The key is an **inversion of
loop ownership**: since `VM` owns `Interpreter` by value (`VM::new` moves it), a VM cannot be constructed inside a
`&mut Interpreter` method, and the borrow checker forces "move the drive loop itself into `impl VM`".

- **PR1 = pure relocation (#3038, zero behavior change)**: moved the drive-loop family (`run_react_event_loop` / `_drain` /
  `run_react_consumer` / `drive_react_subscriptions` / `replay_static_supply` + the `extract_*` helpers) from
  `impl Interpreter` (`runtime/subtest.rs`) to `impl VM` (new `vm/vm_react_loop.rs`). Dispatch stays
  `self.interpreter.call_sub_value` (tree-walk) — valid because the VM owns the Interpreter — isolating the high-risk
  structural change from semantic change. The `react{}` path (`exec_react_scope_op`, already VM) calls the VM method directly,
  and the `await $supply`/`.Promise` path (`supply_promise_on_demand`, Interpreter) reaches it via a thin
  `Interpreter::drive_react_subscriptions` bridge (`mem::take`/`VM::new`/`into_interpreter` — the same established pattern as the native map
  loop). Supplier registry accessors (`supplier_snapshot`, etc.) promoted to `pub(crate)`.
- **PR2 = compiled dispatch (#3039, the main event)**: switched the 19 `call_sub_value` sites for whenever bodies / LAST / CLOSE callbacks
  to **compiled bytecode** execution via the new helper `VM::call_react_callback`.
  **Trap**: whenever bodies are compiled on the fly as routines by `compile_routine_closure_body`, and
  routines reset `$_` to `Any` on entry, so a naive `vm_call_on_value` swap loses the topic in `react whenever
  Supply { ... $_ ... }` (the implicit topic; the explicit `-> $v` form is unaffected). Resolved by force-binding `$_` after the reset using
  the explicit-topic mechanism of `vm_call_map_block` (precedent: native `.map`). Control signals
  (`done`/`next`/`last`) still propagate as `Err`, so the loop-side signal mapping is unchanged, and capture/writeback of outer lexicals
  (`@x.push($_)` etc.) works on the compiled path just like `.map` blocks.
- **PR3 = cleanup**: removed the `// TODO: compile to bytecode` in `vm_register_ops.rs`, updated the ledger's react-loop row and
  PLAN.md Track C to complete.

`supply_emit_buffer` remains an Interpreter field (reachable from the VM via `self.interpreter.`; the originally planned
globalization turned out to be unnecessary). Verification: each PR is build+fmt+clippy clean; control signals / writeback / user-function calls
match rakudo; 40 t/ react/supply files; whitelist S17-supply/promise 64 files (872); make test (7129).

### Track C shared cells, slice 3: atomic scalar compound assignment (`$x OP= rhs`) (#3059, 2026-06-14)

Following slice 1 (`start`-captured scalars → shared `ContainerRef` cells, atomic RMW for `++`/`--`) and
slice 2 (`state $n` thread sharing), scalar **compound assignment** (`+= -= *= /= ~= min= +&= ...`) is generalized to
an atomic read-modify-write on the shared cell. A `start`-captured lexical now accumulates deterministically across threads:

```raku
my $c = 0;
await (^100).map: { start { for ^100 { $c += 1 } } };
say $c;   # 10000 (previously ~4000, racing with lost updates)
```

raku does not guarantee atomicity of `+=`, but mutsu is intentionally deterministic, just like slice 1's `++`.

**Design**:
- Emit the new opcode `OpCode::AtomicCompoundVar { name_idx, op: CompoundBaseOp }` at the compiler's
  choke points (`compile_expr_assign` / the tail of `Stmt::Assign`),
  **only for plain env-named scalars**. Local slots and twigil/special targets are excluded, so the
  hot literal `$x = $x + y` own-local loop is not fused (fusing it regressed a 3M-iteration loop from 31.6s to 41s+).
- Register with the mutation/escape analysis (`op_name_const_idx` / `op_name_write_const_idx` / `has_env_writes`) so that
  `$c` stays boxed in a shared `ContainerRef` cell.
- Non-cell writes share `store_named_scalar_rmw_result`, extracted from `exec_post_increment_op_inner` (the env
  writeback tail of `++`), with `++`/`--` → METHOD captured-outer scalar propagation is structurally identical to `++`.
- The ContainerRef cell branch holds the lock for the whole RMW (atomic). The Proxy branch is fetch/op/store.
- `apply_compound_base_op` reuses the same `exec_*_op` as the plain `Binary` path (operator semantics, coercion, and
  user `infix` overloads agree).

**The true nature of the wall the first (reverted) attempt hit** was not the "SetGlobal store-tail reuse
refactor" the post-mortem predicted, but a single line in the compiler: compile the fused rhs with `compile_expr` (not
`compile_call_arg`). `compile_call_arg` itemizes/escape-boxes scalar operands, breaking captured-outer env propagation
when the rhs is a variable reference (`$acc += $!step` inside a method).

Verification: `t/concurrent-compound-assign.t` (new, 8 tests), regression anchors `t/method-env-dirty.t` (24/24) and
`t/test-util-test-iter-opt.t` (3/3), `make test` green, the 3M local loop is ~baseline (locals are not fused).

### Track C shared cells, slice 4: cross-thread hash element assignment (`%h{$k} = $v`) (#3061, 2026-06-14)

Following slice 3, **hash element assignment** inside `start` blocks now goes through shared cells.

```raku
await (^50).map: -> $i { start { %h{$i} = $i * $i } };

```

raku says 50. mutsu now deterministically matches. Cross-thread sharing of `@a.push` already worked
(`push_to_shared_var`), but **element/index assignment** — `@a[i]=` / `%h{k}=` — was broken.

- `Interpreter::assign_hash_elem_to_shared_var` (next to `push_to_existing_shared_array`) holds the

  mutates in place, dirty-marking once via the `__mutsu_shared_dirty::` marker. The parent updates env as well.

  `exec_index_assign_expr_named_op`. It applies the same simplicity guard as `try_fast_hash_element_assign`
  (rejecting type/key constraints, default, bound index, complex indexes) and passes through if the hash is not shared.

Single-threaded (`shared_vars_active == false`) completely skips the new path, so behavior is unchanged. Verification:
`t/concurrent-hash-assign.t` (new, 7: 50 distinct keys, value correctness, pre-seed survival, 20×50 heavy contention 1000),
`make test` green, `roast/S02-types/hash.t` 112/112.

### Resolving VM↔Interpreter duplicated implementations (dedup) — policy clarification + first batch

**User policy clarification (2026-06-07)**: the true purpose of VM decoupling is neither perf nor env_dirty, but **eliminating the
maintenance cost of the same logic being implemented twice, in the VM (`vm/` + native `builtins/`) and in the tree-walking Interpreter (`runtime/`)**. The real harm is
"AI/humans get misled when studying the code, fix the wrong copy, or update only one side". Whether the fallback actually fires (measured at
roughly 0%) is not the point — **the very existence of the duplication** is the problem. → The procedure and priority map were codified in `docs/vm-interpreter-dedup.md`,

### Lever C: closure upvalue-ification (per-iteration / container capture)

Raku closures capture the **binding (container)** of free variables, and a `my` in a loop body becomes a
fresh binding on each iteration. mutsu lazily read free variables **by name**, and body-local `my` occupied a
VM slot while `sync_env_from_locals` re-injected all slots into env before every call, so the
loop's final value overwrote each closure's captured value.

- **Slice 0 (#2663)**: per-iteration capture of the **loop variable** in `for 1..N -> $i`. The int-range fast path did not
  save/restore named params, so `$i` leaked past the loop.
- **Slice 1 (#2668, owned_captures)**: per-iteration capture of loop **body-local** `my` (all loop kinds).
  Body declarations are registered in `VM::loop_local_vars`; at closure creation, loop-local free vars are recorded as
  `SubData::owned_captures`, and at call time the closure **override-reads them from its own frozen env**,
  immunizing against slot re-injection. `for 1..3 { my $x=$_; @c.push({$x}) }` → `123`.
- **Slice 2 (#2669, box-on-capture)**: for loop body-locals, closures capture the **container, not the value**.
  Captured loop-body-local scalars of closures created inside a loop body are replaced with a shared `ContainerRef` in both
  slot and env before the env snapshot. This makes same-iteration mutation **after** capture visible
  (`for 1 { my $x=1; my $c={$x}; $x=2; $c() }` → `2`), and sibling-closure sharing within the same iteration also works.
  Per-iteration freshness is maintained by also clearing the slot's stale ContainerRef at vardecl time. Along the way, also fixed
  the missing ContainerRef deref in Pre{Inc,Dec} (string decrement of `--$b`) and the for-loop rw writeback
  `$pair.value` clobber. **The scope was limited to loop-body-locals** — boxing all captured scalars caused
  non-loop captures (Test's `lives-ok {}` capturing surrounding `$obj`/type objects) to hit the many paths that do not handle ContainerRef
  (immutability, type dispatch, `.kv` rw writeback, etc.) and break widely.
- **Slice 2b (precision)**: introduced the compiler analysis `CompiledCode.captured_mutated_locals` (own locals that are captured by a nested
  closure AND mutated outside their declaration; declaration and reassignment are both `SetLocal` but distinguished by the preceding `MarkVarDeclContext`),
  limiting boxing to `loop_local_vars` ∩ `captured_mutated_locals` (read-only loop locals are left to
  owned_captures, avoiding wasted boxing). In closure dispatch, `data.env`'s ContainerRefs are overwritten at merge time
  and cap_overrides skipped → **repeated mutation** of a loop local is correctly visible (... `$x=2; $c(); $x=5; $c()` → `2,5`).
- **Remaining (deferred, large)**: general non-loop container capture (`my ($g,$s)=mk(); $s(42); $g()` → `42`, top-level
  mutation-after-capture). captured_mutated_locals can detect non-loop cases too, but boxing non-loop captured scalars hits
  the ContainerRef-unaware paths (typed `.=`, isa-ok, subset-typed assignment, submethod state, Mix immutability).
  It needs a deref audit of all lvalue/type/dispatch paths (the heart of the container architecture) and will be done in dedicated PRs.
- **Slice 3 (root fix for loop body-local `my` clobber)**: fixed the **closure-independent correctness bug** where
  `my $x=99; for/while/loop/repeat { my $x=... }; say $x` clobbered the outer `$x` and printed the final value (raku says `99`).
  The dual-store env is name-keyed, and the loop body makes all locals `needs_env_sync`, so the inner `my $x`'s write-through
  overwrote the outer one in both env["x"] and the shared slot. **The fresh-slot (compiler) approach was rejected** — multiple same-named slots
  make the name-based box-on-capture (lever C) analysis misapply across blocks and break closure capture. Adopted **runtime restoration**:
  only for `my` names that shadow an existing outer binding in the loop body, record the outer value in `loop_local_saved_env` at declaration time,
  and restore into env + the shared slot on loop exit. `my` in a loop **condition** (including the `... until COND` statement-modifier form) is enclosing-scoped,
  so it is excluded via `loop_cond_active` (avoiding an `iterator-regressions.t` regression). pin `t/block-local-my-scope.t`.
  Remaining: (1) → **resolved by Slice 3b (below)**. (2) post-loop reads of a pure leak with no outer binding (raku: undeclared
  error) yield an undefined value in mutsu (strict-vars unimplemented, globally unenforced, no practical harm).

- **Slice 3b (root fix for `if`/`unless`/`else` body-local `my` clobber)**: fixed the bug where `my $x=99; if c { my $x=5 }; say $x`
  gave mutsu `5` where raku gives `99`. Added the new opcode **`BlockLocalScope { body_end }`**, wrapping branches that directly
  declare block-local `my` (determined by `branch_declares_block_local` — direct + `my` inside `SyntheticBlock`;
  state/our/dynamic excluded). The VM handler `exec_block_local_scope_op` executes the body once with the same
  `push_loop_local_scope`/`pop_loop_local_scope` (**shadow-only restoration**) as loop bodies. The BlockScope (full env restoration) approach
  tried and withdrawn in Slice 3 was rejected because it rolls back `:=` (binding to outer variables) inside the branch — shadow-only
  restores only the `my`-declared shadow names and never touches `:=`, so the `S03-sequence/exhaustive.t` regression does not occur. Since the branch is
  treated as a loop body, closure capture of if-branch locals (owned_captures / box-on-capture) becomes correct at the same time
  (`if True { my $cx=5; $c={$cx} }` → `$c()`=5, outer `$cx`=1). pin `t/block-local-my-scope.t` (19→24 cases).
  With this, **all the clearly-bounded slices of Lever C (1/2/2b/3/3b) are complete**. The only remainder is general non-loop container capture,
  which will be absorbed into and solved by 🟣 second priority "first-class containers" Phase 1.

## First-class containers Phase 0 — decont helper consolidation (behavior-invariant, #2736/#2737/#2738)

Kickoff of the PLAN.md 🟣second-priority "first-class containers (container identity)" work. To avoid the
"deref everywhere" regressions that the past ContainerRef prototype hit, we started with a behavior-invariant
refactor that **consolidates the scattered decont (de-containerization) expansions into canonical helpers**.
Implementation log: [docs/container-identity.md](../docs/container-identity.md).

**The key design decision: do NOT fuse the 3 axes.** mutsu's decont is fragmented across **3 distinct axes**:
(1) the Scalar `$(...)` itemization wrapper (recursive), (2) the ContainerRef `:=` shared cell
(single-cell read-through), and (3) the ArrayKind itemization flag (list flattening — a different kind
entirely). A single `decont()` that strips everything at once would break the Pair-kind detection for
`is rw` writeback and the `@a=$l` flattening (reduce.t 62, targeted for Phase 1), so we settled on
**placing a canonical helper on each axis** and consolidating into those.

- **PR1 (#2736)**: renamed `Value::decontainerize` → `descalarize` (to distinguish the axis from
  `ArrayKind::decontainerize`), added the owned recursive variant `into_descalarized`, and established
  the decont family doc. Removed the duplicated recursive `strip_scalar`, consolidated 4 recursive
  `while let Scalar` loops in `methods.rs`. Made the difference between `OpCode::Decont` (single level)
  and the recursive helper explicit.
- **PR2 (#2737)**: consolidated recursive ad-hoc expansions in single-Scalar-arm functions (bag/mix multiply
  in ops, coerce_to_hash/numeric/set/quanthash in utils, 1arg/2arg in methods_narg). Single-level,
  `.VAR`-guarded dispatch, mixed-axis (eqv/truthy/isa), and exhaustive matches are intentionally kept inline.
- **PR3 (#2738)**: the ContainerRef read axis. The existing `deref_container` always clones, which turned out
  to be unsuitable both for in-place reads (locking and using `&inner`) and for hot read opcodes. Introduced
  the **non-cloning canonical reader `Value::with_deref<R>(&self, f) -> R`**, redefined `deref_container` on
  top of it, and consolidated 6 in-place read sites
  (value_type_name / dispatch_what / to_string_value / eqv, truthy, isa_check, what_type_name).

All confirmed behavior-invariant (`make test` PASS 5274; the binding/flatten/set/bag/mix/eqv roast files match

## Regex validator/matcher double implementation unified ([ANALYSIS.md](../ANALYSIS.md) §3.1)

mutsu had Raku regex grammar knowledge **independently encoded in 2 places**:
`src/regex_validate.rs` (1108 lines, parse-time syntax validation only — builds no AST, just throws
`X::Syntax::Regex::*`) and the structural parser in `src/runtime/regex_parse.rs` (`parse_regex`, the real
matcher). The two implemented quantifier, character class, escape, and anchor rules separately, with 0
"keep in sync" comments — drift where one side is fixed and the other left behind was a breeding ground
for bugs (PLAN.md "🔴top priority: 1 operation = 1 implementation").

**Unified on the structural parser as the sole grammar authority and deleted `regex_validate.rs`.** Method:

- **Introduced parse modes**: threaded `RegexParseMode { Match, Validate }` through `parse_regex_uncached`.
  `Match` is the traditional runtime path (interpolation, grammar token resolution). `Validate` is a
  parse-time dry run that skips interpolation, LTM expansion, and grammar token resolution, and accepts
  variable references as opaque atoms (variable values do not exist at parse time).
- **Parse-time validation as a free function**: the parser (`parser/primary/regex.rs`) has no Interpreter,
  so introduced `validate_regex_structurally(pattern) -> Result<(),RuntimeError>`. It reuses a thread-local
  validation Interpreter as the `&self` receiver (avoiding a heavy `Interpreter::new()` per literal),
  structurally parses in `Validate` mode, and picks up `PENDING_REGEX_ERROR`. The most important correctness
  rule: "`None` but no pending error" = "not a syntax error (the old validator also accepted it)" → `Ok(())`.
- **Ported the validator-only checks into the structural parser** (all gated on `Validate`; `Match` behavior
  is byte-identical): orphaned quantifier / non-quantifiable (anchors, code blocks) / unknown metacharacter
  `-`!` / unknown modifier `:11` / orphaned backtrack control `:` / old P5 escapes `\A`\Z`\z` / unknown
  backslash sequences / P5 character range `a-z` / reversed-range and NFG range endpoints / missing operator
  in composite classes `<[a][b]>` / `<sym>` outside proto / attribute interpolation `$!attr` /
  hash variable `%var` reserved / long-name alias `<::IO::File=bar>`.
- **Also fixed latent bugs found via the drift**: runtime resolution errors such as "No such method" are
  gated to `Match` only (`dies-ok` expects a runtime death, so they must not become compile errors), and LTM
  expansion is `Match`-only (avoiding false positives where expansion products trip parse-time checks).

`make test` PASS, 2 unit tests added (accept/reject for `validate_regex_structurally` plus exception types).
The regex-related roast files (S05-metasyntax / S05-metachars / S05-mass / S05-interpolation) match the baseline.

### First-class containers: hash itemization flag (mirroring ArrayKind, #3151)

Resolved the root cause of `my %c = (%h,)` / `%(%h,)` aborting with "Odd number of elements". mutsu could not
distinguish a bare `%h` (flattens to pairs in list context) from a `$`-derived itemized hash `$hashitem` (opaque —
Raku throws X::Hash::Store::OddNumber), and the old heuristic (flatten only when element count > 1) was wrong in
both directions.

For arrays, mutsu already solves this leak-free with **`ArrayKind::ItemList`/`ItemArray`** (a flag attached to the
value; value operations ignore it, only list-flatten looks at it). Hashes had no counterpart, and the previous
attempt #3147 relied on a leaking `Scalar(Hash)` wrapper that broke many consumers (set membership, hashes held as

- `HashData.itemized: bool` (excluded from `PartialEq` — only `map` compares it). `Value::item()` and the

  preserved). **The value stays `Value::Hash` → no wrapper leaks into any consumer**. Set/Bag/Mix have no itemized
  kind, so they keep the `Scalar` wrapping as before, but only on the `ItemizeVar`
  (`@a=$var`) path (wrapping on plain `Itemize` as well regressed set_union.t with a leak; detected locally and fixed).
- `build_hash_from_items` (the single chokepoint for `%m=(...)` and `.hash`/`%(...)`) and `value_to_list` (the
  chokepoint for list/array flatten) flatten only bare Hashes; itemized ones are treated as a single element.
- **Verification**: full whitelist, 1272 files / 207429 tests, zero leaks. set_elem.t/categorize.t/map.t (broken by
  #3147) and context.t (`[item %h]` non-flatten) are all green. `t/hash-itemization-flag.t`(18).
- **Impact**: resolved the test-37 abort in objecthash.t → all 62 now run (remaining 7 = object-hash type

### 2026-06-16: panic→`X::` boundary for `start{}`/Promise worker threads (ANALYSIS rev2 §8.7 (2))

The body of `start { ... }` / `Promise` runs on a worker thread via `call_value`, outside the main thread's
`run_top` panic boundary (#3045). So when user code inside a worker triggered a Rust panic (integer overflow /
capacity overflow / index OOB, etc.), only that thread died silently and **the Promise stayed unresolved forever,
hanging `await`** (under `panic=abort` the whole process aborts).

- Added a reusable helper `guard_worker_panic` to `vm.rs` (sharing the same
  `install_vm_panic_hook` + `IN_VM_PANIC_BOUNDARY` + `catch_unwind` + `vm_panic_error` as
  `run_inner_guarded`/`run_range_guarded`). Since workers are separate threads, the thread-local boundary flag is
  raised independently (the panic hook reads the panicking thread's flag and suppresses the backtrace dump).
- Wrapped the worker body of `spawn_callable_promise`, `thread_interp.call_value(block, vec![])`, in
  `guard_worker_panic`. A caught panic becomes a catchable `X::AdHoc` (`Internal error: ...`) `RuntimeError` and
  turns into a broken Promise via the existing `promise.break_with` path → `await` rethrows it as a normal
  `X::Await::Died` (no hang).
- ANALYSIS rev2 §8.7 (1) (infinite Range `.Supply` coercion aborting on an unguarded `i64::MAX` collect) was already
  resolved by an earlier PR routing it through the capped `value_to_list` (`t/supply-infinite-range.t`). This PR
  fully closes §8.7.
- **Verification**: new `t/start-panic-boundary.t`(6). The concurrency regression suite (vm-panic-boundary /
  start-await / await-died-exception / thread / promise-*, etc.) all PASS.

## 2026-06-21: Template::Mustache delimiter change `{{=<% %>=}}` — reduce-time `$*` dynamic-variable actions (#3390)

**Implemented Mustache's last-tier blocker, "delimiter change mid-match".** With `{{=<% %>=}}`, the tag's finalizer
rewrites
`($*LEFT,$*RIGHT)=@delim` so that subsequent tags match with the new delimiters. mutsu ran actions post-parse, so
the write had no effect on the live match. Since the regex match engine takes `&self` and cannot modify the env
mid-match, introduced a
**parse-scoped thread-local overlay** (`REGEX_DYNVAR_OVERLAY`): after each iteration of `<subrule>*`/`+` is
finalized, its action runs in a scratch interp (the actions object is **deep-copied** to isolate self mutation), and
the changed `$*` variables are
published to the overlay. `interpolate_regex_scalars` consults the overlay before the env when resolving the `$*`
twigil. Thanks to a **SEEN gate**
(set on the first `$*` interpolation), ordinary grammars that don't depend on dynamic variables pay zero cost.

3 pitfalls: (1) `<x>*` in a `token` (ratchet) takes the fast path where the hook cannot attach — only `regex` uses
the general path (mustache TOP/hunk are
`regex`); (2) the candidate-selection pass polluted the overlay so the real match started in a polluted state →
inserted `dynvar_overlay_reset_scan()` immediately before the parse driver's real match
(placing it inside the regex entry would reset per subrule match and destroy in-scan state); (3) an independent
pre-existing bug: interpolated scalars should match literally, but `escape_regex_scalar_literal` left `%`/`&`
unescaped → the `%>` delimiter fused with the preceding `\h*` and was misparsed as
`\h* % ...` (a separator quantifier). Added `%`/`&` to the escape set (multi-character `<% %>` delimiters were a
blocker on their own).

**Results**: Template::Mustache **01-basic 8/10 → 10/10** (delimiter change + `{{=delim=}}` substitution),
**13-pragmas** all pass.
No regressions (all 91 whitelisted S05 + S03/S32-str + make test 9763). pin: `t/grammar-reduce-time-dynvar.t`.

## 2026-06-21: Template::Mustache fully working — cross-frame resumable warn (06-logging, #3395)

**Resolved Mustache's last blocker, 06-logging; Template::Mustache is now fully working.** All tests pass except
91/92-specs, which depend on the external `JSON::Fast` (06-logging 3/3, 01-basic 10/10, 02-file 8/8, 03-cascade 5/5,
11-iterable 6/6,
12-inheritence, 13-pragmas, 50-readme 4/4, and more).

**Problem**: when a `warn` thrown in a deep frame (render→…→`&warn.()`) was caught by a unit-level
`CONTROL { default { …; .resume } }`, mutsu unwound the warn signal as a Rust `Err` up to the CONTROL frame. The
deep frames in between were discarded, so a single `resume_ip` (one frame-relative IP) could not re-enter the
unwound depths, and the computation after the warn was silently lost (render was cut short).

**Solution = execute resume_safe CONTROL inline at the raise point**:
1. **At compile time**: determine from the AST whether a CONTROL block is "an unconditional `.resume` (no
   `when`/`succeed` escape)" and bake it into
   `OpCode::TryCatch{resume_safe}` (the runtime cannot see the AST; `control_block_is_resume_safe`).
2. **At runtime**: added `control_handlers: Vec<ControlHandlerEntry>` to the Interpreter, kept fully in sync with
   `control_handler_depth`
   (both push/pop points of `exec_try_catch_op_inner`). A resume_safe entry holds its own `Arc<CompiledCode>` +
   range + function table
   (the clone cost occurs only on the rare CONTROL entry).
3. **`builtin_warn` (depth>0)**: if the innermost handler is resume_safe, run it inline at the raise point
   (`try_resume_safe_control_inline`) and return
   `Ok(Nil)` → the deep code resumes in place. **The key**: the handler's bytecode references the local slots of the
   installing frame, but
   the current `self.locals` belongs to the deep frame. So the handler's locals are reconstructed from the env (the
   cross-frame-visible store; env keys = `code.locals` names), the handler runs, only the changed slots are written
   back to the env, and `env_dirty` is set
   → the installing frame picks them up on call return via `reconcile_locals_from_env_at_site` (`$out ~=` survives
   past the render). Non-resume_safe `when CX::Warn { }` keeps the traditional unwinding path (the implicit
   `succeed` escapes to the block boundary — the killer in #3372).

**Results**: `t/warn-cross-frame-control-resume.t` (7 cases, rakudo-verified). All #3372 regression guards green
(control/warn/catch/top-level/
indent/quietly/gather). make test 9819 all pass. This completes Template::Mustache in PLAN.md §H.
