# mutsu — 2026年6月の実装状況

## 概要

6月は **バイトコード VM を tree-walking Interpreter から切り離す（PLAN.md 🔴 最優先）** に集中。
mutsu の VM は実態として Interpreter を共有実行状態 + フォールバック先として使っており
（[ANALYSIS.md](../ANALYSIS.md) §1）、これを strangler-fig 方式で段階的に剥がしている。

**重要な方針（ユーザー指摘）:** これは結合削減＝アーキテクチャ改善が目的で、パフォーマンス改善ではない。
CI（`make test` + 包括的 `make roast`）が全マージをゲートするので、本質的リファクタは大胆にやり CI を
安全網にする（CLAUDE.md「Refactor boldly」セクションを追加）。

## ハイライト

### 計測の強化: フォールバックの名前別 histogram

`MUTSU_VM_STATS=1` が関数・メソッドそれぞれについて「どの builtin/method が interpreter に
フォールバックしたか」を件数順に出力するようになった。推測でなくデータで最大の結合源を特定できる。

### レバー A: ディスパッチの native 化

- **`sprintf`/`zprintf`** を VM の native_function テーブルへ (#2601)。`builtin_sprintf` は `&self` 不使用の
  純粋関数だった。sprintf.t の function fallback **96% → 3.5%**、実スクリプトでは **0%**。
- **属性アクセサ読み** `$obj.x` を共有ヘルパ `try_fast_accessor_read` に抽出し mut/非mut 両 opcode で
  native 化 (#2604)。変数上のメソッド呼びは CallMethodMut にコンパイルされるため、従来は全アクセサ読みが
  フォールバックしていた。method-call.raku の method fallback **60% → 20%**。private 属性漏洩の潜在バグも修正。
- **`.new`（デフォルトコンストラクタ）** の単純構築 fast path を `try_native_default_construct` で VM 直呼び
  (Step 5)。method-call.raku **20% → 0%**、bench-class **13.8% → 0%**。
- **`.map` 単純ブロック** を `vm_native_map.rs::try_native_array_map` で native 化 (#2619)。real array・
  単一 `Sub`・body whitelist スキャナ（escape/`$_` mutation/Pair を弾く）に限り `vm_call_on_value` で
  VM 内反復。probe の map fallback **50 → 0**。

#### クロージャ/正規表現を取るメソッドの第2弾（ユーザー依頼「subst/sort/map remainder/Test を一個ずつ全部」）

これらの「フォールバック」はブロック本体ではなく orchestration の入口のみ（本体は既に VM コンパイル実行）。
スタック PR（subst ← sort ← map ← test、各マージ後 `git rebase --onto origin/main` で前 commit を落とす運用）で実装:

- **`.subst`** を native 化 (#2621)。Str invocant・plain `Value::Regex`/`Value::Str` パターン・Str(or無し)
  置換・`:g` のみ。演算子形 `s///` の VM 部品（`regex_find_first_from_with_captures`・`expand_capture_refs`・
  Match 構築）を再利用し、`$/` ポリシーまで `dispatch_subst` と byte 一致。closure 置換/RegexWithAdverbs/
  Perl5/その他 adverb は fallback。最頻の subst method fallback を解消。
- **`.sort`** を native 化 (#2622)。plain `Array`/`List`・比較子なし(`compare_values`)or `detect_simple_cmp_block`
  該当(`{$^a<=>$^b}`/reverse/`cmp`)を inline 比較。interpreter の fast-path 検出を `pub(crate)` 共有し
  **perf 中立**（interpreter も既に block 呼びを回避済み）。method-keyed/mapper/Routine/`:k`/Shaped 等は fallback。
- **`.map` 多 arity + pointy ブロック** (#2623)。`-> $a,$b` / `{$^a+$^b}` を arity チャンクで反復。
  ついでに **pointy `Lambda` body の先頭 `SetLine` を scanner が拒否していた既存制約も修正**し、単一 arity
  pointy も native 化。短い末尾 chunk（`len % arity != 0`）は interpreter の error 経路へ fallback。
- **Test 関数のディスパッチ層** を VM 化 (#2625)。`is`/`ok`/`plan`/`is-deeply`/… を `call_function` の
  giant match を経由せず `call_test_function` へ直接ルート。**スコープは正直に「ディスパッチ層のみ」**:
  本体は TAP 状態・I/O・method dispatch・eval に本質的に結合（TAP 状態所有の移管はレバー B の課題）。
  正しさ3点 = (1) ゲートは `test_module_loaded()`（最初の `plan` 前でも真）、(2) builtin と衝突する名
  （特に `run` = Proc spawner）は `is_builtin_function` で除外、(3) user-func 解決後の位置なので `sub ok`
  シャドーイング尊重。whitelist の `line-numbers.t`（test 内で builtin `run` を多用）が `run` 誤ハイジャック
  回帰を検出→修正。
- **`.grep` は native 化せず**: 結果が元配列への rw ビュー（`@a.grep(...)>>++` が元更新）で clone 不可。

#### レバー A 終着: `EVAL` / pseudo-package を carrier として分類

レバー A 最後の未完了項目（`EVAL` / symbolic deref / `CALLER::`）を解決し、レバー A を**終着**。
これらは **tree-walk フォールバックではない**: `EVAL`/`EVALFILE` は既に `eval_block_value` で
compile→バイトコード→サブ VM 実行（`run_compiled_block`/`VM::new`）し eval 対象は tree-walk されない。
pseudo-package 読み（`CALLER::`/`OUTER::`/`SETTING::`/`DYNAMIC::`）も reflective な env lookup。両者は
interpreter を**キャリア**として使うだけ（compile/lookup に必要な env/classes/roles レジストリ所有のため）。
これらをフォールバック計上していたのは計測ノイズだった。

- `VM::is_interpreter_carrier_function` を新設し、2つの interpreter 入口（`call_function_compiled_first` と
  `dispatch_func_call_inner` 最終 else）で `vm_stats::record_function_carrier` に振り分け。
- `MUTSU_VM_STATS=1` が carrier を別バケット報告（`interpreter_carrier=N` + `function-carrier by name`）。
- t/ 集計で関数 tree-walk-fallback **18.8% → 18.6%**（EVAL 54 件が carrier へ移動）。残るフォールバックの
  支配項は `__mutsu_atomic_post_inc_var`（atomic `$x⚛++`）= concurrency/共有状態 = **レバー B 領域**。
- pin: `cargo test carrier_functions_are_not_tree_walk_fallbacks`。`EVAL` の残結合（共有 env/レジストリの
  Interpreter 所有）は明示的にレバー B の課題。

### レバー B: locals↔env 二重ストアの解消

- 序盤スライス（base-tier 化・closure capture を free-var に限定・format!/Symbol churn 除去）に続き、
  **Slice 5 step 1** (#2608): slot-only local（GetLocal でしか読まれない、例: 再帰関数の param）を
  Interpreter env にミラーしないよう `ensure_env_synced` を `needs_env_sync` でゲート。
  「locals が権威、env は名前経由リーダーが必要とするものだけ」への前進。fib で per-call の
  flushed-slot が 635593→0（param が interpreter env を越えなくなった）。ループ/ブロックは制御
  テンポラリの env 往復のため保守的に全 sync（needs_env_sync の不完全性への対処）。

- **Slice 6（scoped overlay）→ 6.1/6.2（dirty-flag 機構撤廃）→ multi-tier overlay #2684**:
  per-call deep copy・per-nested-call O(env) flatten を撲滅（`mb_nested` env_deep_copies 10001→1）。
- **Slice 6.3 stage 1/1b（#2689/#2692）— メソッド dispatch 毎回の env→locals pull を撲滅**:
  opcode の無条件 `env_dirty=true` を `method_dispatch_pure` フラグ（default false=保守的、純粋証明時のみ
  mark 省略＝取りこぼしは余分な pull 止まりで stale read なし）に置換。compiled/native/merge 各パスが精密に
  純粋性を報告し、`merge_method_env` は `cheaply_unchanged`(Arc-ptr/scalar 等価)＋`could_name_caller_local`
  (`?LINE`/`?FILE`/`*`動的/`__mutsu_` を除外)で精密 changed-flag を返す（ネスト呼びの overlay flatten と
  `?LINE` 行差による偽陽性を解消）。fast accessor read（`$obj.x`）と native `.new` も env-pure 化。
  **結果: benchmarks/method-call 20000→1, bench-class 14001→2, mb_method/mb_nested 10001→1。**
  make test PASS、pin `t/method-env-dirty.t`(24)。env_dirty フラグ自体の削除は残り（regex `~~` の by-name
  書き込み step2 / I/O+carrier step3、後者は interpreter bridge 撤去待ち）— 詳細は docs/vm-dual-store.md の
  「NEXT SESSION — resume map」。教訓: caller env の Arc を保持して比較する案は refcount を上げ make_mut を
  誘発して失敗→「dispatch 内から純粋性を報告」が正解。計測は必ず benchmarks/*.raku の実ベンチで。

- **Slice 6.3 step2 — dispatch fast path の per-call pull 撲滅（3経路）**: stage 1/1b で残っていた dispatch
  経路の毎回 pull を精密化（いずれも `for ^5000 {...}` で `locals_pulls` 5001→1〜2）。
  ①**0-arg fast path**: `call_compiled_function_fast`（vm_call_dispatch.rs）が 0-local 関数で merge 経路を持たず
  呼び出し元の blanket `env_dirty=true` に依存していた。コンパイル時 `cf.code.has_env_writes` ゲートで精密化
  （env 書き込みの無い本体は caller slot を汚せない＝dispatch の routine `_` も復元されるので pull 不要）。
  ※0-local に scoped overlay を被せる案は routine `_` 挿入が make_mut を誘発し env_deep_copies 1→5000 で逆効果（不採用）。
  pin `t/zeroarg-env-dirty.t`（16）。
  ②**heavy named path**: 複雑シグネチャ（default param / return type / where / typed / multi）が通る
  `call_compiled_function_named` を `call_compiled_function_fast` と同型に精密化（env_dirty を保存→body 前リセット→
  return merge で `env_dirty = saved || changed`。changed＝plain-lexical/captured-outer writeback の実変化、非空の
  `is rw` writeback、`is raw` return のいずれか。動的変数 writeback は slot を持たないので除外）。hot call 元
  （heavy/OTF/cached named、bareword-term named）の blanket マークを撤去。EVAL/carrier 元は step3 なので保守的に維持。
  pin `t/named-call-env-dirty.t`（16）。
  ③**native `@a.push`**: vm_data_ops の push fast path は target 配列のみを env で変異させ結果を直前で local slot へ
  復元済み（reverse-write-through）なので blanket マークは冗長。撤去（`@a.push($_)` in loop は頻出）。
  interpreter-fallback push 枝（shared/shaped/非単純配列）は保守的に維持。pin `t/push-env-dirty.t`（13）。
  make test PASS（prove t/ 591）、cargo test 456、clippy clean。

- **Slice 6.3 step2 — regex/smartmatch の pull 撲滅（正攻法で完遂）**: `exec_smart_match_expr_op` の per-match
  `env_dirty=true` を精密化（bench-string per-match 5001→1）。最初の naive な無条件撤去は whitelisted 回帰
  （`S02-magicals/sub.t` の `regex foo { a { $var = &?ROUTINE } }` 埋め込みブロックが caller local 書き込み、
  `S05-modifier/pos.t` の `my $/`+`:pos` 継続マッチで `$/` がローカル）を出した。原因は **regex RHS が埋め込み `{}`
  ブロックを interpreter regex エンジン経由で実行し任意の caller local を名前書き込みする**こと（VM op を通らず
  env_dirty が立たない）。**正攻法 = 副作用を精密にシグナル**: subagent でエンジンを精査し、埋め込みブロックが
  既に `Interpreter::pending_local_updates` に書き込み変数を記録している（が誰も消費していない）ことを発見。
  env_dirty を skip するのは全条件成立時のみ — ①compile-time `rhs_pure_regex`（plain `Value::Regex` リテラル＝
  RegexWithAdverbs/`:pos`/named/Sub/value-smartmatch を除外、`OpCode::SmartMatchExpr` の新フィールド）、②runtime で
  `pending_local_updates` が空（埋め込みブロックが caller var を書いていない。op が match 前にクリア・後でチェック、
  ついでに従来 unbounded だった leak も解消）、③subst/translit でない、④`$/` がローカルスロットでない。lhs alias は
  slot へ reverse-write-through。pin `t/smartmatch-env-dirty.t`（18: pure match / 埋め込みブロック書き込み / named
  regex / `my $/` 継続 / 置換 / given-when / RHS-sub 副作用）。これで **Slice 6.3 step2 の hot な per-call/per-match
  pull は dispatch 4経路 + smartmatch すべて撲滅**。残りは真に cold な setter（register trait-handler / `:=` 要素
  SlotRef / loop-control 往復）と step3 carrier のみ。

### VM↔Interpreter 重複実装の解消（dedup）— 方針の明確化 + 第1バッチ

**ユーザー方針の明確化（2026-06-07）**: VM decoupling の真の目的は perf でも env_dirty でもなく、**VM(`vm/`+native
`builtins/`) と tree-walking Interpreter(`runtime/`) に同じ処理が二重実装されている保守コストの解消**。実害は
「AI/人がコードを調べる時に惑わされる・間違った方を直す・片方だけ更新する」。fallback が実際に発火するか（計測上
ほぼ 0%）は本質ではなく、**重複の存在そのもの**が問題。→ 手順・優先マップを `docs/vm-interpreter-dedup.md` に明文化、
方針を PLAN.md「最終ゴール」とメモリ（feedback_dedup_over_perf）に記録。

**第1バッチ**: 純粋値 builtin 9個の Interpreter コピー（`builtin_abs`/`lc`/`uc`/`tc`/`trim`/`flip`/`chr`/`ord`/`chars`）を
削除。Interpreter の `call_function` catch-all → `call_function_fallback` → `crate::builtins::native_function` で native
にフォールスルーするので、arm + `builtin_*` 本体を消しても EVAL/正規表現/carrier 経路は native を使う（EVAL 経路で
raku と同値確認、make roast PASS）。**教訓（subagent の誤分類を検出）**: native arm の存在は必要条件だが十分でない。
VM の `try_native_function` と interpreter fallback の `native_function` は**カバレッジが違う** — `chrs`/`ords` は native
arm があるのに fallback 未到達で `EVAL(q{chrs(72,105)})` が "Unknown function" になったため**残した**。`words` も IO 版
（ハンドル読み）で純粋 `.words` と別物のため対象外。**EVAL 経路での同値確認を必須手順とする**。残: Category B
（genuine fork: min/max/sort/join/first…）、Category C（method/arith 重複）。

### レバー C: クロージャ upvalue 化（per-iteration / コンテナ捕捉）

Raku のクロージャは自由変数の**バインディング（コンテナ）**を捕捉し、ループ本体の `my` は
反復ごとに fresh binding になる。mutsu は自由変数を**名前で遅延読み**し、body-local `my` が
VM slot を占有 + `sync_env_from_locals` が毎呼び出し前に全 slot を env へ再注入するため、
ループ最終値が各クロージャの捕捉値を上書きしていた。

- **Slice 0（#2663）**: `for 1..N -> $i` の**ループ変数** per-iteration 捕捉。int-range fast path が
  named param を save/restore しておらず `$i` がループ後に漏れていた。
- **Slice 1（#2668, owned_captures）**: ループ**本体ローカル** `my` の per-iteration 捕捉（全ループ種）。
  `VM::loop_local_vars` に本体宣言を登録、クロージャ生成時に loop-local な free var を
  `SubData::owned_captures` として記録し、呼び出し時に**自分の凍結 env から上書き読み**して
  slot 再注入に免疫化。`for 1..3 { my $x=$_; @c.push({$x}) }` → `123`。
- **Slice 2（#2669, box-on-capture）**: ループ本体ローカルについて、クロージャは**値でなくコンテナ**を捕捉。
  ループ本体内で生成されるクロージャの被捕捉 loop-body-local スカラを slot+env 両方で共有 `ContainerRef` に
  置換してから env スナップショット。これで捕捉**後**の同一反復内変異が見え
  （`for 1 { my $x=1; my $c={$x}; $x=2; $c() }` → `2`）、同一反復内の兄弟クロージャ共有も対応。
  per-iteration freshness は vardecl 時に slot の stale ContainerRef もクリアして維持。付随で
  Pre{Inc,Dec} の ContainerRef deref 欠落（`--$b` の文字列デクリメント）と for-loop rw writeback の
  `$pair.value` clobber も修正。**スコープは loop-body-local に限定**した — 全被捕捉スカラを box すると
  非ループ捕捉（Test の `lives-ok {}` が周囲の `$obj`/型オブジェクトを捕捉）が ContainerRef 未対応の多数経路
  （不変性・型 dispatch・`.kv` rw writeback 等）を踏み広範囲に壊れたため。
- **Slice 2b（精密化）**: コンパイラ解析 `CompiledCode.captured_mutated_locals`（nested closure に捕捉され、
  かつ宣言以外で変異される own local。宣言/再代入は両方 `SetLocal` だが前置 `MarkVarDeclContext` で区別）を
  導入し、boxing を `loop_local_vars` ∩ `captured_mutated_locals` に限定（read-only ループ local は
  owned_captures に委ね無駄な box を回避）。closure dispatch で `data.env` の ContainerRef を merge 時に上書き
  ＋ cap_overrides スキップ → ループ local の**繰り返し変異**が正しく見える（… `$x=2; $c(); $x=5; $c()` → `2,5`）。
- **残（延期・大）**: 非ループの一般コンテナ捕捉（`my ($g,$s)=mk(); $s(42); $g()` → `42`、top-level
  mutation-after-capture）。captured_mutated_locals は非ループも検出可能だが、非ループ被捕捉スカラを box すると
  ContainerRef 未対応経路を踏む（型付き `.=`、isa-ok、subset 型代入、submethod state、Mix 不変性）。
  全 lvalue/型/dispatch 経路の deref 監査（コンテナ・アーキの本丸）が必要で、専用 PR 群でやる。
- **Slice 3（ループ本体ローカル `my` の clobber 根治）**: `my $x=99; for/while/loop/repeat { my $x=... }; say $x` が
  外側 `$x` をクロバーして最終値（raku は `99`）になる**クロージャ非依存の正しさバグ**を修正。dual-store の env が
  name-keyed で、ループ本体が全 local を `needs_env_sync` 化するため内側 `my $x` の write-through が env["x"] と
  共有スロットの両方で外側を上書きしていた。**fresh-slot（コンパイラ）案は却下** — 同名複数スロットが
  name-based の box-on-capture（lever C）解析をブロック跨ぎで誤適用させ closure 捕捉を壊す。**ランタイム復元**を
  採用: ループ本体で既存外側束縛を shadow した `my` 名のみ宣言時に外側値を `loop_local_saved_env` へ記録し、

## 正規表現 validator/matcher 二重実装の統合（[ANALYSIS.md](../ANALYSIS.md) §3.1）

mutsu には Raku 正規表現の文法知識が**2 箇所に独立してエンコード**されていた:
`src/regex_validate.rs`（1108 行、パース時の構文検証専用・AST を作らず `X::Syntax::Regex::*` を投げるだけ）と、
`src/runtime/regex_parse.rs` の構造パーサ（`parse_regex`／実マッチャ）。両者が量化子・文字クラス・エスケープ・
アンカー規則を別々に実装し "keep in sync" コメントは 0 件 — 片方を直してもう片方を放置する drift がバグの
温床だった（PLAN.md「🔴最優先: 1操作=1実装」）。

**構造パーサを唯一の文法権威に統合し、`regex_validate.rs` を削除した。** 手法:

- **parse モード導入**: `RegexParseMode { Match, Validate }` を `parse_regex_uncached` に通した。`Match` は
  従来の実行時パス（補間・grammar token 解決）。`Validate` はパース時のドライランで、補間・LTM 展開・
  grammar token 解決をスキップし、変数参照を不透明アトムとして受理する（パース時には変数値が無いため）。
- **パース時検証の自由関数化**: パーサ（`parser/primary/regex.rs`）は Interpreter を持たないので、
  `validate_regex_structurally(pattern) -> Result<(),RuntimeError>` を新設。thread-local の検証用
  Interpreter を `&self` レシーバとして使い回し（毎リテラルごとの重い `Interpreter::new()` を回避）、
  `Validate` モードで構造パースして `PENDING_REGEX_ERROR` を拾う。「`None` だが pending 無し」＝
  「構文エラーではない（旧 validator も受理）」→ `Ok(())` とするのが最重要の正しさルール。
- **validator 限定チェックを構造パーサへ移植**（すべて `Validate` ゲートで `Match` の挙動はバイト等価）:
  孤立量化子 / 量化不可（アンカー・コードブロック）/ 未知メタ文字 `-`!`/ 未知修飾子 `:11` /
  孤立バックトラック制御 `:` / 旧 P5 エスケープ `\A`\Z`\z` / 未知バックスラッシュ列 / P5 文字範囲 `a-z` /
  逆順範囲・NFG 範囲端点 / 複合クラスの演算子欠落 `<[a][b]>` / proto 外の `<sym>` / 属性補間 `$!attr` /
  hash 変数 `%var` 予約 / 長名エイリアス `<::IO::File=bar>`。
- **ドリフトで見つけた潜在バグも修正**: 「No such method」など実行時解決エラーは `Match` 限定にゲート
  （`dies-ok` が実行時死を期待するため compile error にしない）、LTM 展開は `Match` 専用（展開産物が
  パース時チェックを誤検知するのを回避）。

`make test` PASS、ユニットテスト 2 件追加（`validate_regex_structurally` の受理/拒否＋例外型）。
roast の正規表現系（S05-metasyntax / S05-metachars / S05-mass / S05-interpolation）はベースライン一致。

  take-limit suspend 時に VM の `ip` をループ opcode に留め、locals/env 状態から条件チェックに再入。
  `.first` は 1 要素ずつ pull して副作用回数を Rakudo と一致させた。pin=`t/gather-infinite-coroutine.t`（21 cases）。
- **本丸: method 形 `.map`/`.grep` の eager materialize 撲滅 (#2804)**: 従来 `(1..Inf).map(* * 2)` は
  無限ソースを 1M cap で eager 展開していた（`.head(3)`=7s、`.elems`=1000001、`.is-lazy`=False、連鎖
  `.map(...).grep(...).head(3)`=27s）。これを **真に遅延な `LazyList`（新 `MapGrepSpec` を `lazy_pipe` に
  持つ pipeline stage）** を返し、ソースを 1 要素ずつ pull する方式に変更。
  - ゲートは **無限整数 Range / 既存 lazy_pipe への連鎖のみ** に限定し回帰最小化（gather/scan/sequence/
    有限ソースの map/grep は不変）。multi-arity/slurpy コールバック・grep adverb は eager パス維持。
  - stage 適用は **VM ネイティブ `vm_call_on_value` / `vm_grep_item_matches`**（interpreter の
    `eval_grep_over_items` は `mem::take` でネスト VM を作り外側フレームを破棄するため不可）。
    これにより map コールバックの副作用が外側スコープへ正しく伝播し、locals/env も整合。
  - strict 全要素 op（`.elems`/`.sort`/…）on 無限 pipe = `X::Cannot::Lazy` throw（raku 一致、従来 HANG）。
    遅延保持 coercion（`.Seq`/`.List`/`.list`/`.cache`/`.values`/`.lazy`）は pipe をそのまま返す。
  - `force_lazy_pipe`（`vm_helpers.rs`）を `force_lazy_list_vm_n` / for-loop / index dispatch に配線。
  - pin=`t/lazy-method-map-grep.t`（22 cases: head/first/index/連鎖/for-loop/is-lazy/X::Cannot::Lazy/有限不変）。

### ディスパッチ統一（PR-1〜4）

### Package.new コンストラクタの native 化（属性 shape 別、#2826〜2844）
§1 catch-all 最大カテゴリ `Package.new` を、**pure-data で構築できる属性 shape** から native 化。共有ゲート
`is_native_default_constructible` ＋ 構築 `build_native_default_instance`（`methods_object.rs`、VM の
`try_compiled_method_or_interpret` と interpreter の `dispatch_new` が共有）。**保守的フォールスルー**で不確実は全て
interpreter に委ね behavior-invariant（対象は raku ではなく現 interpreter）。
- **splice native (#2826)**: ctor とは別の `try_native_array_splice`。
- **typed `$` (#2833)**: `has Int $.x`。simple class 制約。
- **untyped `@`/`%` (#2834)**: `has @.items`/`has %.map`。`coerce_attr_value_by_sigil` 再利用。
- **required (#2835)**: 提供→build / 未提供→fallthrough。
- **native 型 (#2837)**: `has int $.x`。未提供→native zero、提供→そのまま格納。literal 高速パスで 20k-loop timeout 回避。
  CUnion クラスはゲート除外（byte-overlay 構築）。
- **coercion 型 (#2838)**: `has Int() $.x`。パーサ auto-default を target type object に修正＋共有
  `coerce_value_for_constraint` 経由 coercion（属性だけ未対応だったバグ修正込み）。
- **typed `@`/`%` (#2841)**: `has Int @.nums`。**provide が element 型を喪失していたバグ修正込み**（`.of`=Mu→Int）。

- **is-Type / shaped correctness fix (#2844)**: **native 化はしない**（is-Type/shaped 構築は型/Array の `.new`

  クラッシュ regression（finalize pass を leaf まで再帰）②is-Type provided coercion（`has @.a is Buf` 提供値を
  `coerce_value_to_is_type` で declared 型へ）。

**§1 ctor の結論**: pure-data で構築できる属性 shape（untyped/typed `$`・`@`/`%`・native 型・coercion 型・required）は
カバー完了。残る shape（is-Type/shaped/where/native-element-container）は**いずれも interpreter method dispatch /
user code 実行に依存**し、保守的 pure-data 原則では native 化できない → phase ④（method dispatch 自体の native 化）後。

**重要な現状認識**: §1/§2 の「安い native 化」は枯渇した。残るフォールバックは全て構造的ブロッカー
（③ state 所有 / 第一級コンテナ Phase 2 / lever B 並行）が前提。次の実質進捗はこれら 3 つの構造リファクタであり、
**互いに独立なので並列実行可能**（下記 PLAN.md の再編を参照）。

`given @a { .push(4) }` は Raku ではコンテナをエイリアス topicalize し、`$_` 経由のコンテナ変異が

- **#3015 = `given` element-source topic alias**: `given %h<k> { .push }` / `given @a[i] { $_ = v }` が
  element に伝播。**element は lvalue なので whole-container topic（read-only-for-reassign）と非対称＝完全 rw**
