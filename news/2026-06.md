# mutsu — 2026年6月の実装状況

## 概要

6月は **バイトコード VM を tree-walking Interpreter から切り離す（PLAN.md 🔴 最優先）** に集中。
mutsu の VM は実態として Interpreter を共有実行状態 + フォールバック先として使っており
（[ANALYSIS.md](../ANALYSIS.md) §1）、これを strangler-fig 方式で段階的に剥がしている。

**重要な方針（ユーザー指摘）:** これは結合削減＝アーキテクチャ改善が目的で、パフォーマンス改善ではない。
CI（`make test` + 包括的 `make roast`）が全マージをゲートするので、本質的リファクタは大胆にやり CI を
安全網にする（CLAUDE.md「Refactor boldly」セクションを追加）。

## ハイライト

### grep-rw-view 側道機構の撤去 — `.grep` を第一級要素 cell 化（#3466, 2026-06-23）

`for @a.grep(...) { $_++ }` / `@a.grep(...)>>++` の source 書き戻しは、tree-walk 時代の専用側道で実装されていた:
`for_grep_view`（Interpreter フィールド・vm.rs で save/restore）、filtered `ArrayData` に埋め込んだ
`GrepView`/`grep_source` バインディング、Arc 同一性で source array を特定する `overwrite_array_items_by_identity_for_vm`
writeback。これを全廃し、**第一級コンテナ（element cell）** に置き換えた（PLAN §C）:

- `.grep` over array が matched source slot を共有 `ContainerRef` cell に昇格し、結果が同じ cell を参照する。
  writeback ループは通常の要素 cell write パス（for-loop は既に cell 経由で書く）で @a の slot に書き戻る。
  `>>++`/`>>--` は hyper ループで cell を in-place inc。`=` 代入は cell を decont するので、名前付きコピー
  （`my @g = @a.grep(...)`）は書き戻さない。
- 結果が実 cell を運ぶため、その上の全 read パスが decont 必須に。**~30 op を実測して欠けていたのは 6 箇所のみ**
  （いずれも `:=`-bound 要素 cell の既存バグでもあった）: `compare_values`（.sort/min/max）/ `collect_minmax_candidates`
  （.minmax）/ hyper メソッドループ（`>>.method`）/ `.join` の Instance 判定 3 箇所（narg fast / runtime / 0-arg）/
  **メソッド dispatch の invocant**（`exec_call_method_op` で `ContainerRef` invocant を `.VAR` 以外 decont）/
  `smart_match` の LHS（`does-ok ... .grep(Rational).head`・roast S32-num/rat.t）。
- **★中心知見の非対称性**: CallMethod invocant の decont は安全（読み）だが **CallMethodMut は decont 不可**＝
  `my @x := @src; @x.push` の `:=`-bound 容器変異が source に届かなくなる。read dispatch だけ decont する。
  `.head`/`.first` が cell を返すのは正しい（Raku で `.head` は rw）→消費側で deref。
- net **-90 行**。pin=`t/grep-cell-read-deref.t`（19）。`for_grep_view` フィールド / `GrepView`・`grep_source`
  ArrayData フィールド / `grep_view_writes`・`grep_view_topic_value` を物理削除。CI（make test + make roast・release）green。

**★教訓**: cell 化の blast radius は GrepView（metadata のみ・read 透過）より桁違いに大きい（全 read consumer が
deref 必須）が、実測 gap は少数。事前 PLAN の「全 read パス deref 必要・単独スライス不可」は過大評価だった。
この read-deref chokepoint 整備は次の Phase-2「最終 SlotRef キル」の地ならしにもなる（同じ 6 箇所を踏む）。

### `env_dirty` dual-store 機構の物理削除 — cell-boxing が唯一のコヒーレンス機構に（2026-06-23）

#3450 で cell-boxing を恒久コヒーレンス機構にした（`blanket_reconcile_disabled()`/`precise_reconcile_disabled()`
を恒久 `true` にフリップ）後の機械的フォローアップ。tree-walk 時代から残っていた env→locals reconcile の
**dead code を物理削除**した（挙動不変・約 604 行削除）:

- 関数: `reconcile_locals_from_env_at_site`（精密 reconcile・早期 return で no-op 化済）/ `blanket_reconcile_if_dirty`
  （blanket reconcile・常に false ゲート）/ `ensure_locals_synced`（`env_dirty=false` のみ）/ `blanket_reconcile_disabled`
  / `precise_reconcile_disabled` / `cell_boxing_active`（恒久 `true` を返すゲート）。
- フィールド: `Interpreter::env_dirty`（342 uses・大半は `= true`/`= false` の no-op マーク）/ `VmCallFrame::saved_env_dirty`

### CP-3 collapse: `struct VM` を `Interpreter` へ溶かし ping-pong を撤去（PR #3102）

VM decoupling の最終ゴール本体。それまで `VM { interpreter: Interpreter, ... }` という 2 つの相互参照 struct が、
`mem::take(self)` + `VM::new(self)` + `*self = interp` の **ping-pong**（env は CP-1 で loan）で実行を再入していた。
これを撤去し、**単一 struct が bytecode VM そのもの**になるよう VM を Interpreter へ統合した。

- **方向**: survivor = `Interpreter`（`Interpreter::new()`/`.run()` が main/lib/repl/test の公開エントリのため、API・
  エントリを無改修に保つ）。`pub(crate) type VM = Interpreter` alias で ~40 個の `impl VM` を無改修化（cosmetic rename は後続）。
- VM の実行レジスタ（stack/locals/call_frames/caches/flags）を Interpreter へ統合。ping-pong 全 10 サイトを in-place
  再入へ: carrier（`run_block_raw`/`run_compiled_block`/`eval_precompiled_block_fast`）は `run_nested`、map/grep/sort の
  `run_reuse` ループは `with_nested_registers`（レジスタ save/reset/restore を抽出）、thread spawn は clone 済 Interpreter を直接実行。
- パニック境界を `run_inner_guarded` に抽出し run_top/run_nested 双方へ配線（`dies-ok`/`try` 内の Rust panic が
  try/CATCH を通る。`t/vm-panic-boundary.t` 回帰を発見・修正）。
- `loan_env!`/`loan_env_for` を thin self-call 化、重複アクセサ/ヘルパ・dead な loan/handle 機構（`loan_out_env`/`*_handle`/
  `Env::poisoned`）を削除。**net −794 行**。`env_dirty` dual-store は CP-2 通り存続。
- 検証: cargo test 462/0、フル `prove t/` 810 files / 7549 tests PASS、CI フル `make roast` PASS。
- **#3104（cleanup）**: `type VM = Interpreter` alias を完全撤去（`impl VM`→`impl Interpreter` 等）しコードベースから
  `VM` 型名を排除、dead な `Env::poisoned` ガードを除去。

**前提として landed していた CP-1/CP-2（VM decoupling の土台）**:
- **CP-1（env を VM 所有へ・env-loan flip, #3075）**: env を VM 所有へ物理移管し、env を読む全 carrier/helper（~350 サイト）を
  `loan_env!`/`loan_env_for` で貸借。poison ガード駆動 discovery で未包み env 読みを順次 wrap。
- **CP-2（dual-store）= 「機構削除は構造的に不可」と確定**。`env_dirty`/`ensure_locals_synced`/`sync_locals_from_env` は
  EVAL + slot-locals 共存下で構造的に必須（cyclic-bind clobber を `t/element-bind-cell.t` で実証・fib perf 回帰）。
  達成できたのは spurious マーク/precision の footprint reduction（#3080/#3083/#3084）のみで、機構は VM 内部最適化として**永続**。
- **結論**: 数ヶ月来の 🔴 最優先「tree-walking Interpreter 廃止 = VM decoupling」が CP-1→CP-2→CP-3 で**完結**。終状態 =
  単一 struct `Interpreter` が bytecode VM そのもの・全状態 VM 所有・`env_dirty` のみ存続。

### 安定性: VM の panic→`X::` 変換境界（ANALYSIS §2.1）

ユーザコード起因の Rust パニック（整数オーバーフロー・`Vec` capacity overflow・index OOB・`unwrap`/`unreachable!`
等）は、これまでプロセス全体をクラッシュさせていた（exit 101 / core dump）。これを VM 内の境界で捕捉し、
catchable な `X::AdHoc`（`Internal error: <payload>`）に変換するようにした。クラッシュ → 通常のエラー経路
（メッセージ表示・exit 1・`try`/`CATCH` で捕捉可能）へ。

- **二段の境界**:
  - `VM::run` を `catch_unwind` でラップ（ループ本体を `&mut self` の `run_inner` に切り出し、`self` の
    move-return を排除して `catch_unwind(AssertUnwindSafe)` 可能に）。トップレベル文（`try` の外）の
    パニックの安全網。EVAL / サブ VM も `VM::run` 経由なので自動的にカバー。
  - `try`/`CATCH` のボディ実行に `run_range_guarded` を使用。これにより、try 内で何重にネストした
    （未ガードの `run_range`/`run_reuse` を貫く）パニックも、アンワインドがこの単一境界まで巻き戻って
    `Err` に変換され、既存の例外機構経由で CATCH ハンドラへ流れる。中間フレームはガード不要。
- **perf 影響なし**: ホットなループ本体の `run_range` は未ガードのまま。境界は `try` ブロックとトップレベル
  のみ。
- **stderr のバックトレース抑制**: `thread_local` フラグ + カスタム panic フックで、境界内のパニックの
  デフォルトバックトレース出力を抑制（`RUST_BACKTRACE`/`MUTSU_TRACE` 指定時は従来どおり表示し、本物の
  内部バグのデバッグ性を維持）。
- **スコープ外**: スタックオーバーフロー（`sub f { f() }; f()`）は unwind せず `abort` するため対象外
  （別途、再帰深度ガードが必要）。巨大アロケーション（`"x" x 2**62`）も allocator abort で対象外。
- テスト: `t/vm-panic-boundary.t`（9 サブテスト — try 生存・`X::AdHoc` 化・メッセージ保持・`$!` 設定・
  `dies-ok`・深いネスト・通常配列成長は無影響）。

### トラック C: react/supply 駆動ループの単一エンジン統合（Stage 1 完了, #3027 / #3029 / #3031）

react/supply ランタイムの「駆動ループ（source 解決 → イベント poll → body 実行 → done/last/quit/close）」は
**4 箇所に二重実装**されていた（react / await-promise / tap×2）。これを単一エンジンへ統合する Stage 1 を完了。

- **react↔await-promise 統合（#3027）**: 完了ポリシーを表す `enum SupplyDrivePolicy { React, Promise }` を
  導入し、react のイベントループを `drive_react_subscriptions(subs, policy)` として抽出。`run_react_event_loop`
  は subscription を組み立てて `Policy::React` で委譲、`supply_promise_on_demand`（`await $supply`）は
  `ReactSubscription` を組み立てて `Policy::Promise` で委譲する。Promise 専用の独自ポーリングループと
  `SupplyPromiseSub` 構造体（約90行）を削除。`React` は全 subscription を `run_react_consumer`（done/next/last
  をマップ）で完了まで駆動し CLOSE phaser を発火、`Promise` は supply body の内側 `done` が supplier レジストリ
  経由で promise を keep するか deadline まで駆動し最後の emit 値で keep する。
- **ボイラープレート削減（#3029）**: `ReactSubscription` の構築8箇所が全15フィールドを毎回列挙していたのを
  `ReactSubscription::new(callback)` + struct-update 構文へ（−56行）。
- **god-file 分割（#3031）**: 3671行に肥大化した `native_supply_methods.rs` を機能別6サブモジュール
  （dispatch / supplier / supply_mut / classify / promise / transform）へ分割し、共有ヘルパー + `QuitOutcome`
  のみ189行で残した。純粋な move（メソッド数 26=26、private `fn`→`pub(super) fn` のみ）。

### トラック C Stage 2: react/supply 駆動ループの VM 所有・コンパイル済みディスパッチ（#3038 / #3039）

Stage 1 で残した「whenever body が AST のまま tree-walk される真フォールバック」を消した。鍵は **ループ所有権の
逆転**: `VM` は `Interpreter` を値で所有する（`VM::new` が move）ため、`&mut Interpreter` メソッド内では VM を
構築できず、borrow checker が「駆動ループ自体を `impl VM` へ移す」ことを強制する。

- **PR1 = 純粋な移設（#3038, 挙動変更ゼロ）**: 駆動ループ群（`run_react_event_loop` / `_drain` /
  `run_react_consumer` / `drive_react_subscriptions` / `replay_static_supply` + `extract_*` ヘルパー）を
  `impl Interpreter`（`runtime/subtest.rs`）から `impl VM`（新 `vm/vm_react_loop.rs`）へ移動。ディスパッチは
  `self.interpreter.call_sub_value`（tree-walk）のまま——VM が Interpreter を所有するので有効——とし、高リスクな
  構造変更を意味変更から隔離。`react{}` 経路（`exec_react_scope_op`、既に VM）は VM メソッドを直接呼び、
  `await $supply`/`.Promise` 経路（`supply_promise_on_demand`、Interpreter）は薄い
  `Interpreter::drive_react_subscriptions` ブリッジ（`mem::take`/`VM::new`/`into_interpreter`——native map
  ループと同じ確立パターン）で到達。supplier レジストリアクセサ（`supplier_snapshot` 等）を `pub(crate)` へ昇格。
- **PR2 = コンパイル済みディスパッチ（#3039, 本命）**: whenever body / LAST / CLOSE callback の19箇所の
  `call_sub_value` を新ヘルパー `VM::call_react_callback` 経由の **コンパイル済みバイトコード**実行へ切替。
  **罠**: whenever body は on-the-fly で `compile_routine_closure_body` によりルーチンとしてコンパイルされ、
  ルーチンは入口で `$_` を `Any` にリセットするため、素朴な `vm_call_on_value` 差し替えだと `react whenever
  Supply { ... $_ ... }`（暗黙トピック）でトピックが失われる（`-> $v` 明示形は無傷）。`vm_call_map_block` の
  explicit-topic 機構（native `.map` の前例）でリセット後に `$_` を強制束縛して解決。制御シグナル
  （`done`/`next`/`last`）は従来通り `Err` として伝播するためループ側のシグナルマッピングは不変、外側 lexical
  の捕捉・書き戻し（`@x.push($_)` 等）も `.map` ブロック同様にコンパイル経路で機能する。
- **PR3 = 後始末**: `vm_register_ops.rs` の `// TODO: compile to bytecode` を除去、台帳の react-loop 行と
  PLAN.md Track C を完了へ更新。

`supply_emit_buffer` は Interpreter field として残置（`self.interpreter.` 経由で VM から到達でき、当初計画の
global 化は不要だった）。検証: 各 PR で build+fmt+clippy クリーン、制御シグナル/書き戻し/ユーザー関数呼び出しが
rakudo 一致、t/ react/supply 40ファイル、whitelist S17-supply/promise 64ファイル(872)、make test(7129)。

#### Stage 3 follow-up: supply `QUIT` handler も VM ネイティブ化

Stage 2 で「残（minor）」としていた最後のドライブループ tree-walk を消した。VM 駆動ループ
（`vm/vm_react_loop.rs`）の 2 箇所（React 経路と Channel quit 経路）が `self.interpreter.call_supply_quit_handler`
（tree-walk `call_sub_value`）へ戻っていたのを、新メソッド `VM::call_supply_quit_handler` へ差し替え。これは
QUIT phaser body を `call_react_callback`（`vm_call_map_block` で `reason` を `$_` に束縛しつつ
**コンパイル済みバイトコード**実行）で走らせ、`when`/`default`/`succeed` を handled、それ以外の例外は伝播、と
Interpreter 版と同一セマンティクスを保つ。**これで VM react ループの whenever / LAST / QUIT / CLOSE
コールバックがすべて native dispatch**になり、駆動ループから tree-walk interpreter へ戻る経路はゼロ。
`Interpreter::call_supply_quit_handler` は Interpreter 単独の on-demand/tap supply 経路からのみ使われる（別系の
tree-walk runtime で対象外）。検証: `t/supply-quit-handler.t`(7 — when 一致 done・default・不一致で quit
継続・emit+done)、S17-supply/syntax.t は変更前後とも 87/90（失敗 57/75/90 は既存で QUIT 無関係）、
whitelist S17-supply 代表 6 ファイル(119) PASS、make test。react 形式 `react { whenever {...QUIT...} }` の
チャネル quit ハングは**変更前から存在する別バグ**でスコープ外。

### トラック C 共有セル スライス 3: スカラ複合代入（`$x OP= rhs`）のアトミック化（#3059, 2026-06-14）

スライス 1（`start` 捕捉スカラ → 共有 `ContainerRef` セル、`++`/`--` のアトミック RMW）、
スライス 2（`state $n` スレッド共有）に続き、スカラ**複合代入**（`+= -= *= /= ~= min= +&= …`）を
共有セル上のアトミック read-modify-write に一般化。`start` 捕捉 lexical がスレッド間で決定的に積算される:

```raku
my $c = 0;
await (^100).map: { start { for ^100 { $c += 1 } } };
say $c;   # 10000（従来は ~4000、lost-update でレース）
```

raku は `+=` のアトミック性を保証しないが、mutsu はスライス 1 の `++` と同様に意図的に決定的。

**設計**:
- 新オペコード `OpCode::AtomicCompoundVar { name_idx, op: CompoundBaseOp }` を
  コンパイラのチョークポイント（`compile_expr_assign`／`Stmt::Assign` 末尾）で
  **平の env 名前付きスカラのみ**に発行。local スロット・twigil/特殊ターゲットは除外するので、
  ホットなリテラル `$x = $x + y` own-local ループは融合しない（融合すると 3M ループが 31.6s→41s+ に退行した）。
- mutation/escape 解析（`op_name_const_idx`／`op_name_write_const_idx`／`has_env_writes`）に登録し、
  `$c` が共有 `ContainerRef` セルに box され続けるようにする。
- 非セル書き込みは `exec_post_increment_op_inner` から抽出した `store_named_scalar_rmw_result`（`++` の
  env 書き戻し末尾）を `++`/`--` と共用 → METHOD captured-outer スカラ伝播が `++` と構造的に同一。
- ContainerRef セル分岐はロックを RMW 全体で保持（アトミック）。Proxy 分岐は fetch/op/store。
- `apply_compound_base_op` は平の `Binary` 経路と同じ `exec_*_op` を流用（演算子セマンティクス・強制・
  ユーザ `infix` オーバーロードが一致）。

**最初の（リバートされた）試行が突き当たった壁の正体**は、事後分析が予測した「SetGlobal ストア末尾の再利用
リファクタ」ではなく、コンパイラの 1 行だった: 融合 rhs を `compile_expr` でコンパイルする（`compile_call_arg`
ではない）。`compile_call_arg` はスカラ被演算子を itemize/escape-box してしまい、rhs が変数参照
（`$acc += $!step` をメソッド内で）のとき captured-outer の env 伝播を壊していた。

検証: `t/concurrent-compound-assign.t`（新規, 8 tests）、回帰アンカー `t/method-env-dirty.t`（24/24）・

#### クロージャ/正規表現を取るメソッドの第2弾（ユーザー依頼「subst/sort/map remainder/Test を一個ずつ全部」）

これらの「フォールバック」はブロック本体ではなく orchestration の入口のみ（本体は既に VM コンパイル実行）。
スタック PR（subst ← sort ← map ← test、各マージ後 `git rebase --onto origin/main` で前 commit を落とす運用）で実装:

- **`.subst`** を native 化 (#2621)。Str invocant・plain `Value::Regex`/`Value::Str` パターン・Str(or無し)
  置換・`:g` のみ。演算子形 `s///` の VM 部品（`regex_find_first_from_with_captures`・`expand_capture_refs`・
  Match 構築）を再利用し、`$/` ポリシーまで `dispatch_subst` と byte 一致。closure 置換/RegexWithAdverbs/
  Perl5/その他 adverb は fallback。最頻の subst method fallback を解消。
- **`.sort`** を native 化 (#2622)。plain `Array`/`List`・比較子なし(`compare_values`)or `detect_simple_cmp_block`
  該当(`{$^a<=>$^b}`/reverse/`cmp`)を inline 比較。interpreter の fast-path 検出を `pub(crate)` 共有し
  **perf 中立**（interpreter も既に block 呼びを回避済み）。method-keyed/mapper/Routine/`:k`/Shaped 等は fallback。
- **`.map` 多 arity + pointy ブロック** (#2623)。`-> $a,$b` / `{$^a+$^b}` を arity チャンクで反復。
  ついでに **pointy `Lambda` body の先頭 `SetLine` を scanner が拒否していた既存制約も修正**し、単一 arity
  pointy も native 化。短い末尾 chunk（`len % arity != 0`）は interpreter の error 経路へ fallback。
- **Test 関数のディスパッチ層** を VM 化 (#2625)。`is`/`ok`/`plan`/`is-deeply`/… を `call_function` の
  giant match を経由せず `call_test_function` へ直接ルート。**スコープは正直に「ディスパッチ層のみ」**:
  本体は TAP 状態・I/O・method dispatch・eval に本質的に結合（TAP 状態所有の移管はレバー B の課題）。
  正しさ3点 = (1) ゲートは `test_module_loaded()`（最初の `plan` 前でも真）、(2) builtin と衝突する名
  （特に `run` = Proc spawner）は `is_builtin_function` で除外、(3) user-func 解決後の位置なので `sub ok`
  シャドーイング尊重。whitelist の `line-numbers.t`（test 内で builtin `run` を多用）が `run` 誤ハイジャック
  回帰を検出→修正。
- **`.grep` は native 化せず**: 結果が元配列への rw ビュー（`@a.grep(...)>>++` が元更新）で clone 不可。

#### レバー A 終着: `EVAL` / pseudo-package を carrier として分類

レバー A 最後の未完了項目（`EVAL` / symbolic deref / `CALLER::`）を解決し、レバー A を**終着**。
これらは **tree-walk フォールバックではない**: `EVAL`/`EVALFILE` は既に `eval_block_value` で
compile→バイトコード→サブ VM 実行（`run_compiled_block`/`VM::new`）し eval 対象は tree-walk されない。
pseudo-package 読み（`CALLER::`/`OUTER::`/`SETTING::`/`DYNAMIC::`）も reflective な env lookup。両者は
interpreter を**キャリア**として使うだけ（compile/lookup に必要な env/classes/roles レジストリ所有のため）。
これらをフォールバック計上していたのは計測ノイズだった。

- `VM::is_interpreter_carrier_function` を新設し、2つの interpreter 入口（`call_function_compiled_first` と
  `dispatch_func_call_inner` 最終 else）で `vm_stats::record_function_carrier` に振り分け。
- `MUTSU_VM_STATS=1` が carrier を別バケット報告（`interpreter_carrier=N` + `function-carrier by name`）。
- t/ 集計で関数 tree-walk-fallback **18.8% → 18.6%**（EVAL 54 件が carrier へ移動）。残るフォールバックの
  支配項は `__mutsu_atomic_post_inc_var`（atomic `$x⚛++`）= concurrency/共有状態 = **レバー B 領域**。
- pin: `cargo test carrier_functions_are_not_tree_walk_fallbacks`。`EVAL` の残結合（共有 env/レジストリの
  Interpreter 所有）は明示的にレバー B の課題。

### レバー B: locals↔env 二重ストアの解消

- 序盤スライス（base-tier 化・closure capture を free-var に限定・format!/Symbol churn 除去）に続き、

  Interpreter env にミラーしないよう `ensure_env_synced` を `needs_env_sync` でゲート。
  「locals が権威、env は名前経由リーダーが必要とするものだけ」への前進。fib で per-call の
  flushed-slot が 635593→0（param が interpreter env を越えなくなった）。ループ/ブロックは制御
  テンポラリの env 往復のため保守的に全 sync（needs_env_sync の不完全性への対処）。

- **Slice 6（scoped overlay）→ 6.1/6.2（dirty-flag 機構撤廃）→ multi-tier overlay #2684**:
  per-call deep copy・per-nested-call O(env) flatten を撲滅（`mb_nested` env_deep_copies 10001→1）。
- **Slice 6.3 stage 1/1b（#2689/#2692）— メソッド dispatch 毎回の env→locals pull を撲滅**:
  opcode の無条件 `env_dirty=true` を `method_dispatch_pure` フラグ（default false=保守的、純粋証明時のみ
  mark 省略＝取りこぼしは余分な pull 止まりで stale read なし）に置換。compiled/native/merge 各パスが精密に
  純粋性を報告し、`merge_method_env` は `cheaply_unchanged`(Arc-ptr/scalar 等価)＋`could_name_caller_local`
  (`?LINE`/`?FILE`/`*`動的/`__mutsu_` を除外)で精密 changed-flag を返す（ネスト呼びの overlay flatten と
  `?LINE` 行差による偽陽性を解消）。fast accessor read（`$obj.x`）と native `.new` も env-pure 化。
  **結果: benchmarks/method-call 20000→1, bench-class 14001→2, mb_method/mb_nested 10001→1。**
  make test PASS、pin `t/method-env-dirty.t`(24)。env_dirty フラグ自体の削除は残り（regex `~~` の by-name
  書き込み step2 / I/O+carrier step3、後者は interpreter bridge 撤去待ち）— 詳細は docs/vm-dual-store.md の

  誘発して失敗→「dispatch 内から純粋性を報告」が正解。計測は必ず benchmarks/*.raku の実ベンチで。

- **Slice 6.3 step2 — dispatch fast path の per-call pull 撲滅（3経路）**: stage 1/1b で残っていた dispatch
  経路の毎回 pull を精密化（いずれも `for ^5000 {...}` で `locals_pulls` 5001→1〜2）。
  ①**0-arg fast path**: `call_compiled_function_fast`（vm_call_dispatch.rs）が 0-local 関数で merge 経路を持たず
  呼び出し元の blanket `env_dirty=true` に依存していた。コンパイル時 `cf.code.has_env_writes` ゲートで精密化
  （env 書き込みの無い本体は caller slot を汚せない＝dispatch の routine `_` も復元されるので pull 不要）。
  ※0-local に scoped overlay を被せる案は routine `_` 挿入が make_mut を誘発し env_deep_copies 1→5000 で逆効果（不採用）。

  ②**heavy named path**: 複雑シグネチャ（default param / return type / where / typed / multi）が通る
  `call_compiled_function_named` を `call_compiled_function_fast` と同型に精密化（env_dirty を保存→body 前リセット→
  return merge で `env_dirty = saved || changed`。changed＝plain-lexical/captured-outer writeback の実変化、非空の
  `is rw` writeback、`is raw` return のいずれか。動的変数 writeback は slot を持たないので除外）。hot call 元
  （heavy/OTF/cached named、bareword-term named）の blanket マークを撤去。EVAL/carrier 元は step3 なので保守的に維持。

  （`S02-magicals/sub.t` の `regex foo { a { $var = &?ROUTINE } }` 埋め込みブロックが caller local 書き込み、
  `S05-modifier/pos.t` の `my $/`+`:pos` 継続マッチで `$/` がローカル）を出した。原因は **regex RHS が埋め込み `{}`

  既に `Interpreter::pending_local_updates` に書き込み変数を記録している（が誰も消費していない）ことを発見。
  env_dirty を skip するのは全条件成立時のみ — ①compile-time `rhs_pure_regex`（plain `Value::Regex` リテラル＝

  ついでに従来 unbounded だった leak も解消）、③subst/translit でない、④`$/` がローカルスロットでない。lhs alias は
  slot へ reverse-write-through。pin `t/smartmatch-env-dirty.t`（18: pure match / 埋め込みブロック書き込み / named
  regex / `my $/` 継続 / 置換 / given-when / RHS-sub 副作用）。これで **Slice 6.3 step2 の hot な per-call/per-match
  pull は dispatch 4経路 + smartmatch すべて撲滅**。残りは真に cold な setter（register trait-handler / `:=` 要素
  SlotRef / loop-control 往復）と step3 carrier のみ。

### VM↔Interpreter 重複実装の解消（dedup）— 方針の明確化 + 第1バッチ

**ユーザー方針の明確化（2026-06-07）**: VM decoupling の真の目的は perf でも env_dirty でもなく、**VM(`vm/`+native
`builtins/`) と tree-walking Interpreter(`runtime/`) に同じ処理が二重実装されている保守コストの解消**。実害は
「AI/人がコードを調べる時に惑わされる・間違った方を直す・片方だけ更新する」。fallback が実際に発火するか（計測上
ほぼ 0%）は本質ではなく、**重複の存在そのもの**が問題。→ 手順・優先マップを `docs/vm-interpreter-dedup.md` に明文化、
方針を PLAN.md「最終ゴール」とメモリ（feedback_dedup_over_perf）に記録。

**第1バッチ**: 純粋値 builtin 9個の Interpreter コピー（`builtin_abs`/`lc`/`uc`/`tc`/`trim`/`flip`/`chr`/`ord`/`chars`）を
削除。Interpreter の `call_function` catch-all → `call_function_fallback` → `crate::builtins::native_function` で native
にフォールスルーするので、arm + `builtin_*` 本体を消しても EVAL/正規表現/carrier 経路は native を使う（EVAL 経路で

### レバー C: クロージャ upvalue 化（per-iteration / コンテナ捕捉）

Raku のクロージャは自由変数の**バインディング（コンテナ）**を捕捉し、ループ本体の `my` は
反復ごとに fresh binding になる。mutsu は自由変数を**名前で遅延読み**し、body-local `my` が
VM slot を占有 + `sync_env_from_locals` が毎呼び出し前に全 slot を env へ再注入するため、
ループ最終値が各クロージャの捕捉値を上書きしていた。

- **Slice 0（#2663）**: `for 1..N -> $i` の**ループ変数** per-iteration 捕捉。int-range fast path が
  named param を save/restore しておらず `$i` がループ後に漏れていた。
- **Slice 1（#2668, owned_captures）**: ループ**本体ローカル** `my` の per-iteration 捕捉（全ループ種）。
  `VM::loop_local_vars` に本体宣言を登録、クロージャ生成時に loop-local な free var を
  `SubData::owned_captures` として記録し、呼び出し時に**自分の凍結 env から上書き読み**して
  slot 再注入に免疫化。`for 1..3 { my $x=$_; @c.push({$x}) }` → `123`。
- **Slice 2（#2669, box-on-capture）**: ループ本体ローカルについて、クロージャは**値でなくコンテナ**を捕捉。
  ループ本体内で生成されるクロージャの被捕捉 loop-body-local スカラを slot+env 両方で共有 `ContainerRef` に
  置換してから env スナップショット。これで捕捉**後**の同一反復内変異が見え
  （`for 1 { my $x=1; my $c={$x}; $x=2; $c() }` → `2`）、同一反復内の兄弟クロージャ共有も対応。
  per-iteration freshness は vardecl 時に slot の stale ContainerRef もクリアして維持。付随で
  Pre{Inc,Dec} の ContainerRef deref 欠落（`--$b` の文字列デクリメント）と for-loop rw writeback の
  `$pair.value` clobber も修正。**スコープは loop-body-local に限定**した — 全被捕捉スカラを box すると
  非ループ捕捉（Test の `lives-ok {}` が周囲の `$obj`/型オブジェクトを捕捉）が ContainerRef 未対応の多数経路
  （不変性・型 dispatch・`.kv` rw writeback 等）を踏み広範囲に壊れたため。
- **Slice 2b（精密化）**: コンパイラ解析 `CompiledCode.captured_mutated_locals`（nested closure に捕捉され、
  かつ宣言以外で変異される own local。宣言/再代入は両方 `SetLocal` だが前置 `MarkVarDeclContext` で区別）を
  導入し、boxing を `loop_local_vars` ∩ `captured_mutated_locals` に限定（read-only ループ local は
  owned_captures に委ね無駄な box を回避）。closure dispatch で `data.env` の ContainerRef を merge 時に上書き
  ＋ cap_overrides スキップ → ループ local の**繰り返し変異**が正しく見える（… `$x=2; $c(); $x=5; $c()` → `2,5`）。
- **残（延期・大）**: 非ループの一般コンテナ捕捉（`my ($g,$s)=mk(); $s(42); $g()` → `42`、top-level
  mutation-after-capture）。captured_mutated_locals は非ループも検出可能だが、非ループ被捕捉スカラを box すると
  ContainerRef 未対応経路を踏む（型付き `.=`、isa-ok、subset 型代入、submethod state、Mix 不変性）。
  全 lvalue/型/dispatch 経路の deref 監査（コンテナ・アーキの本丸）が必要で、専用 PR 群でやる。
- **Slice 3（ループ本体ローカル `my` の clobber 根治）**: `my $x=99; for/while/loop/repeat { my $x=... }; say $x` が
  外側 `$x` をクロバーして最終値（raku は `99`）になる**クロージャ非依存の正しさバグ**を修正。dual-store の env が
  name-keyed で、ループ本体が全 local を `needs_env_sync` 化するため内側 `my $x` の write-through が env["x"] と
  共有スロットの両方で外側を上書きしていた。**fresh-slot（コンパイラ）案は却下** — 同名複数スロットが
  name-based の box-on-capture（lever C）解析をブロック跨ぎで誤適用させ closure 捕捉を壊す。**ランタイム復元**を
  採用: ループ本体で既存外側束縛を shadow した `my` 名のみ宣言時に外側値を `loop_local_saved_env` へ記録し、
  ループ脱出で env＋共有スロットに復元。ループ**条件**の `my`（`... until COND` 文末修飾含む）は enclosing-scoped
  なので `loop_cond_active` で除外（`iterator-regressions.t` 回帰回避）。pin `t/block-local-my-scope.t`。

  エラー）は mutsu では undefined 値（strict-vars 未実装で全体的に未強制、実害なし）。

- **Slice 3b（`if`/`unless`/`else` ボディローカル `my` の clobber 根治）**: `my $x=99; if c { my $x=5 }; say $x` が

## 正規表現 validator/matcher 二重実装の統合（[ANALYSIS.md](../ANALYSIS.md) §3.1）

mutsu には Raku 正規表現の文法知識が**2 箇所に独立してエンコード**されていた:
`src/regex_validate.rs`（1108 行、パース時の構文検証専用・AST を作らず `X::Syntax::Regex::*` を投げるだけ）と、
`src/runtime/regex_parse.rs` の構造パーサ（`parse_regex`／実マッチャ）。両者が量化子・文字クラス・エスケープ・
アンカー規則を別々に実装し "keep in sync" コメントは 0 件 — 片方を直してもう片方を放置する drift がバグの
温床だった（PLAN.md「🔴最優先: 1操作=1実装」）。

**構造パーサを唯一の文法権威に統合し、`regex_validate.rs` を削除した。** 手法:

- **parse モード導入**: `RegexParseMode { Match, Validate }` を `parse_regex_uncached` に通した。`Match` は
  従来の実行時パス（補間・grammar token 解決）。`Validate` はパース時のドライランで、補間・LTM 展開・
  grammar token 解決をスキップし、変数参照を不透明アトムとして受理する（パース時には変数値が無いため）。
- **パース時検証の自由関数化**: パーサ（`parser/primary/regex.rs`）は Interpreter を持たないので、
  `validate_regex_structurally(pattern) -> Result<(),RuntimeError>` を新設。thread-local の検証用
  Interpreter を `&self` レシーバとして使い回し（毎リテラルごとの重い `Interpreter::new()` を回避）、
  `Validate` モードで構造パースして `PENDING_REGEX_ERROR` を拾う。「`None` だが pending 無し」＝
  「構文エラーではない（旧 validator も受理）」→ `Ok(())` とするのが最重要の正しさルール。
- **validator 限定チェックを構造パーサへ移植**（すべて `Validate` ゲートで `Match` の挙動はバイト等価）:
  孤立量化子 / 量化不可（アンカー・コードブロック）/ 未知メタ文字 `-`!`/ 未知修飾子 `:11` /
  孤立バックトラック制御 `:` / 旧 P5 エスケープ `\A`\Z`\z` / 未知バックスラッシュ列 / P5 文字範囲 `a-z` /
  逆順範囲・NFG 範囲端点 / 複合クラスの演算子欠落 `<[a][b]>` / proto 外の `<sym>` / 属性補間 `$!attr` /
  hash 変数 `%var` 予約 / 長名エイリアス `<::IO::File=bar>`。
- **ドリフトで見つけた潜在バグも修正**: 「No such method」など実行時解決エラーは `Match` 限定にゲート
  （`dies-ok` が実行時死を期待するため compile error にしない）、LTM 展開は `Match` 専用（展開産物が
  パース時チェックを誤検知するのを回避）。

`make test` PASS、ユニットテスト 2 件追加（`validate_regex_structurally` の受理/拒否＋例外型）。
roast の正規表現系（S05-metasyntax / S05-metachars / S05-mass / S05-interpolation）はベースライン一致。

### Category B ＝ 完了（genuine fork: min/max/minmax/sort/join/first/flat/elems/reverse）
native/pure 層が comparator ブロック・lazy で bail し Interpreter に落ちていた重複を全解消。**2 種の fork を別手法で**:

- **ブロック系 fork（comparator/mapper/matcher）**: オーケストレーション（fold/sort/scan 本体）を engine 非依存の
  単一実装に抽出し、**ブロック呼び出しだけ**を VM(`vm_call_on_value`) / interpreter(`call_sub_value`) のクロージャ
  （trait）で差し替え。`interpreter_fallbacks=0` 実測。
  - **sort (#2727)**: `sort_items_generic`/`sort_indices_generic` ＋ `SortCaller`。comparator/mapper/`:k`/Hash sort を native 化。
  - **min/max (#2728)**: `extrema_from_values_generic`（`:by`・`:k`/`:v`/`:kv`/`:p` 込み）。
  - **minmax (#2730)**: `minmax_from_values_generic`（`:by` 込み）。
  - **first (#2731)**: `find_first_match_generic` ＋ `FirstMatcher` trait（block/regex/smartmatch、Hash 要素は `pair_as_positional`）。
- **lazy-fork（elems/flat/join/reverse）**: 原因は lazy 強制／多態が native↔interpreter で別実装だったこと。単一実装へ
  委譲し、過程で **drift 由来の潜在バグを多数発見・修正**:
  - **elems (#2733)**: `.elems` へ委譲。`elems("hello")` 5→1、`elems(Seq)` 1→3、`elems(lazy)`→`X::Cannot::Lazy`、`end("hello")` 4→0。
  - **flat (#2734)**: 単一 `flat_val`。nested array の over-flatten 修正（List vs Array の段数）。
  - **join (#2735)**: 単一 `join_flat`。`join("-",1..4)` のセパレータ無視バグ・無引数 panic 修正。
  - **reverse (#2739)**: 単一 native へ委譲。`reverse()`=`Nil`→`()`、Range/Slip/shaped 修正。

**dedup の教訓（実証）**: 重複は drift してバグの温床になる。`[%] 2**70` 精度・`[~]` NFC・`[minmax]` ネスト・
`%h.first` の named 束縛・`elems("hello")`・`flat` over-flatten・`join(sep,Range)`・`reverse()`=Nil・
`.first(-> $p)` の pointy 未束縛は全て dedup 作業中に発見・修正された。配置監査（comb/substr/split）で
「単一実装が**正しいレイヤ**に在るか」も dedup の問いに加えた。

## 遅延リストの pull/Iterator モデル統一（PLAN.md §8.1 / ANALYSIS §8.1・§8.2）

無限 Range（`1..Inf`）/ 無限コルーチン gather が無ガードの eager 展開に届くと `capacity overflow` で
**プロセスごと panic** したり HANG する欠陥クラスを撲滅。単一ガード展開点 `ValueIterator` +
`materialize_capped`（`src/runtime/value_iterator.rs`, cap=1M）を新設し、各 eager arm を委譲。

- **PR-1 (#2791)**: `grep` の method 形 + sub 形の Range arm を `materialize_capped` 経由に。pin=`t/grep-lazy-range.t`
- **PR-2 (#2793)**: builtin `map`/`first` の sub 形 Range arm を統一。pin=`t/map-first-lazy-range.t`
- **PR-4 (#2796)**: eager 操作（sort/combinations/permutations/classify/categorize）が無限/遅延ソースで
  **X::Cannot::Lazy を throw**（従来 HANG/誤結果）。`Interpreter::lazy_guard_error` を全 dispatch
  チョークポイントに配線。pin=`t/eager-ops-cannot-lazy.t`
- **PR-3 は計測の上 見送り**: for-loop 全面 pull 化は crash が PR-1/2 + `value_to_list` cap で既に解消済み・
  ホット経路は既に真に遅延（`for 1..Inf { last }`=0.01s）のため、最ホット 200 行の書き換えは低 ROI と判断。
- **無限コルーチン gather の `.head`/`.first`/index 遅延化 (#2799)**: `gather { loop { take … } }` への
  `.head(n)`/`.head`/`.first(pred)`/インデックスが（無限の）`LazyList` を全 force して HANG していた問題を、
  `force_lazy_list_vm_n` による bounded/incremental pull に置換。本丸は **bare `loop`・`while`/`until`・
  C-style `loop` のコルーチン suspend/resume 対応**（従来 `for` ループだけが resume 状態を保存しており、
  2 要素目より先のアクセスが暗黙に `Nil` を返していた）。新マーカー `ForLoopResumeState::CStyleLoop` で
  take-limit suspend 時に VM の `ip` をループ opcode に留め、locals/env 状態から条件チェックに再入。
  `.first` は 1 要素ずつ pull して副作用回数を Rakudo と一致させた。pin=`t/gather-infinite-coroutine.t`（21 cases）。
- **本丸: method 形 `.map`/`.grep` の eager materialize 撲滅 (#2804)**: 従来 `(1..Inf).map(* * 2)` は
  無限ソースを 1M cap で eager 展開していた（`.head(3)`=7s、`.elems`=1000001、`.is-lazy`=False、連鎖
  `.map(...).grep(...).head(3)`=27s）。これを **真に遅延な `LazyList`（新 `MapGrepSpec` を `lazy_pipe` に
  持つ pipeline stage）** を返し、ソースを 1 要素ずつ pull する方式に変更。
  - ゲートは **無限整数 Range / 既存 lazy_pipe への連鎖のみ** に限定し回帰最小化（gather/scan/sequence/
    有限ソースの map/grep は不変）。multi-arity/slurpy コールバック・grep adverb は eager パス維持。
  - stage 適用は **VM ネイティブ `vm_call_on_value` / `vm_grep_item_matches`**（interpreter の
    `eval_grep_over_items` は `mem::take` でネスト VM を作り外側フレームを破棄するため不可）。
    これにより map コールバックの副作用が外側スコープへ正しく伝播し、locals/env も整合。
  - strict 全要素 op（`.elems`/`.sort`/…）on 無限 pipe = `X::Cannot::Lazy` throw（raku 一致、従来 HANG）。
    遅延保持 coercion（`.Seq`/`.List`/`.list`/`.cache`/`.values`/`.lazy`）は pipe をそのまま返す。
  - `force_lazy_pipe`（`vm_helpers.rs`）を `force_lazy_list_vm_n` / for-loop / index dispatch に配線。
  - pin=`t/lazy-method-map-grep.t`（22 cases: head/first/index/連鎖/for-loop/is-lazy/X::Cannot::Lazy/有限不変）。

## ③ 最終ゴール: §1/§2 catch-all の native 化（フォールバック台帳の消化）

②（レジストリ VM 所有化）完了後、主作業を ③（VM が借りる Interpreter 状態の所有移管＝フォールバック撲滅駆動）へ。
一次台帳は [docs/vm-interpreter-fallback-ledger.md](../docs/vm-interpreter-fallback-ledger.md)。**「生ディスパッチを統一エントリへ
降ろすだけ」の安いサイトは枯渇**し、以後は「pure-data で native 化できるカテゴリ」を 1 つずつ降ろした。

### ディスパッチ統一（PR-1〜4）
- **PR-1 Routine dispatch (#2755 ほか)**: Routine 値解決を統一 compiled-first へ（builtin 名 Routine は builtin 優先維持）。
- **PR-2 builtin-shadow fork**: ユーザ sub が同名 builtin を shadow する fork の compilable 単一候補を OTF compile。
- **PR-3 §1 catch-all = ユーザーメソッド compile-on-demand**: `.^add_multi_method` 等の compile-gap を埋める。
  **実態訂正**: 通常宣言/multi/submethod/role 合成/継承/`.^add_method` は登録時に既に bytecode 化済で、catch-all に
  来る Instance は native coercion / MOP / role-qualified のみと実測。
- **PR-4 §2 非proto multi fork**: winner を VM 側 `resolve_function_with_types` で解決し OTF compile。
  callsame/nextsame の決定的 winner 化で **既存のハッシュシード依存 flaky（callsame.t 間欠 fail）も修正**。

### pure-data カテゴリの native 化（PR-5〜10）
- **PR-5 plain `@`-array mutators**: `append`/`prepend`/`unshift`/`pop`/`shift` を `try_native_array_mut`。Arc-ptr keying
  aliasing 対処（変異後 env 配列の stale メタ除去）。
- **PR-6 §2 末端実測 + junction ctor 降ろし**: any/all/one/none を `builtins/functions.rs::build_junction` へ一本化。
  末端到達は枯渇（残＝lexical `&`-var 名呼び / `__mutsu_*` / no-match エラー生成）と実測。
- **PR-7 mutable Buf write methods**: `write-bits`/`write-int*`/… を `try_native_buf_mut`。純粋バイト変換を
  `builtins/{buf_bits,buf_write_num,buf_write_int}` へ降ろし dedup。
- **PR-8 QuantHash coercion**: `.Set`/`.Bag`/`.Mix`/… を `try_native_quanthash_coerce` ＋ `builtins/quanthash_coerce` へ降ろし。
- **PR-9 array-backed Iterator protocol**: `pull-one`/`skip-one`/… を `try_native_iterator`（`$it.pull-one` は CallMethodMut）。
- **PR-10 plain `@`-array splice**: 単純形 splice を `try_native_array_splice`（範囲外/lazy/typed は fallthrough）。

### Package.new コンストラクタの native 化（属性 shape 別、#2826〜2844）
§1 catch-all 最大カテゴリ `Package.new` を、**pure-data で構築できる属性 shape** から native 化。共有ゲート
`is_native_default_constructible` ＋ 構築 `build_native_default_instance`（`methods_object.rs`、VM の
`try_compiled_method_or_interpret` と interpreter の `dispatch_new` が共有）。**保守的フォールスルー**で不確実は全て
interpreter に委ね behavior-invariant（対象は raku ではなく現 interpreter）。
- **splice native (#2826)**: ctor とは別の `try_native_array_splice`。
- **typed `$` (#2833)**: `has Int $.x`。simple class 制約。
- **untyped `@`/`%` (#2834)**: `has @.items`/`has %.map`。`coerce_attr_value_by_sigil` 再利用。
- **required (#2835)**: 提供→build / 未提供→fallthrough。
- **native 型 (#2837)**: `has int $.x`。未提供→native zero、提供→そのまま格納。literal 高速パスで 20k-loop timeout 回避。
  CUnion クラスはゲート除外（byte-overlay 構築）。
- **coercion 型 (#2838)**: `has Int() $.x`。パーサ auto-default を target type object に修正＋共有
  `coerce_value_for_constraint` 経由 coercion（属性だけ未対応だったバグ修正込み）。
- **typed `@`/`%` (#2841)**: `has Int @.nums`。**provide が element 型を喪失していたバグ修正込み**（`.of`=Mu→Int）。
  共有ヘルパ `finalize_typed_container_attr`（element 型チェック＋metadata 登録、Arc 共有で最終 attrs に効く）を
  dispatch_new / native build で共有。
- **is-Type / shaped correctness fix (#2844)**: **native 化はしない**（is-Type/shaped 構築は型/Array の `.new`
  ディスパッチに依存し pure-data でない → phase ④ 待ち）。代わりに 2 バグ修正: ①#2841 由来の shaped typed 配列
  クラッシュ regression（finalize pass を leaf まで再帰）②is-Type provided coercion（`has @.a is Buf` 提供値を
  `coerce_value_to_is_type` で declared 型へ）。

**§1 ctor の結論（#2826〜2844 時点）**: pure-data で構築できる属性 shape（untyped/typed `$`・`@`/`%`・native 型・
coercion 型・required）はカバー完了。残る shape（is-Type/shaped/where/native-element-container）は当時「interpreter
method dispatch / user code 実行に依存し pure-data 原則では native 化できない」と判断 → phase ④ 待ちとした。

### Package.new コンストラクタの native 化 第2波: post-assembly フェーズ方式（#3028〜3036, 2026-06-14）
上記の「pure-data only 原則」を破り、**インスタンスを pure-data 組み立てした後に submethod 実行・制約検証・role mixin を
post-assembly フェーズとして走らせる**手法を確立。これで user code を走らせる shape も native 化できた。`.new` 本経路の
各フェーズを共有ヘルパに抽出（dedup）し interpreter/native 両方から呼んで byte-identical を保証。
- **TWEAK (#3028)**: `submethod TWEAK` を `run_tweak_phase` で post-assembly 実行。`TWEAK(:$y)` への引数渡しが
  必要（コンストラクタ経路は2系統＝`self.bless`/`.new` 本体で引数の扱いが違う）。
- **where (#3030)**: `enforce_attribute_where_constraints` を post-check。**TWEAK の前（代入時）と後の両方**でチェック
  （raku/interpreter 一致＝where は assignment-time セマンティクス）。
- **BUILD (#3032)**: `run_build_phase`（`fail`→Failure 返し）。**カスタム BUILD ありで named 引数の属性自動代入は抑制**
  （`P.new(:666x)` で x=42 のまま）＋**positional 引数は `.new` が拒否**（pre-existing バグ `S.new("pos")` 誤受理も修正）。
- **is rw (#3034)**: `ClassAttributeDef` タプルのフィールド取り違え修正（pos3=is_rw を is_required と誤読していた）。
  未提供 is-rw が native 化、付随的に `@`/`%` の `is required` 強制も正しく。
- **does Role (#3036)**: `apply_attribute_does_role_mixins` で role を属性値に mixin。**mutsu の does-Role は raku 非準拠の
  文書化済み近似**（raku は immutable コンテナに mixin）なので検証は native==interpreter で実施。

**第2波の結論**: ユーザークラスの `.new` 主要 shape（BUILD/TWEAK/where/is-rw/does-Role）はほぼ native 化完了。
**再計測（セッション末）**: whitelist サンプルの `new` method fallback はほぼ不変（5230→5225）だが、その大半（4686）は
**`Buf.new`＝組み込み型コンストラクタ**（`read-write-bits.t`）でユーザークラスではない。＝ユーザークラス ctor の native 化は
完遂、これ以上 ctor を削っても `new` fallback 数は動かない。次の実質ターゲットは別カテゴリ（組み込み型 `.new` / MOP `name`
=3257 / Routine 値 lexical &-var `op`=1418 / iterator push-* protocol / coercion）。

**旧 §1 ctor の結論（#2826〜2844 時点）の残 shape**: is-Type/shaped/native-element-container は依然 interpreter
（型/Array の `.new` ディスパッチ依存）。where は #3030 で解決済み。

**重要な現状認識**: §1/§2 の「安い native 化」は枯渇した。残るフォールバックは全て構造的ブロッカー
（③ state 所有 / 第一級コンテナ Phase 2 / lever B 並行）が前提。次の実質進捗はこれら 3 つの構造リファクタであり、
**互いに独立なので並列実行可能**（下記 PLAN.md の再編を参照）。

## 2026-06-14: given/with topic aliasing + read-only topic 意味論 (#3009)

`given @a { .push(4) }` は Raku ではコンテナをエイリアス topicalize し、`$_` 経由のコンテナ変異が

### 2026-06-18: env↔locals coherence Stage 0 — `:=` bound container readonly の論理分離を完了（constant 要素書込 ＋ closure 内 whole-reassign）

`docs/env-locals-coherence.md` Stage 0 の残バグ#1（`my %a := %b; %a = (z=>9)` が readonly 誤発火）は前セッションの
#3255 が `:=` bound container 専用マーカー `__mutsu_bound::%name` を導入して slot 経路の whole-reassign を直したが、

- **対称バグ — `constant %M<a> = 9` が誤って成功**: 要素 path（`vm_var_assign_ops.rs` の `is_bound_hash_var`）が

## 2026-06-21: Template::Mustache デリミタ変更 `{{=<% %>=}}` — reduce-time の `$*` 動的変数アクション (#3390)

**Mustache の最後級ブロッカー「マッチ途中のデリミタ変更」を実装。** `{{=<% %>=}}` はタグの finalizer が
`($*LEFT,$*RIGHT)=@delim` を書き換え、以降のタグを新デリミタでマッチさせる。mutsu はアクションを post-parse 実行
していたため書き込みが live match に効かなかった。正規表現マッチエンジンは `&self` で env を mid-match 変更できないため、
**parse スコープのスレッドローカル overlay**（`REGEX_DYNVAR_OVERLAY`）を導入: `<subrule>*`/`+` の各イテレーション確定後に
そのアクションを scratch interp（actions オブジェクトは**ディープコピー**して self 変更を隔離）で走らせ、変化した `$*` 変数を
overlay へ publish。`interpolate_regex_scalars` は `$*` twigil 解決時に env より先に overlay を参照する。**SEEN ゲート**
（最初の `$*` 補間で立つ）により、動的変数に依存しない通常グラマーはゼロコスト。

3つの落とし穴: ①`token`（ratchet）の `<x>*` は高速経路を通り hook が刺さらない＝`regex` のみ一般経路（mustache TOP/hunk は
`regex`）②候補選択パスが overlay を汚染し本マッチが汚染状態で開始→`dynvar_overlay_reset_scan()` を parse driver の本マッチ
直前に挿入（regex エントリ内に置くとサブルールマッチ毎にリセットして scan 内状態を破壊する）③独立した既存バグ: 補間スカラーは
リテラルマッチすべきだが `escape_regex_scalar_literal` が `%`/`&` を未エスケープ→`%>` デリミタが直前 `\h*` と結合し
`\h* % ...`（セパレータ量化子）と誤解釈。`%`/`&` をエスケープ集合に追加（複数文字 `<% %>` デリミタ単独でもブロッカーだった）。

**結果**: Template::Mustache **01-basic 8/10 → 10/10**（delimiter change + `{{=delim=}}` substitution）, **13-pragmas** 全パス。
回帰なし（全91 whitelisted S05 + S03/S32-str + make test 9763）。pin: `t/grammar-reduce-time-dynvar.t`。
残る Mustache ブロッカー＝06-logging（unit-level `CONTROL{default{.resume}}` のクロスフレーム継続、最深）, grammar マッチ perf。

## 2026-06-21: Template::Mustache 完動 — クロスフレーム再開可能 warn（06-logging, #3395）

**Mustache の最後のブロッカー 06-logging を解決し、Template::Mustache が完全動作した。** 外部 `JSON::Fast` に依存する

**問題**: 深いフレーム（render→…→`&warn.()`）で投げた `warn` を unit レベルの `CONTROL { default { …; .resume } }` で
受けると、mutsu は warn シグナルを Rust の `Err` として CONTROL フレームまで巻き戻していた。途中の深いフレームが破棄されるため、
単一の `resume_ip`（フレーム相対 IP 1つ）では巻き戻された深部に再入できず、warn 以降の計算が静かに失われていた（render が中断）。

**解決＝resume_safe な CONTROL を raise 地点でインライン実行**:
1. **コンパイル時**: CONTROL ブロックが「無条件 `.resume`（`when`/`succeed` 脱出なし）」かを AST から判定し
   `OpCode::TryCatch{resume_safe}` に焼き込む（ランタイムは AST を見られない。`control_block_is_resume_safe`）。
2. **ランタイム**: `control_handlers: Vec<ControlHandlerEntry>` を Interpreter に追加し `control_handler_depth` と完全同期
   （`exec_try_catch_op_inner` の両 push/pop 地点）。resume_safe なエントリは自前の `Arc<CompiledCode>` + 範囲 + 関数表を保持
   （clone コストは稀な CONTROL 進入時のみ）。
3. **`builtin_warn`（depth>0）**: 最内ハンドラが resume_safe なら raise 地点でインライン実行（`try_resume_safe_control_inline`）し
   `Ok(Nil)` を返す → 深部がそのまま再開。**鍵**: ハンドラのバイトコードは設置フレームのローカルスロットを参照するが、
   現在の `self.locals` は深部フレームのもの。そこで env（クロスフレーム可視ストア、env キー＝`code.locals` 名）から
   ハンドラのローカルを再構成して実行し、変化したスロットだけ env へ書き戻して `env_dirty` を立てる
   → 設置フレームが呼び出し復帰時に `reconcile_locals_from_env_at_site` で取り込む（`$out ~=` が render 後まで残る）。
   resume_safe でない `when CX::Warn { }` は従来の巻き戻し経路を維持（暗黙の `succeed` がブロック境界へ脱出する＝#3372 のキラー）。

**結果**: `t/warn-cross-frame-control-resume.t`（7 ケース・rakudo 検証）。#3372 回帰ガード全緑（control/warn/catch/top-level/
indent/quietly/gather）。make test 9819 全パス。これで PLAN.md §H の Template::Mustache が完了。

## 2026-06-22: hash 要素 cell の pair-value デコンテナ化（#TBD）

- **原因**: hash 要素は `ContainerRef` cell で格納される（第一級コンテナ Phase 2）。`%h<k>` 読みと `.values` は

## 2026-06-22 (session 11): `for $blob` / `.map` / `.grep` が Blob のバイト列を反復 — MIME::Base64 アンブロック

**5 つ目の動作モジュール MIME::Base64**（own test 3/4 ファイル完全パス：basic 18/18, rfc4648 16/16, oneline 2/2。

  `value_to_list` 自体は不変（`my @a = $buf` は `[Buf]` のまま）。
- **既知の divergence（受容）**: Blob に itemization マーカーが無いため、itemized な Blob（`my $b = Blob.new(...)`・

  `pack("S2",a,b)` は `a` のみ）、各数値ユニットは 1 アイテムを消費。反復はレター反復（`"S" x $n`）で表現。`unpack` では
  count は意味を持つ（`"C*"` は残り全バイト抽出）。`unpack` は結果 1 要素のときベア値（`Int`）に collapse（raku 一致）。
- **配線**: `experimental` を compiler の no-op pragma リストに追加。`pack`/`unpack` を `BUILTIN_FUNCTION_NAMES` と
  `native_function_variadic` に追加（arity 非依存ルーティング＝`chrs` と同方式）。`pack` は末尾アイテムを flatten
  （`pack 'S' x N, $list` が単一 List 引数を個別要素として見えるよう）。`unpack` の sub 形式は Blob（Instance）引数を取るため
  `try_native_function` の Instance ゲートを通す例外に追加。メソッド形式 `.unpack` は `methods_dispatch_match.rs` に追加。
- **テスト**: `t/pack-unpack.t`（24 件、全て raku でも PASS）。副産物として MIME::Base64 の `binary-and-long-line.rakutest`
  が 0→7/11 に改善（残り 4 は `blob16`/`blob8` 型付き Blob 宣言 + UTF-16 encode/decode 待ち＝別機能）。roast 影響なし
  （`pack` を使う roast テストは存在しない）。

## 2026-06-22 (session 11): ユーザインスタンスのデフォルト `.gist` を `ClassName.new(...)` に修正

定義済みユーザクラスインスタンスの デフォルト `.gist` を、型オブジェクト形式 `F()` から raku 通りの
`ClassName.new(attr => value, ...)`（= `.raku` と同形）に修正。`say $obj`（gist を使う）が `F()` ではなく
`F.new(...)` を表示するようになった。`($var = expr).=method` や `$o .= meth` の round-trip も正しく動く。

- **修正**（`methods_instance_ops.rs`）: インスタンスの `.raku`/`.perl` を構築する分岐の条件に `gist` を追加
  （`gist_default`）。型オブジェクトは従来通り `(ClassName)`、ユーザ定義 `gist` メソッドは優先。
- **coercion gist のガード**: Real/Numeric/Stringy を do するクラス、または `Bridge` メソッドを持つクラスは
  数値/文字列ブリッジ gist を保持（`class Plain does Real { method Bridge {...} }` は `Plain.new(...)` ではなく
  その数値 `7` を gist）。`does_check` だけでは `does Real` が見えないため native-bypass と同じく `Bridge` メソッド
  検出も併用。
- **callable インスタンスのガード**: Method/Submethod/Sub/Routine/Block/Code/WhateverCode/Callable のインスタンスは
  名前ベースの独自 gist を保持（`Method` は `Method.new(...)` ではなく `foo`）。これがないと
  `roast/S10-packages/precompilation.t` test 31（RT123276、`Method` の gist 比較）が回帰した（CI で検出・修正）。
- **ローカルテスト更新**: `t/class.t` が旧（誤った）`Point()` を期待していたのを raku 準拠の `Point.new(x => 3, y => 4)` に

- **テスト**: `t/instance-gist.t`（10 件、全て raku でも PASS）。型オブジェクト gist・カスタム gist 優先・Real ブリッジ
  gist・ネスト・`.=` round-trip をカバー。`make test` PASS（t/ 回帰ゼロ）。
