## Overview

July carried forward the momentum from late June, advancing the dispatch cluster (wrap/dispatching) and the remaining S17 concurrency-infrastructure work in one push. For detailed remaining items and in-progress analysis, see [TODO_roast/BLOCKERS.md](../TODO_roast/BLOCKERS.md).

## 2026-07-15: roast-wide raku-vs-mutsu speed measurement + perf re-baseline

Measured wall-clock of raku vs mutsu (release) across all 1358 runnable
whitelisted roast tests (`scripts/roast-speed-diff.sh`, single-run; ratio =
mutsu/raku, so <1.0 = mutsu faster). **mutsu beats raku (Rakudo 2022.12) on
~1348/1358 tests, usually 2–30×** thanks to fast startup + JIT. The only files
where mutsu is meaningfully slower all converge on **one root cause — the
interpreter function-call path in hot loops**:

- `S04-declarations/state.t` **4.2×** (`for ^2000000 { $ = foo }`),
  `S06-signature/named-parameters.t` **2.6×** (`for ^1000000 { foo(:color($_)) }`),
  `S07-iterators/range-iterator.t` **1.7×**, `S12-methods/private.t` **1.6×**.
- Isolated micro-repro: a 1M-iteration loop calling a sub is **1.85× slower**
  than raku (positional) / **2.6× slower** (named arg).
- **The JIT bails at the call boundary** — the positional 1M loop runs at
  `MUTSU_JIT=on` 0.74s ≈ `off` 0.72s (JIT contributes nothing), so any loop that
  calls a sub runs the interpreter call path. Its profile is **~15% malloc/free
  churn per call** (frame + named-args structure + `Env::cow_mut` +
  `Arc::drop_slow`) plus per-call `current_package` / `Env::get_sym` /
  param-binding. Named args cost mutsu **+46%** vs raku **+6%** — a
  `String`-keyed named-args structure rebuilt every call.

This re-baselines the §5 perf roadmap: the call path is now the top perf target
(it hits real spec tests and the most common real-world shape), while the
`needs_env_sync` blanket and the `BlockScope` declone — both investigated the
same day — turned out to polish benchmarks mutsu already wins (see below). The
synthetic `bench-grammar-parse` 6.4× gap does not surface in the roast whitelist.

## 2026-07-15: `once` fires exactly once per clone (threads, repeated calls, methods)

`once { ... }` broke in three independent ways (PLAN §3). Fixed across two PRs:

- **Subs / closures / threads (#4567).** The site key was built from a global
  compile-time counter, so a worker thread — which re-OTF-compiles the routine
  into a fresh body — drew a different key and never matched a peer's fire; and
  a named sub's second call took a fast/light call path that skipped the
  clone-id setup, keying under the root scope and re-firing. The site key is now
  the `OnceExpr` op's **bytecode position** (deterministic across recompiles),
  the store is an `Arc`-shared `OnceStore` with a **claim/condvar** protocol
  (fires once even under a genuine race — a thread re-entering its own
  in-progress `once` is served `Nil` rather than deadlocking), and
  `once`-bearing routines are kept off the fast/light paths via
  `CompiledCode::has_once` so every call runs the same clone-id setup.
- **Methods (this PR).** A method's `__mutsu_callable_id` is a fresh *per-call*
  id, so a method's `once` had no stable clone key and re-fired every call.
  `once_scope_key` now derives a method's clone key from the innermost routine
  frame's `(owning class, method)` — stable per clone, and consulting the
  *innermost* frame means a `once` in a closure nested in the method still keys
  on that inner clone. This matches Rakudo: once total for a normal/inherited
  method (the defining class's single clone), once per composing class for a
  role method, and it leaves the per-call id recursive `return`-targeting relies
  on untouched. `once` now also fires once across threads for methods.

Pins: `t/once-clone-scope.t` (10 cases), `t/once-phaser.t` (12);
`roast/S04-statements/once.t` stays whitelisted.

## 2026-07-15: map routine callbacks, recursive index-rw temp corruption, (<=) Range — 99problems-21-to-30.t 6/15 → 14/15

Three general fixes root-caused from the file's abort:

- **`return` in a `sub` callback invoked via map ends that call** (routine
  semantics): `map $f, @xs` / `@xs.map($f)` with a compiled routine `$f` now
  routes through `call_sub_value` instead of the inline bare-block fast path
  (whose escaping return signal surfaced as X::ControlFlow::Return). Bare/
  pointy blocks, WhateverCode, and placeholder blocks (`{ $^x }` — Blocks,
  not Routines, and Pair elements must not bind as named args) stay on the
  fast path.
- **Recursive slice-arg writeback corruption**: the call-site Index-argument
  `is rw` writeback temps (`__mutsu_index_rw_*`) are compile-time-fixed
  global names, so a recursive execution of the same call site (or the
  callee's env merge) clobbered the caller's pending compare, firing a bogus
  `@xs[1..*] = ...` that exploded on an immutable List (P26 `combination`).
  Fixed on both ends: the cross-frame env merges (compiled-call, method,
  interpreter-sub) never copy `__mutsu_index_rw_*`/`__mutsu_call_result_*`
  back to the caller, and the writeback adds an `eqv`-against-current-slot
  guard so a no-op difference does not fire.
- **`(<=)` enumerates a Range operand**: `coerce_value_to_quanthash` lacked
  the Range arm its `coerce_to_set` sibling had, so `@n (<=) (1..49)` built a
  one-element Set of the string "1..49".

Remaining 1 subtest (P27 nested-map Seq structure) is recorded in BLOCKERS.
Pin: t/map-sub-return-index-rw.t.

## 2026-07-15: statement-form loop-phaser scope + sub-body INIT lifting — advent2012-day15.t whitelisted (11/11)

Three fixes, one per layer:

- **Statement-form loop phasers share the enclosing block's scope** (rakudo
  semantics): `NEXT (state $best) max= $score;` declares `$best` where the
  loop body can see it, so `LAST say "BEST: $best"` works. The parser marks
  the statement form as `[SyntheticBlock([stmt])]` (block form keeps its own
  `Stmt::Block` scope in `expand_loop_phasers`); the phaser-reorder pass
  preserves the marker instead of flattening it.
- **INIT/CHECK inside a named sub body are program-init-time**: the phaser
  lift now recurses into `Stmt::SubDecl` bodies, so `my $fh = INIT Open(...)`
  calls Open before the mainline runs (and memoizes the value for call time).
- **The I/O ops no longer seed not-yet-declared locals into env**: Say/Put/
  Print/Note's pre-sync (`sync_env_from_locals_declared`) skips slots whose
  name is absent from env — a premature seed made a later `state` declaration
  in the same loop body record itself as a "shadowed outer binding", and the
  loop-local restore wiped the variable at loop exit (the same guard the
  regex-interpolation sync already had).

## 2026-07-15: child-class attribute accessors now shadow parent methods (MRO-level accessor dispatch) — zef populate 3-5min → `zef info Zef` ~50s

An mzef-campaign session (PLAN §1 B2) that turned out to be a general **method-dispatch
correctness bug**: mutsu treated "user-defined method beats auto-generated accessor" as a
global rule, so a **parent class's explicit method shadowed a child class's `has $.x`
accessor** of the same name. In Raku the accessor is an ordinary method of its declaring
class and participates in the MRO like any other method (child accessor wins).

- **Impact on zef**: `Zef::Distribution has $.name`, but its parent
  `Zef::Distribution::DependencySpecification` declares `method name` = spec-parts =
  `Zef::Identity.new` = a full REQUIRE identity-grammar parse (~4ms/call, plus `.Str`/`.trans`
  churn). Every `.name` read during Ecosystems populate paid a grammar parse. This — not the
  suspected "plain Named subrule per-call cost" — was the bulk of the populate slowdown.
- **Fix**: new `resolve_user_method_or_accessor` (class_introspection.rs) decides the winner
  **per MRO level**: explicit class-local method > public attribute accessor > role-composed
  method (class prioritization, 6.c S14-roles/attributes.t), walking the MRO once so a
  more-derived accessor beats a less-derived method. Applied at 4 dispatch sites: the
  interpreter instance-dispatch gate (methods_instance_ops), the fast accessor read
  (vm_call_method_ops — the shadowing case now takes the *fast* attribute read), the
  compiled-path defer check (vm_call_method_compiled_interpret, prefiltered to
  role-origin/ancestor-owned resolutions so the hot path stays cheap), and the lvalue
  assignment path (methods_mut_method_lvalue — a shadowing `is rw` accessor is writable).
  Replaced the old `has_class_local_method` MRO-blind predicate.
- **Numbers** (release, real fez index 7648 dists): `.name` micro-stage 1.31s → 0.02s (300
  dists, 65x); full fez populate ~60s → **19.5s** (raku 2.8s); warm `zef info Zef`
  end-to-end **~50s** (raku warm ~3.8s). Bonus: the short-name index now has **9259/9259
  keys, exactly matching raku** — the former 3-key diff was the same bug (dists whose
  identity fails to parse lost their `.name`).
- pin = `t/accessor-mro-shadowing.t` (9 cases: shadowing both directions, same-level
  method-suppresses-accessor, role-vs-class prioritization both levels, rw write path,
  deep hierarchy).

Remaining populate hot spots (per dist): `bless` on the `run_instance_method` slow path,
TWEAK x2 via the resolver path, allocation churn — see PLAN §1 B2.

## 2026-07-15: custom grammar metaclasses (Metamodel::GrammarHOW + EXPORTHOW) — advent2011-day07.t whitelisted

The last "major metamodel work" item of the integration cluster:
`Advent::GrammarProfiler` (a grammar profiler that swaps in a custom metaclass
via `EXPORTHOW.WHO.<grammar> = ProfiledGrammarHOW`) now works end to end, and
`integration/advent2011-day07.t` passes (whitelist 1409/1463). Four general
mechanisms, no test-specific hacks:

1. **`Metamodel::GrammarHOW` is inheritable** (added to the builtin parent
   list next to `Metamodel::ClassHOW`), and `callsame` from a user HOW method
   now reaches the **native** metamodel implementation once the user MRO is
   exhausted: `run_instance_method`/`push_method_dispatch_frame` record a
   metamodel dispatch context (gated by a registry flag so ordinary programs
   pay one bool read), and `dispatch_next_candidate` serves the default
   `find_method` (via `classhow_find_method`) / `publish_method_cache` as the
   final candidate.
2. **EXPORTHOW installation**: a `grammar` declared while an
   `EXPORTHOW::<grammar>` stash entry is visible gets an instance of that HOW
   (`install_custom_grammar_how`) — recorded as its `.HOW` and, when the HOW
   class defines a user `find_method`, in `Registry::grammar_custom_how`.
3. **Token methods are callable**: `Routine { is_regex: true }` stubs (what
   `^find_method` returns for a token) called as `$meth($cursor)` run the
   token anchored at the cursor position and return a Match
   (`try_call_token_method_value`, hooked into both `call_sub_value` and the
   VM's `vm_call_on_value`).
4. **Subrule dispatch routes through the custom HOW**: the regex engine's
   named-subrule path consults `grammar_custom_how` and calls
   `find_method(TypeObj, name)` in a scratch interpreter; a returned wrapper
   closure is invoked with a cursor and its Match converted back into an
   engine candidate (inner captures recovered via a thread-local side
   channel). Module-level `our %timing` mutations inside the wrapper persist
   because env values are shared cells.

Pin: `t/exporthow-grammar-how.t`. Known simplifications (TODO comments in
code): EXPORTHOW is approximated as globally visible rather than lexically
scoped, and left-recursive rules bypass the LR seed-growing loop on the
custom-HOW path.

### Archived: breakdown of integration cluster ① (stack-overflow aborts; all 4 root-caused 2026-07-14)

The initial diagnosis "deep recursion overflows the stack" was **wrong**. The aborts look
identical but the causes are distinct:

| File | Actual cause | Status |
|---|---|---|
| `99problems-41-to-50.t` | **Infinite recursion.** A closure's frame env was a scoped child of the *caller's* env, and the captured env was merged with "don't insert if already present", so a caller lexical of the same name shadowed the callee's own capture (i.e. lexical scoping degenerated into dynamic scoping). The P46 actions build closures that each hold their own `my @args` per clause, so an inner clause read the outer `@args` (which contained itself) and self-recursed | **Fixed in #4510.** P41 and P46 pass. Lesson: **the shape of the abort (`stack overflow`) says nothing about the cause.** Count the
depth and compare against raku first, to separate infinite recursion from finite deep recursion
(you can tell instantly when `raku` finishes in 8 calls while mutsu emits a 1900-line trace).
3 of the 4 were infinite recursion, and the remaining one was a **measurement error**.

A second lesson: **the ① measurements were taken with a debug build.** Debug consumes several
times the native stack per Raku frame compared with release, and
`deep-recursion-initing-native-array.t` (~20,000 levels) overflowed only in debug. CI uses
release, so **any failure where stack depth is suspect must be confirmed with a release build**
(the same point as the "Delegate the full roast run to CI" section of CLAUDE.md).

The decisive probe when suspecting a dual-store (locals ↔ env) bug: **read the same variable
twice within a single expression**.

```raku
sub f($tree, $d) {
    say $tree.WHAT.^name ~ ' | ' ~ $tree.WHAT.^name;
    return if $d >= 1;
    f($tree[1], $d + 1);
}
f([3, Any, Any], 0);
# raku:  Array | Array  /  Any | Any
# mutsu before the fix:    Any | Array   ← first read correct; second rolls back to the caller's value
```

### Archived: man-or-boy diagnosis (2026-07-14, measured after #4516)

Tracking the identity of each `&x` with `.WHICH` (`Sub|ID`) pinpoints the divergence point.
**The probe must not *call* `&x`** (`x4('TAG')` would fire the B closure and change the execution
itself). Even if you give it a name like `sub bee`, mutsu returns an empty `&x.name` (a separate
small gap), so use `.WHICH` for identification.

```
A#1 k=4 d=0 B=Sub|33  x=[27 28 29 30 31]    <- the original 5 literals
A#2 k=3 d=1 B=Sub|41  x=[33 27 28 29 30]    <- A#1's B fired from inside A#1 = correct
...
A#5 k=0 d=4 B=Sub|65  x=[57 49 41 33 27]    <- k<=0, so it calls x4 (=Sub|33 = A#1's B)
A#6 k=2 d=1 B=Sub|73  x=[33 57 49 41 33]    <- expected [33 27 28 29 30]
```

A#6 is "A#1's B fired **from inside A#5**". B's body is
`A(--$k, $B, &x1, &x2, &x3, &x4, ...)`, so `&x1..&x4` must be read from **A#1's captured env**
(= 27,28,29,30), but mutsu reads `&x1..&x4` from **the calling frame A#5** (= 57,49,41,33).
Invocations fired from inside A#1 (A#2) happen to have a coinciding chain and thus look correct;
**it breaks the moment B fires from a different frame**.

This is the same bug as the shadowing fixed in #4510. #4510 overwrite-installs only "plain
lexicals the creator does not modify after capture" (`CompiledCode::authoritative_free_vars`);
`&x1..&x5` are not in that set, fall into the don't-overwrite path, and lose to the caller's
same-named parameters.

**Resolution (2026-07-15)**: the cause was two layers deep, and both had to be fixed (which is
why experiments fixing only one of them "did not work"):

1. **Free-var scan omission**: reads of `&x1` go through `GetCodeVar`/`CallOnCodeVar`, whose
   constants are the sigil-less `"x1"`. They are not included in `op_name_const_idx` (the
   GetGlobal family), so they never entered `free_var_syms` at all, and the capture filter
   dropped them (`&x1` is a plain user lexical, so the system-name rules would not keep it
   either) — **they were not even captured** = resolved purely via dynamic scope.
   → The scan now records them under the `&`-prefixed key `&x1`.
2. **rw-arg sink exclusion**: A forwards `&x1..&x5` as arguments in `x4($k, &x1, …)`, so
   `own_call_arg_sources` rejected the vouch. But **raku rejects `is rw` on `&`-sigil
   parameters** ("Can only use 'is rw' on a scalar ('$' sigil) parameter"), so per spec there
   is no path for a `&` argument to be rebound by the callee → exempt `&` names.
   (Rebinding via `is raw` theoretically remains, accepted as a known gap. A direct `&f = …`
   rebind is a name-write and is captured as before.)

**Remaining gaps also resolved (follow-up commits in the same PR)**:

- **Bare calls `x()`**: the `CallFunc`/`ExecCall` family record the read only as a callee name,
  so it was outside the free-var scan. Registering every callee name would bloat
  `free_var_syms` with `&say` etc. (paying a `free_at_entry` lookup per closure call), so only
  callee names that match **the set of enclosing `&`-sigil lexicals visible at the definition
  point** (`CompiledCode::outer_code_var_names`, propagated transitively from the parent's
  `local_map` by `compile_closure_body`/`compile_sub_body`) are recorded as `&name` free vars.
  Zero bloat, lexically precise.
- **Authority loss across multiple nesting levels** (a pre-existing bug that hit scalars too):
  the vouch was burned only onto "direct children of the creator", so the inner closure in
  `sub mk($x) { my $mid = sub { sub { $x } }; $mid.() }` could not be vouched by the
  intermediate frame (`$x` is not the intermediate's local) and was shadowed again by the
  caller's same-named lexical. Since the vouch already guarantees "the entire subtree performs
  no writes" (nested free-var writes fold up to the creator), this was resolved by
  **propagating authority down to descendants as long as no intermediate closure redeclares the
  same name** (`propagate_authoritative_down`).

## 2026-07-15: J4d round 3 — making the light-call ceremony alloc-free (#4529)

Even after round 2, fib was ~50% per-call ceremony. Three cuts that leave soundness untouched:

- **`Env::scoped_child` made steady-state alloc-free**: it used to pay 2 `Arc`s per call
  (fresh allocation of an empty overlay map + cloning the walked-to parent Env and re-wrapping
  it in an Arc); now the empty-tier walk reuses the existing parent `Arc` directly, and the
  empty overlay is a process-shared singleton (on first insert, `cow_mut`'s `Arc::make_mut`

- **Skipping the param-slot seed**: `positional_light` seeded every local from the env, but
  param slots are unconditionally overwritten by the bind immediately after (arity shortfall
  already early-returns before the seed) = pure waste. In an all-params body like fib, the
  seed (a per-call overlay-chain hash walk) disappears entirely.
- **Pool reuse of the args Vec**: the pos-light call site's
  `stack.drain(..).collect::<Vec>()` (a malloc/free per call on the hottest path) now borrows
  from and recycles into the locals pool.

## 2026-07-13: baseline optimizations round 5 — retiring `SetSourceLine` for a static ip→line table (ADR-0006 §2.3 complete) + ★the "opcode count ≠ time" lesson

An instruction's line number is **static data fixed at compile time**, and no dispatched
instruction is needed to carry it. Retired `SetSourceLine`, which ran once per statement
(**1.9M of bench-fib's 8.9M executed opcodes = 21%**, 11% on time-parts), moving it to
`CompiledCode::op_lines` (a `Vec<u32>` parallel to `ops`) (#4489).

- **Compiler**: `Stmt::SetLine` emits no opcode; it only advances the emit cursor
  (`set_emit_line`). `emit()` bakes the current line into each instruction. The two sites that
  `ops.insert` — the `SetLocalDecl` fusion (#4488) and the accessor ref marker — update
  `op_lines` in lockstep, with index consistency pinned by `debug_assert`.
- **VM**: `cur_source_line` is pulled via `sync_source_line(code, ip)` **only by instructions
  that can observe the line** (each arm for calls / re-entry into user code / frame push /
  declaration registration / raise sites, plus the JIT call shim). Pure hot ops — arithmetic,
  locals, branches — cost 0.
- **★The most important lesson (measured)**: **the reduction in opcode count does not match the
  reduction in time.** The first naive version refreshed the line for **every instruction** in
  the `exec_one` prologue, which cost more than the dispatch it removed — **+7.8% instructions
  on fib**. Only after restricting to "observation points only" did it become **-3.4%** (fib,
  interpreter path). `SetSourceLine` is the cheapest possible op (one store), and mutsu's
  average op is orders of magnitude heavier, so a 21% opcode reduction is not a 21% time
  reduction. **±0% on the JIT path (the default configuration)**; bytecode size down ~15%.
  → Before tackling the next administrative ops (`SetVarDynamic` 500k, `CheckReadOnly`),
  **first confirm with perf instruction counts that the op actually consumes time**
  (measurement procedure = [ADR-0006 §4](../docs/adr/0006-baseline-interpreter-optimizations.md)).
  A profile retaken the same day (release, JIT on, P-core pinned, `cycles:u`) showed the real
  targets: on bench-fib, **the JIT native code body itself runs only 5.7%**, dominated by
  allocation (`_int_malloc` + `_int_free` 11.8%), env table duplication and hashing
  (`Env::scoped_child` 5.4% / `get_sym` 4.4% / **SipHash 4.1%** /
  `hashbrown RawTable::clone` 3.7%), and the call path (10.9%).
  bench-class is **~20% in the allocator alone** + 5.2% `memcmp` on String-keyed attribute
  names. → Next targets: (1) the env's per-call clone and SipHash (= the lexical slot
  campaign); (2) Symbol-ifying the attribute `HashMap<String,Value>` (recorded with a table in
  PLAN §5).
- **By-product: line numbers actually became more accurate.** The old scheme left the line
  stale after executing a block/loop body, requiring a defensive re-emit shim of
  `SetSourceLine` before method calls (`emit_source_line_if_known`). Deleted that too.
- pin = `t/source-line-table.t` (top level / inside a loop body / after executing an inner
  body / inside an expression containing a `do` block / sub→sub caller line / backtrace line
  of a sub that died — all 7 cases match raku).

## 2026-07-13: baseline optimizations round 4 — fusing declaration sequences (ADR-0006 §2.3 peephole, first half)

`my $x = <expr>` dispatched 3 instructions every time: `MarkExplicitInitializerContext` +
`MarkVarDeclContext` + `SetLocal`. The two markers' only effect is "set a flag that the
`SetLocal` body reads and clears", so they were fused into the single instruction
`SetLocalDecl { slot, explicit_init }` (#4488).

- **Safety**: the fusion rewrites, inside `CompiledCode::emit()`, **only the marker sequence it
  itself just pushed**. Jump targets are recorded only at statement boundaries, and the 3
  instructions are emitted contiguously, so a path that "jumps into the middle of the markers"
  structurally cannot exist (even if a path jumped to the first marker, the fused instruction
  does the same thing as that marker sequence + `SetLocal`, hence equivalent). A post-pass that
  deletes ops would require reindexing jumps/compound ops; this avoids that entirely.
- `compute_free_vars`' "declaration vs reassignment" distinction (keyed on the presence of
  `MarkVarDeclContext`) and the JIT (added the dedicated helper `set_local_decl`) were also
  adapted to the new instruction. The JIT still compiles the hot loop (compiles=1 / bailouts=0).
- **Measured**: time-parts executed opcodes **6.40M→5.40M (−15.6%)**. Together with §2.1,
  7.60M→5.40M (−29%). Wall-clock in release: time-parts 0.63→0.58s (interp), 0.58→0.55s (JIT on).
- pin = `t/decl-marker-fusion.t` (declaration vs reassignment, with/without explicit
  initializer, closure capture. Same results under raku).
- **Next target**: `SetSourceLine` is **1.9M of bench-fib's 8.9M executed opcodes (21%)** and
  600k (11%) on time-parts. The right fix is to delete the opcode and move to a static ip→line
  table (PLAN §5 Lever 7 §2.3-b).

## 2026-07-13: baseline optimizations round 3 — `constant` inlining + constant-condition DCE (ADR-0006 §2.2)

Guarded logging like `constant DEBUG = False; if DEBUG { note ... }` is a frequent pattern in
real code, but mutsu **executed** it every time as `GetBareWord` + `JumpIfFalse` (#4487).

- **Implementation**: track the values of `constant`s whose initializer is a compile-time
  constant scalar and replace reads with `LoadConst` (this propagates to child compilers =
  sub/closure bodies, so a file-level `constant` read from inside a sub is inlined too).
  `if`/`unless` whose condition thereby becomes constant resolve the branch at compile time.
- **Safety design**:
  - **Never delete a dead branch containing a declaration.** Raku installs declarations at
    compile time even if the branch is never taken, so only branches consisting of "plain
    expression statements only" (= the logging-guard shape) may be removed.
  - **No inlining when a non-constant local of the same name exists.** mutsu strips sigils in
    the AST, so `my $DEBUG` and a sigilless `constant DEBUG` land on the same `local_map` key.
    A name in `local_map` but not in `constant_vars_in_scope` = treated as a shadow and falls
    back to normal resolution.
  - Scope lifetime is identical to `constant_vars_in_scope` (a constant that leaves its block
    reverts to an `our`-scoped package symbol and is not an inlining target).
  - Container-valued constants are excluded since their identity is observable (per ADR §2.2).
- **Measured** (release, median of 5): debug-guard executed opcodes **3.00M→2.60M (−13.3%)**,
  with 200k each of `GetBareWord` and `JumpIfFalse` completely gone. Wall-clock:
  **JIT on 0.34s→0.25s (−26%)**, interp 0.32s→0.26s. Ratio vs raku **2.2x → 1.3x**.
- pin = `t/const-inline-dce.t` (16 cases; verified raku produces the same results).

## 2026-07-13: baseline optimizations round 2 — constant pool dedup (ADR-0006 §2.4)

`CompiledCode::add_constant` unconditionally did `constants.push`, so the same name string or
the same literal was duplicated into a separate slot per use site (#4486).

- **Implementation**: added a reverse index (`FxHashMap<ConstKey, u32>`) to `add_constant`.
  Only scalars that are *indistinguishable when equal in value* (Int/Num(bit)/Str/Bool/Rat)
  get keys; values with **observable identity** like containers or Instances get no key and
  always take a new slot. `0.0` and `-0.0` are distinguished by bit pattern (`1/0.0` and
  `1/-0.0` differ, so they are not merged). The index is compile-time scaffolding, discarded
  at `compute_needs_env_sync()` (the chunk-finalization point), so executed code carries no
  extra memory.
- **Measured** (the new `const-pool:` line of `MUTSU_VM_STATS=1`): **29–45% of `add_constant`
  calls share an existing slot** (time-parts 44.8%, roast S06 workout.t 28.9%, t/wrap.t
  37.1%). The constant pool's real size shrinks accordingly, benefiting `CompiledCode` memory
  and cache locality (same direction for mzef's tens-of-thousands-of-dists compile).
- Direct effect on execution speed is small (as anticipated in ADR §2.4).

## 2026-07-13: baseline optimizations round 1 — constant folding (ADR-0006 §2.1)

Implemented **§2.1 constant folding**, the first of the measures adopted in
[ADR-0006](../docs/adr/0006-baseline-interpreter-optimizations.md) (#4485). Pure expressions
consisting only of literals (`60 * 60 * 24`, `'a' ~ 'b'`, `-1`) are evaluated at emit time and
folded into a single `LoadConst`.

- **Implementation**: `src/compiler/const_fold.rs`. Evaluation calls **the same native
  operations as the VM** (`builtins::arith_add/sub/mul/div/mod/pow`,
  `Interpreter::concat_values`) directly, so `Int/Int → Rat`, the BigInt promotion of
  `2 ** 100`, and Num formatting automatically match runtime (no separate "compile-time
  arithmetic" implementation = ADR §2.1-2). Covers arithmetic (numeric operands only), `~`
  concatenation, and numeric comparison (only Int/Int and Num/Num, where the VM has a
  fast path). String operands like `'3' + 4` are not folded, preserving the VM's numeric
  coercion path (which can throw X::Str::Numeric).
- **Operator-override safety condition (ADR §2.1-1)**: `sub infix:<+>` overrides even native
  `Int + Int`, and mutsu registers the declaration in the global `user_declared_infix_ops`,
  so if even one declaration exists anywhere in the unit (nested block, sub body, class body),
  folding is forbidden. A static AST walk was rejected ("miss one spot = mis-fold"); instead,
  **the compile sites a declaration must pass through (`SubDecl`/`ProtoDecl`/`VarDecl`) mark a
  shared `FoldCtx`, and if a declaration shows up after folding has already happened, the
  whole unit is recompiled with folding off**. Structurally sound (no walk can fall behind),
  and the double work is paid only by files that declare operators. Additionally, folding is
  disabled when the JIT's `USER_INFIX_DECLS` (a process-wide counter) is nonzero (covers
  EVAL/blocks compiled after module loading).
  pin = `t/const-fold-infix-override.t` (even a `1 + 2` written before the declaration is not
  folded).
- **Measured** (release, median of 7, local): time-parts executed opcodes
  **7.60M → 6.40M (−15.8%)**, `Mul` 1.0M→0.4M, `LoadConst` 1.7M→1.1M.
  Wall-clock: interp **0.65s → 0.61s (−6.2%)**, JIT on 0.60s → 0.59s (the JIT already folds
  these constant `Mul`s natively, hence the small delta). Time drops less than the opcode
  reduction rate because time-parts' remaining time is dominated by administrative ops
  (`SetSourceLine`, `MarkVarDeclContext`, `SetVarDynamic`, `CheckReadOnly`) — that is ADR §2.3
  peephole territory, the target of subsequent slices.
- debug-guard is unchanged by this slice (reads of `constant DEBUG` = §2.2's job).
- Side effect: folding also shortens the JIT's input opcode sequence (ADR §4).
- `MUTSU_CONST_FOLD=0` disables folding (for A/B and isolation).

## 2026-07-12: layer-4 JIT J1 complete — hot Tier A chunks run native on a Cranelift skeleton (#4471)

Achieved the **J1 gate** of ADR-0004 §2.5: with `MUTSU_JIT=on`, fib runs under the JIT

- **Approach (ADR-0004 §2.3 Tier A = subroutine threading)**: supported opcodes are
  transliterated into a sequence of `extern "C"` helper-shim calls (`vm_jit_helpers.rs`, each

  NumLt JumpIfFalse Jump ContainerizePair SetSourceLine Return CallFunc.
- **feature `jit`** (default on; inert at runtime until `MUTSU_JIT=on`). The wasm build is

  vm-stats gained `jit: compiles/entries/bailouts` + a bailout opcode histogram, and

## 2026-07-11: GC — candidate buffer now holds `Weak` (grammar-parse acyclic churn resolved, PLAN §2)

Resolved the churn where zef's dist-identity grammar parse buffered every acyclic
`Match`/capture tree as a cycle candidate while the collector reclaimed `reclaimed_cycles=0`
every time ([docs/grammar-parse-gc-churn.md](../docs/grammar-parse-gc-churn.md)) by
**changing candidate-buffer retention from strong `Arc` clones to `Weak` handles
(`ErasedWeakGc`)**.

- A ceiling measurement (direction (3) in the doc) identified the real harm: wall-clock is
  nearly identical GC on/off (~1%), but **RSS was 153MB vs 43MB (3.6x)** — the buffer's
  strong retention kept refcount-dead garbage (whole capture trees) alive until the next
  collect. The ~54% `drop_in_place<HashMap<String,Value>>` in the CPU profile was a
  *concentration* of deallocation cost paid even with GC off, not an excess.
- Soundness of weak retention: true cycle garbage by definition keeps itself alive (internal
  edges are strong), so `Weak::upgrade` at drain time always succeeds and trial deletion is
  unchanged. Refcount-dead entries are skipped on upgrade failure (the header is gone with
  the node, so it is never touched).
- Effect (zbench, debug, interleaved measurement): RSS **3.6x → +6%**, pause_max 41ms →
  3.6ms, pause_total 210ms → 99ms, dead reaching the collector **244,598 → 0** (moved to
  inline deallocation). `MUTSU_GC=on` wall-clock is noise-level vs off (~2%).
- Derived simplification: deleted the off-thread deallocation helper for 16k-node dead
  batches inside collect (dead can no longer accumulate in the buffer). The dead partition
  remains in a degenerate form covering only the narrow race window of "the last handle
  dropped after drain".
- Remaining: suppressing the candidate pushes themselves (~510k per 200 parses) is a separate
  axis (the practical harm — memory retention — is resolved).

## 2026-07-08: properly fixed the parallel-load races in `S17-supply/syntax.t` (test 53 / test 57)

Right after whitelisting, `S17-supply/syntax.t` was hitting 2 races under full-file parallel
load (passes standalone but fails deterministically under `make roast`'s high parallelism =
a CI blocker). Instead of "flaky, just re-run", both were fixed properly.

- **test 53 "Can only be in one whenever block at a time" (whenever mutual exclusion within a
  supply block)**: Raku guarantees that per `supply {}` / `react {}` instance, "only one
  `whenever` block can execute at a time". mutsu ran live `whenever` callbacks synchronously
  on the thread that emitted to the source Supplier (the worker of
  `start { $trigger.emit(...) }`), so two sibling `whenever`s ran concurrently on different
  threads from different triggers, interleaving emits. Fix = introduced a per-supply-block
  serialization lock. All `whenever`s registered in one on-demand supply-block body share the
  block's emitter `supplier_id` as their "serialization group", with a secondary map of each
  trigger `supplier_id` → group. On emit, this group lock is held for the whole callback
  execution (including while the `whenever` blocks in `await`), so a sibling `whenever` on
  another thread waits until the lock is released. The lock is per-thread reentrant (so
  re-triggering the same group on the same thread does not self-deadlock) and blocks across
  threads via a condvar (`acquire_supply_serialize` / `set_supplier_serialize_group` in
  `state_supplier.rs`). pin=`t/supply-whenever-mutual-exclusion.t`.
- **test 57 "multiple channels in whenever blocks work" (Channel close-before-drain race)**:
  the react loop's Channel branch did `poll_result()` (queue empty → `Ok(None)`) and
  `can_send()` (closed test) under **separate lock acquisitions**. In that TOCTOU window, if
  the worker ran `$c.send(3); $c.close()`, poll observed "empty" while the close test
  observed "closed", so the queued value 3 was never delivered and the subscription was
  marked done (the order degraded from `'123'` to `'12'`). Fix = added
  `is_drained_closed()` to `SharedChannel`, reading a `drained_closed` flag set atomically
  inside the mutex only when `send_closed` and the queue has become empty, and replaced the
  react loop's done test (`!can_send()`) with it. While values remain queued,
  `drained_closed` stays false, so polling continues and delivers. pin = the existing
  `t/react-whenever-shared-lexical.t` (the same `'123'` ordering scenario) now passes
  deterministically.

Both are in different subsystems (Supplier emit / react-Channel) and only manifested under
`make roast` parallel load (roughly 8/40 and 3-ish per 20-parallel respectively). After the
fixes: 0 failures across multiple rounds at 60-parallel.

## 2026-07-07: hoist nested-block `BEGIN` to compile-time equivalence (`S02-names-vars/variables-and-packages.t` whitelisted)

A `BEGIN` **inside** a bare block also runs at compile time in Raku, so a read that comes
**before** the BEGIN in source order within the block (e.g. via a forward-declared sub)
observes its side effects. Example:

```raku
{
    is baz(), 3, ...;   # expects 3,4,5 (raku)
    is baz(), 4, ...;
    is baz(), 5, ...;
    my $a; BEGIN { $a = 3 };
    sub baz { $a++ }
}
```

mutsu executed nested BEGINs inline at their source position, so `baz()` was called before
`$a` was initialized, yielding 0,1,2. The fix (`src/runtime/phasers.rs::reorder_at_level`)
threads `is_top` through the phaser-reordering pass and hoists BEGIN ahead of the block's
reads only in nested blocks — strictly gated so it never crosses a declaration/assignment/
initialization:

- Top-level BEGINs are still delegated to `run_toplevel_begin_phasers` and not hoisted by
  this pass (`is_top`).
- A nested BEGIN is hoisted only when "a read (`Say`/`Call`/…) precedes the BEGIN and there
  is no barrier (declaration, assignment, variable initializer)"
  (`first_read_idx < idx < first_barrier_idx`). This preserves the in-source ordering of
  `class C; BEGIN {…C…}` (`S04-phasers/begin.t` test 7), `has $.a; BEGIN {…set_build…}`
  (`S12-attributes/defaults.t`), and `my $x=42; BEGIN {…}; constant c=$x`
  (`S04-declarations/constant.t` test 27), while read-after BEGIN
  (`t/begin-compile-time.t` test 11) keeps working in place as before.
- Blocks containing CHECK/INIT keep the legacy behavior (hoist all BEGINs to the leading
  bucket).

This brings `variables-and-packages.t` to 39/39, now whitelisted (resolving the remaining
tests 29-31; 32-34/37-38/39 had already been resolved by #4235/#4305 etc.). Regression test =
`t/begin-nested-block.t`.

`@nbchars` in a trailing BEGIN and reads `.elems` at the top) failed because of this.

**Fix (`src/runtime/run_prelude.rs::run_toplevel_begin_phasers`, `src/runtime/run.rs`)** —
the approach BLOCKERS recommended: "not AST reordering but **eval the BEGIN body at compile
time in a sub-interpreter → reflect into the initial state**":

1. In `run()`, **before** `reorder_phasers` (after `preregister_top_level_subs`), execute
   hoistable top-level `Stmt::Phaser{Begin}`s via `eval_block_value` and remove them from
   `body_main`. `eval_block_value` leaves plain lexical writes in the shared `env` (only
   `&`-callable keys are isolated), so the seed persists. The key point is removing them
   **before** reorder, which avoids the breakdown of the old revert (`hoist_begin_phasers`)
   where "the init of `my $c = @a.elems` gets bucketed before the BEGIN".
2. After hoisting, drop the no-init `my` declarations of seeded variables
   (`Literal(empty Array/Hash)`, `Literal(Nil)`) from `body_main` via
   `drop_seeded_noinit_decls` (descending one level into the SyntheticBlock of
   `my (@a,@b)`). This reproduces raku's "the container is created exactly once at compile
   time; the runtime declaration does not re-initialize". Declarations with a real init
   (`my @a = 9,9,9`) are not dropped, so the runtime init overwrites the BEGIN value
   (raku-conformant).
3. **The hoistable gate is conservative** (`begin_body_is_hoistable`): only pure
   value-assignment BEGINs whose body contains no declaration, BareWord, or Call are hoisted.
   `eval_block_value` diverges from the mainline VM in name resolution and sink-context error
   forcing, so BEGINs that could do symbol resolution or throw (class references / forward
   sub calls / `BEGIN { 1/0 .Int }`) are not hoisted and are left to the in-place path
   (including the `X::Comp::BeginTime` wrap).

Regression test = `t/begin-compile-time.t` (11 tests, green on both mutsu and raku).
`t/begin-phaser-begintime.t` (the BeginTime wrap of BEGIN errors) also has no regression.
BEGIN **inside** nested blocks (`variables-and-packages.t` tests 29-31) is outside this
top-level-only mechanism and needs a deeper layer of the declaration model separately (see
BLOCKERS §3.1).

## 2026-07-05: Phase B (GC) — **`MUTSU_GC` flipped to default on** (GC campaign layer 3a complete)

After the acceptance measurements and 3 finishing optimizations, **mutsu became an
implementation that reclaims cycles by default** (achieving ADR-0001's "GC is table stakes").
`MUTSU_GC=off` is the explicit opt-out.

A 4th item surfaced in the first CI run after the flip (fixed forward):

- **Offloaded the dead sweep's large-batch deallocation off the collector thread**. In
  thread-churn workloads (S17-lowlevel/thread.t), the dead become "env-scale hashes from
  thread clones × 16k per round", and their serial deallocation alone took 3.5–5.5 seconds
  per collect (a serialization of the deallocation cost that, with GC off, each worker paid
  in parallel on its own thread). The glibc arena-contention theory was rejected — swapping
  in mimalloc changed nothing — and it was concluded to be real work, O(garbage). After
  DESTROY queueing (finalize) finishes on the interpreter thread, batches of 1024+ nodes are
  dropped on a registered reclamation thread (`spawn_user_thread`) — being registered, later
  collects' STW correctly waits for it (and if it can't keep up, the existing
  requeue+cooldown deferral applies). Effect: pause 3.5–5.5s → **1.2–5.3ms**; thread.t
  recovered to +6% vs GC-off.

The 3 pre-flip optimizations (all reducing the standing cost of GC=on):

1. **A buffered-bit fast path in `Gc::drop`** (the same pre-test as Bacon-Rajan's Decrement):
   instead of passing an Arc clone to `buffer_candidate` and deduping via swap on every
   survivor drop, already-cancelled nodes take a single relaxed load. This dominated
   workloads that clone/drop the same handle in a hot loop (the Sub node of recursive subs
   etc.) — **fib25 +28.9% → +0.5%, bench-fib +29.8% → +3.0%**.
2. **Skipping the program-end collect**: the only observable effect of the exit-time collect
   is "DESTROY delivery to leaked cycles" (memory is freed by exit). When no user class/role
   defines DESTROY (`registry_has_destroy_methods` — the same predicate as the drain side, a
   single O(number of types) scan at exit; builtin native DESTROYs are out of scope) and
   log/verify/stats are disabled, skip it entirely. Stopped paying the ineffective
   "re-enumerate 4000 live suspects × huge captured envs" that took 160-220ms in bench-class.
3. **A tri-state AtomicU8 cache for `gc_enabled()`** (OnceLock's 2 loads → 1 relaxed load;
   called on every survivor drop path).

Acceptance measurements (release, best of 9 iterations, quiet machine; details = ADR-0003
§5): scalar/function hot paths are fib25 +0.5% / bench-fib +3.0% / int-arith +2.0% (nearly
free, as the type filter intended), churn ≤ +10%. **Only bench-class is +7.5–8.3%** (over
the <5% gate) — the residue is one per-drop branch on the pre-existing traffic of 16.5M
`Value` clone/drops over 28k instances; reducing the traffic is layer 3b (NaN-boxing)
territory. **Flipped with user approval** (recorded in ADR-0003 §5).

The only exception is the crate's own unit-test build (`cfg!(test)`), default off — parallel
`cargo test` shares process-global collector state (GC-on unit testing is covered by the CI
gc-stress job single-threaded). `tests/gc_stress.rs`'s baseline changed to explicit
`MUTSU_GC=off`. Update note added to design doc §9.1.

## 2026-07-05: Phase B (GC) — implemented the ADR-0003 production trigger (size threshold + adaptive backoff)

Implemented the production trigger of the now-Accepted
[ADR-0003](../docs/adr/0003-default-on-gc-trigger.md). With this, `MUTSU_GC=on`
**automatically reclaims cycles during execution** without stress variables (previously only
the program-end collect). The last mechanism before the default-on flip is in place.

- **Trigger**: `buffer_candidate` passes the post-push buffer length to `note_buffer_len`;

- **Adaptive backoff**: per collect, `threshold = clamp(BASE, 2 × survivors, MAX=1M)`

  spin drain/requeue).

  threshold is printed as `gc_threshold=` on the gc line of `MUTSU_VM_STATS` (the observation
  point for adaptation).

  reclaimed, output identical to GC-off. An 800-node live chain (cas-loop shape, BASE=64) =
  **only 4 collects**, with the threshold adapting 64→1604 (a fixed period would be ~13
  collects, repeatedly full-scanning the growing live graph). Same structure as PHP 7.3's
  Zend GC ("root buffer threshold + raise on fruitless collects").

## 2026-07-05: Phase B (GC) — fixed false reclamation (live-data corruption) from phantom edges of shared captured envs

The GC=on advisory roast's `S11-modules/require.t` was deterministically red on 1 subtest
(`$*PROGRAM.parent(2)` degrading to `/packages/...` and the chdir failing). Node-watch
instrumentation traced it to **the attrs node of `$*PROGRAM`'s IO::Path Instance being
falsely reclaimed at the first collect (cycle=0)**, with `drop_gc_edges` emptying the live
attribute map (`.Str` becoming ""). VERIFY was silent (the counts were consistently wrong, so
it could not detect it).

The cause was a **violation of Trace's 1:1 rule**: a closure's captured-env overlay map
(`Arc<SymMap>`) is Arc-shared among all closures from the same frame, but `SubData::trace`
inline-traced the map's contents from each holder. The map owns each `Gc` handle **exactly
once**, yet N sharers each claiming the edges causes trial deletion to over-decrement by N-1
phantom edges, turning live nodes with external references white and reclaiming them. This
was a missed application of the rule already established as `value_gc::uniquely_owned`
(trace shared Arc wrappers only when solely held). `LazyList::trace` was equally guilty.
Additionally, `drop_gc_edges` ran `values_mut()` (= a COW clone) on the shared map, running
`Gc::clone` during reclaim and polluting the counts — a co-resident secondary bug.

Fix: added `Env::gc_overlay_uniquely_owned()` (overlay Arc strong_count==1) and restricted
the env walks of `SubData`/`LazyList`'s `trace`/`drop_gc_edges` to sole-holder cases. Shared
maps lean conservative as external root holders (cycles reachable only through them are
deferred — never corrupted). Normal closure captures flattened via `clone_env` at capture
time are fresh maps = solely held, so the main closure-cycle reclamation capability is
unchanged (confirmed by all gc_stress reclamation tests passing).

pin = `value_gc::tests::shared_captured_env_edges_are_not_claimed_per_sharer` (a real 4-node
cycle of a shared map + 2 closures: the live hash is intact in both strong count and
contents, the real cycle is reclaimed — confirmed to fail reliably with the gate disabled).
`make test` 1513 files PASS; require.t GC=on 58/58.

## 2026-07-05: Phase B (GC) — fixed 2 liveness holes where spawn churn starves the cooperative STW

`roast/S17-lowlevel/semaphore.t` (RT #128628: rapidly spawning 4000 `Promise.start`s) hit a
deterministic CI timeout under `MUTSU_GC=on` (Wstat 124 in the gc-stress advisory roast).
From gdb thread dumps and `MUTSU_GC_LOG` pause analysis (pause_ns≈52ms dominating each
collect), identified 2 holes whereby **`try_stop_the_world` wastes 50ms on nearly every
attempt during short-lived-worker churn**:

1. **The birth window**: `spawn_user_thread` does `enter_mutator_worker` on the parent side
   (quiescence target +1), but the worker cannot contribute to QUIESCENT until it starts and
   reaches its first safepoint/safe region. During a spawn burst there is almost always a
   thread inside the birth window, so STW repeats 50ms timeout → 100ms cooldown. → **The
   parent pre-counts the not-yet-started worker as quiescent right before spawn**
   (`preregister_worker_quiescent`), and the worker's first action (`worker_started`) does a
   checked exit. The worker touches no Gc state until it passes the checked exit, so this is
   sound (if a scan is in progress it parks before starting).
2. **Missing exit notification**: a worker's exit (`exit_mutator_worker`) satisfies the
   quiescence equation by "lowering the target" but does not notify the rendezvous condvar,
   so a waiting collector sleeps out the full remaining timeout and only notices satisfaction
   at the deadline recheck (this is the identity of the 52ms pause). With short-lived-worker
   churn, someone is almost always mid-exit at stop-request time, so nearly every collect
   paid the full amount. → `exit_mutator_worker` wakes the collector via `notify_worker_exit`.

Effect: the N=4000 repro went **from a 60-second timeout → 7.6 seconds** (debug). pin =
`tests/gc_stress.rs::spawn_churn_does_not_starve_stop_the_world` (`pause_ns_total < 2s` over
a 300-spawn churn; measured 3.51s before the fix / tens of ms after — catches a recurrence of
either hole). `make test` 1504 files PASS; S17 concurrency GC=on smoke
(stress/start/nonblocking-await/lock) VERIFY FAIL=0.

  (the number of weak observers; a node drops its value at strong=0, and weaks only extend
  the allocation, so they never resurrect a reclaimed node).
- **Cycle-breaking demonstration** (`gc::gc_ptr` unit tests):
  - `weak_back_edge_breaks_a_cycle_without_the_collector`: making the back-edge of
    `A --strong--> B --weak--> A` weak means the weak does not count toward strong counts
    (untraced by `gc_trace`/root visitors), so **both A and B are freed by refcounting
    alone = the cycle dissolves without the collector** (verified with real drop counters).
  - `weak_observer_of_a_collected_strong_cycle_sees_none`: after the collector reclaims a
    strong cycle, its weak observer gets `upgrade()→None` (weaks do not resurrect reclaimed
    nodes).
  - `upgrade_is_some_while_alive_none_after_drop` / `dangling_weak_upgrades_to_none`.

With this, cycles are handled by both wheels: **(1) cut in advance with weak edges
(proactive; closure self-refs etc.)** and **(2) reclaimed by the trial-deletion collector
(reactive)**. `make test` = Result: PASS (33 gc unit tests green, no regressions). The
primitive is additive only; production execution paths are unchanged.

## 2026-07-03: Phase B (GC) — the trial-deletion collector proper (§11 step 8): actually reclaiming cycles

With the first wave (migrating all container families `Arc → Gc`) complete, implemented the
**Bacon-Rajan trial-deletion collector that actually reclaims cycles**
(`src/gc/collect.rs`). Steps 4-5 so far were "compiled in, default off" groundwork with zero
runtime behavior; this is the first slice where GC actually "runs".

- **Algorithm** (Bacon & Rajan, synchronous variant): drain the candidate buffer and run
  trial deletion on the reachable subgraph. `mark_gray` (tentatively delete internal edges =
  decrement children's GC strong counts) → `scan` (nodes with `strong>0` have external
  references and are restored via `scan_black`; `strong==0` is white = pure cycle garbage) →
  `collect_white` → `reclaim`. The GC strong count is trial-deletion scratch, and survivors
  are fully restored by `scan_black`, so the heap's refcounts are intact after the collect.
- **Reclaiming on Arc**: `Trace::drop_gc_edges(&mut self)` (a container clears its own
  Value-holding collections) cuts white nodes' outgoing edges, dropping internal Arc
  references and letting Arc free them. The `&mut` is obtained by the collector via the
  backing Arc's pointer (`gc::gc_drop_edges`, avoiding `invalid_reference_casting`). During
  reclaim, `CollectGuard` makes `Gc::drop` inert (skipping strong/buffer bookkeeping, Arc
  drop only) → preventing double decrements/underflow.
- **Safety**: **manual/opt-in only** (`gc_debug_collect_now`). There are no production
  callers yet (safepoint wiring is the next slice), and with `MUTSU_GC` unset the candidate
  buffer is empty so a collect is a no-op. Hence normal execution and CI are unchanged.
- **Verification**: unit tests (`gc::collect::tests`) with real drop counters confirm
  reclamation of 2-node / self cycles, preservation of cycles with external references,
  non-reclamation of acyclic candidates, and empty-buffer no-op.
  `make test` = Result: PASS (14076 tests, no regressions).

Next is the rest of §11: safepoint wiring (auto-launching collects at re-entry boundaries) +
Gc-ification of `Promise`/`Channel`/the supply registry (async cycle support) +
`Sub`/`Instance` (second wave) → `LazyList` (third wave).

## 2026-07-02: true lazy-array campaign — resolved the 6 root causes in `S09-subscript/slice.t`

`S09-subscript/slice.t`, the last file of BLOCKERS.md §4 "true lazy arrays / infinite
sequences", used to abort at test 39/56 with a panic on an unimplemented feature; fixed it to

1. **`:=` bind force-materialized genuinely-lazy `LazyList`s**: only the `coroutine` family
   was kept lazy; `sequence_spec`/`closure_seq`/`lazy_pipe` (infinite sequences like

   broad general bug affecting any code using `lazy`/`eager`/`hyper`.
4. **Nested `LazyList` sub-indexes were not recursed on the read side**: the

   slice-assignment branch early-returned, skipping the shared locals-resync tail
   (`vm_var_assign_index_named.rs`). A general correctness bug unrelated to laziness.
6. **Assignment/bind into nested lazy sublists was unimplemented**:

   nesting extends as usual.

   same pattern. Regression test: `t/lock-protect-shared-scalar.t`.
2. **Array-element bind corruption (an independent compiler bug)**: discovered a bug —
   reproducible standalone (no threads) — where running `my $x := @a[$i]` in a loop corrupts
   the element bound in the *previous* iteration. The cause: the `:=` VarDecl/rebind
   compilation shared `compile_call_arg`, which for `Expr::Index` arguments unconditionally
   emitted the temporary global variables meant solely for `is rw` argument write-back of
   `f(@a[$i])` (`__mutsu_index_rw_arg_N`/`__mutsu_index_rw_orig_N`, names fixed at compile
   time). Inside a loop, the same bind statement reuses the same fixed global name every
   iteration, so `SetGlobal`'s "write-through to an existing `ContainerRef`" (meant for
   rebinding `our` variables) kicked in and the new iteration's cell reference was written
   into the previous iteration's cell. Added a new compiler flag `bind_target_direct`, set
   only at the 3 bind/rebind call sites and read-and-immediately-cleared at the top of
   `compile_call_arg_with_escape`, preserving `is rw` write-back for genuine function calls
   nested in a bind RHS (`my $x := f(@a[$i])`) while completely skipping the mechanism for
   the bind target itself. Regression test: `t/bind-array-elem-in-loop.t`.
3. **A TAP output-ordering bug**: `Thread.start` ignored `clone_for_thread`'s
   `parent_in_subtest` and unconditionally set `immediate_stdout = true`, so the `pass` of a
   fire-and-forget thread started inside `subtest { ... }` leaked directly to top-level
   stdout ("ok 5" printed before the subtest header). Made `dispatch_thread_start` respect
   `parent_in_subtest`, and added `drain_shared_thread_output()` to both
   `dispatch_thread_finish` and `finish_subtest` (for the fire-and-forget pattern that never
   calls an explicit `.finish`).

As a side finding, `Lock::Soft` (a type that should behave like `Lock` outside the JVM) was
not recognized as a type at all. Solved by simply registering it as an MRO subclass of
`Lock`, following the existing `Lock::Async` alias pattern (`native_lock` dispatches on
attributes, so it does not depend on the class name).

## 2026-07-02: §2 D — OTF-ification of multi candidates with `state` / caching-proto bodies (#4047)

Part of the pre-GC groundwork (PLAN §2 D, removing multi-dispatch tree-walk fallbacks).
Multi candidates declaring `state` (and caching-proto bodies
`proto cached($a){ state %cache; %cache{$a} //= {*} }`) were forced onto the tree-walk
interpreter fallback on the assumption "OTF-compiling loses the `state` cell's identity". In
fact it is not lost: `otf_compile_cache` (fingerprint+package keyed) returns a stable
compiled body per candidate, so the `state` key (`__state_{pkg}::{$var}@{ip}`, embedding the
compiled opcode position) is stable across calls, and the candidates' `/n`-suffixed packages
separate distinct candidates' `state` cells.

Removed the `function_body_declares_state` exclusion from the 4 multi-candidate OTF forks
and the non-trivial-proto-body fork. Beyond removing the fallback, this **also fixes a
correctness bug**: the tree-walk fallback shared the `state` cell between distinct same-named
candidates (`multi f(Int){state $c} multi f(Str){state $c}`), returning `1,2,3,4,5` (wrong)
via the proto (correct = `1,2,1,2,3`).

The only boundary kept on the interpreter is signature **alternates**
(`multi f(A) | (B) { state $c }`): a single routine sharing `state` across all alternates
(compile-time state_group). Alternates are registered as separate candidates with identical
bodies, so OTF keyed by per-alternate body fingerprint would split the sharing. The resolved
per-candidate `FunctionDef` carries no alternate marker, so registration records them into
`multi_alternate_signature_names` (`FxHashSet<Symbol>`), referenced by the new
`multi_candidate_state_forces_interpreter` gate. pin: `t/multi-candidate-state-otf.t` (the
OTF win path) + the existing `t/multi-signature-alternates.t` (alternate sharing).

  purity check, the IO instance prelude, and comparisons like `== "Failure"`.
- **Per-thread resolve cache**: with as_str now hot, the RwLock read surfaced at 4.1% self,

## 2026-07-12: zef `info` fully works with hyper enabled — the true cause of "%-hash attribute push loss" was shared-array `append` bypassing the atomic store (#4466)

Root-fixed the biggest blocker of PLAN §1 B2, "~80% loss of instance %-hash attribute pushes
on hyper workers". **Both %-hashes and instance attributes were innocent** — the true cause
was plain lexical `@array`s on worker threads:

- `push @a` on a worker has funneled into the `__mutsu_atomic_arr::` store since #4167, and
  subsequent reads prefer the atomic entry. But **only push/unshift** funneled;
  `append`/`prepend`/`pop`/`shift`/`splice` wrote to a stale base/env copy and **silently
  lost everything** (pop/shift "return the value but the removal never lands"). Minimal
  repro: `hyper.map(-> $x { my @a; push @a,"n"; append @a,("p1","p2"); @a.elems })` → 1
  (expected 3).
- In zef, populate's `append @short-names-to-index, $dist.meta<provides>.keys` was wiped
  out, collapsing the keys of `%!short-name-lookup` to 1570, "the number of unique dist
  names" (expected 9256 — the difference being exactly all the provides keys). Pushes into
  the hash itself were fine.
- **fix**: generalized `shared_array_extend` into a generic `shared_array_mutate` (seed the
  atomic entry and apply a closure to the `ArrayData` under the write lock) and extended the
  CallMethodMut shared fast path to all mutators. append/prepend always funnel like push;
  pop/shift/splice are routed "only when an atomic entry shadows the base", preserving the
  legacy slow path's error handling (arity/lazy/immutable) when not shadowed. The splice
  body was extracted from `array_mutate_copy` as the shared `splice_array_data`.
- **Result**: unmodified upstream zef's `zef info Zef` **works end-to-end through the
  Identity/Provides/Depends output with hyper enabled, GC default on, on the real fez index
  (7648 dists)** (stable over 2 release runs). The previously observed "with GC on, the 2nd
  Ecosystems reads an empty `$!name`" also no longer reproduces after the fix (to be
  monitored).
- pin: `t/hyper-array-mutators.t` (12 cases, green under rakudo too).
- **Debugging lessons**: (1) even when it "looks like the hash is corrupted", suspect
  corruption on **the side that generates the keys** (the lexical array) — instrument the
  values at each stage right before they enter the hash (the `idx=[...]` note was the
  clincher). (2) The synthetic repro kept passing because it made the provides key names
  identical to the dist names, rendering the loss invisible — flow **mutually
  distinguishable values** through each data path of a repro. (3) When it looks
  real-data-dependent, trimming the data to 1/150 (jq'ing fez.json 500→50 entries) and
  driving the real module directly turned bisection into minutes.
